**Qora is a small language for *learning* quantum programming.** Quantum code is usually written
through Python libraries (Qiskit, Cirq) that can't tell you when your program stops making physical
sense тАФ or in research languages that assume you already know the theory. Qora sits in between: you
write circuits in a familiar, C#-shaped syntax, the compiler checks that your program actually means
something a quantum computer can do, and then it emits standard **OpenQASM 3** you can inspect line by
line or feed to real toolchains.

Being a teaching language shapes every design decision:

- **Errors come before output, and they explain themselves.** If a program compiles, its OpenQASM is
  valid тАФ never silently wrong.
- **The compiler is see-through.** One command shows how your source became an AST, a typed IR, a
  synthesized inverse, and finally QASM.
- **The hard quantum rules are enforced, not assumed** тАФ you can't accidentally reuse a dirty qubit,
  invert a measurement, or pass a register where a single qubit belongs.

1. Install **[Qora Language](https://marketplace.visualstudio.com/items?itemName=qora-lang.qora-language)**
   from the VS Code Marketplace.
2. Create a file named `bell.qor` and paste the example above.
3. You already have a working toolchain:
   - errors underline **as you type** (hover them for the reason тАФ try `CNOT(q[0], q[0])`);
   - hover any gate or keyword for a short doc (`H`, `use`, `M`, тАж);
   - **`Ctrl+Shift+P` тЖТ `Qora: Transpile to OpenQASM`** opens the compiled output beside your file;
   - **`Qora: Show Compilation Stages`** shows the whole pipeline live тАФ source тЖТ AST тЖТ IR тЖТ
     (inverse IR) тЖТ QASM тАФ and refreshes every time you save.

Prefer the repo? With the .NET 10 SDK:

```bash

cd Qora
dotnet run --project src/Qora          # parses a demo program, prints its OpenQASM
```

(The same binary powers the extension via `qora --json` тАФ one line of JSON per compile:

## The language in five minutes

**Qubits and gates.** Qubits are allocated with `use` and addressed like arrays. Gates are calls:

```
use q = Qubit[3];              // three qubits, all |0тЯй
H(q[0]);                       // superposition
CNOT(q[0], q[1]);              // entangle
Rx(pi/2, q[2]);                // rotations take the angle first
Reset(q[2]);                   // back to |0тЯй
```

Built-in gates: `H X Y Z S T` ┬╖ `CNOT/CX CY CZ SWAP CCX` ┬╖ `Rx Ry Rz` ┬╖ `Reset ResetAll`.

**Measurement** collapses a qubit into a classical `bit` тАФ and that's the only way to get one:

```
bit r = M(q[0]);               // measure (this is irreversible!)
r = M(q[0]);                   // re-measure into the same bit
```

**Classical control** looks exactly like you expect, and can react to measurements:

```
const int n = 2;
var count = 0;
for i in 0..n {  H(q[i]);  }
if (r == 1) {  X(q[1]);  } else {  Z(q[1]);  }
while (count < 2) {  count = count + 1;  }
repeat {  H(q[0]);  r = M(q[0]);  } until (r == 1);
```

**Operations** are subroutines (`Main` is the entry point), and **functors** transform them:

```
operation Prep(Qubit[2] q) {
    H(q[0]);
    Controlled X(q[0], q[1]);   // add a control to any gate
    T(q[1]);
}

Adjoint Prep(q);                // тШЕ the compiler SYNTHESIZES Prep's inverse:
                                //   gates reversed & inverted, for-loops run backwards,
                                //   if-branches inverted, nested calls handled transitively
```

That `Adjoint` line is Qora's flagship: "undo this computation" is a one-word request, and it is the
first stone on the road to [automatic uncomputation](docs/TODO.md).

**Namespaces and imports** organize bigger programs across files, C#-style:

```
import gates_lib;               // loads gates_lib.qor (next to this file)

namespace App {
    open GatesLib;              // unqualified access to GatesLib's operations
    operation Go(Qubit[2] q) {
        Bell(q);                // found via the open
    }
}

operation Main() {
    use q = Qubit[2];
    App.Go(q);                  // qualified call
    GatesLib.Bell(q);           // qualification always works, open or not
}
```

Resolution follows the C#/Q# rules: a name is looked up in the operation's own namespace, then the
global namespace, then the `open`ed namespaces тАФ and if two opened namespaces both provide it, that's
a compile error listing the candidates, not a silent pick. `import lib.gates;` maps dots to
directories (`lib/gates.qor`), files load once no matter how many import paths reach them, and a
cycle is an error that shows you the chain.

Even built-in gate names follow these rules (Q#-style): the built-ins live in the implicit
`Qora.Intrinsic` namespace, so a namespaced operation may reuse a name like `Rx` тАФ and any use that
could mean both is an explicit ambiguity error, resolved by qualifying (`L.Rx(q)` for yours,
`Qora.Intrinsic.Rx(pi/2, q)` for the gate). A bare gate name with no user candidate in scope always
means the built-in, so what the book teaches is what your program means.

**When you get it wrong, the compiler says why** тАФ every violation in one compile, before any output:

```
[QSEM001] in `Main`: `Adjoint Meas` cannot be compiled: it measures a qubit,
          and measurement is irreversible
[QSEM006] in `Main`: `Bell` expects 1 argument(s) but got 2
[QSEM016] in `Main`: index `q[5]` is out of range; `q` has 2 qubit(s) (valid: 0..1)
[QSEM018] in `Go`: `Flip` is ambiguous here: it could be `LibA.Flip` or `LibB.Flip`
          тАФ qualify the call (e.g. `LibA.Flip(...)`)
[QSEM021] imports form a cycle: main.qor -> lib.qor -> main.qor тАФ remove one of these imports
```

**Running your programs**: every Qora program тАФ operations, loops, classical variables, even
`while` тАФ runs today on Amazon Braket's local simulator:

```bash
pip install amazon-braket-sdk            # Python 3.10тАУ3.13
python tools/run_braket.py yourfile.qor --shots 1000
```

Both execution paths are release-gated with measured statistics: `tools/braket_validation.py`
proves the FULL language executes (including whole-operation `Adjoint` returning states to
`|0тАж0тЯй`), and `tools/qiskit_validation.py` covers the flat subset Qiskit's `qasm3.loads(...)`
accepts (its importer doesn't read `def`/`int`/variable loop indices yet; a flattening emission
mode is planned to bridge Qiskit and real hardware).

## How the compiler works

```
source тФАparseтФАтЦ╢ AST тФАlowerтФАтЦ╢ typed IR тФАload importsтФАтЦ╢ merged IR тФАresolve namesтФАтЦ╢ FQN IR
    тФАvalidateтФАтЦ╢ (errors? stop & report ALL of them)
    тФФтФА clean тФАтЦ╢ mangle names тФАтЦ╢ invert (Adjoint synthesis) тФАтЦ╢ OpenQASM 3
```

- The **typed IR** (`src/Qora.Core/Ir/`) is owned end-to-end by the compiler; the parse engine stops
  at the lowering boundary.
- The **module loader** merges the whole `import` graph into one program, and the **resolver**
  rewrites every name to its fully-qualified form (C#/Q# rules, ambiguity as an error) тАФ so every
  later stage sees one unambiguous name ([design](docs/namespaces-design.md)).
- The **validator** (QSEM001тАУ024) is collect-all: one compile reports every problem тАФ argument kinds
  and counts, register sizes, index bounds, reserved names, recursion, misplaced `use`, irreversible
  `Adjoint` targets (with the reason chained through the call graph), and more.
- The **name mangler** appends `_` to every user-defined name (namespaces flatten as
  `MyLib.Bell` тЖТ `MyLib__Bell_`), making collisions with OpenQASM's own names structurally
  impossible. Errors and the stages view keep your original names; only the QASM is mangled.
- The **inverter** is a pure IRтЖТIR pass тАФ the same machinery a future pass will reuse to inject
  automatic uncomputation.
