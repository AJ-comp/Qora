# Qora

**A learning language for quantum programming.**
Write quantum circuits in Q#/C#-like syntax and emit verified **OpenQASM 3**.

**Qora is a learning language for quantum programming.** Quantum code is often written through Python
libraries such as Qiskit or Cirq, or through research languages that assume you already know the theory.
For a learner, it can be hard to move between quick experiments and precise concepts.

Qora focuses on bridging that gap. Write a circuit with familiar programming syntax, and the compiler
first checks whether the program describes something a quantum computer can actually do.
Then it produces standard **OpenQASM 3** that you can inspect line by line or pass to real toolchains.

That direction carries into the design:

- **Errors come before output.** If a program compiles, its OpenQASM is valid.
- **The compiler internals are inspectable.** One command shows how source becomes AST, typed IR,
  symbol table, synthesized inverse operations, and QASM.
- **Quantum rules are checked by the compiler.** Reusing an already-used qubit, inverting a
  measurement, or passing a register where a single qubit is expected is rejected at compile time.

Names pass through unchanged when they are already safe for OpenQASM. When a user name would collide

adds a `// Qora:` note explaining why.

## Quick start

The fastest path is to start with the VS Code extension.
The compiler is bundled inside the extension, so there is **nothing else to install**, and neither .NET nor Python is required.

1. Install [Qora Language](https://marketplace.visualstudio.com/items?itemName=qora-lang.qora-language) from the VS Code Marketplace.
2. Create a file named `bell.qor` and paste the example above.
3. Save the file and the features below are ready to use.

- **As you type**, errors are underlined, and hovering the underline shows the reason. Try `CNOT(q[0], q[0])`.
- Hover a gate or keyword to see a short note for items such as `H`, `use`, and `M`.
- **`Ctrl+Shift+P` → `Qora: Transpile to OpenQASM`** opens the compiled output beside your file.
- **`Qora: Show Compilation Stages`** shows source → AST → IR → symbol table → (inverse IR) → QASM every time you save.

To run the repository directly, you only need the .NET 10 SDK.

```bash
git clone https://github.com/AJ-comp/Qora.git
cd Qora
dotnet run --project src/Qora          # parses a demo program, prints its OpenQASM
```

The VS Code extension uses the same binary through `qora --json`, and each compile returns one JSON line in the form `{success, qasm, errors[]}`.

## The language in five minutes

**Qubits and gates.** A new qubit register is allocated with `use q = Qubit[N];` and addressed like an array.
Here, `use` is not a C# `using`-style automatic cleanup marker; it is the syntax that creates a qubit register for the Qora program.
Helper operations accept a register as `Qubit[]` and can read the caller's concrete length through `.Count`.
Gate calls have the same shape as function calls.

```
use q = Qubit[3];              // allocate qubit register q, size 3
H(q[0]);                       // superposition
CNOT(q[0], q[1]);              // entangle
Rx(pi/2, q[2]);                // rotations take the angle first
Reset(q[2]);                   // back to |0⟩
```

Built-in gates: `H X Y Z S T` · `CNOT/CX CY CZ SWAP CCX` · `Rx Ry Rz` · `Reset ResetAll`.

**Measurement** stores a qubit's measurement result in a classical `bit`, and in Qora measurement is the only way to obtain a `bit`.

```
bit r = M(q[0]);               // measure, irreversible
r = M(q[0]);                   // re-measure into the same bit
```

**Classical control** looks exactly like you expect, and can react to measurements:

```
const int n = 2;
var count = 0;
for i in 0..n {  H(q[i]);  }
if (r == 1) {  X(q[1]);  } else {  Z(q[1]);  }
while (count < 2) {  count = count + 1;  }
repeat {  H(q[0]);  r = M(q[0]);  } until (r == 1);
```

**Classical arrays** use the familiar `T[]` form for `int`, `float`, `bit`, and `angle` values.
Declare them anywhere a scalar goes — `Main`, helper operations, loops, branches (the compiler handles
OpenQASM's global-scope rule automatically) — then read or update elements by index and use `.Count` when iterating.

```qora
int[] counts = [1, 2, 3];
float[] weights = new float[3];     // three zero-initialized elements
weights[1] = 0.5;
for i in 0..counts.Count-1 { counts[i] = counts[i] + 1; }
```

User-defined operations are written as a name, parameters, and a block; `Main` is the entry point.
**Functors** transform user-defined operations or gates:

```
operation Prep(Qubit[] q)
{
    H(q[0]);
    Controlled X(q[0], q[1]);   // add a control to any gate
    T(q[1]);
}

Adjoint Prep(q);                // ★ the compiler SYNTHESIZES Prep's inverse:
                                //   gates reversed & inverted, for-loops run backwards,
                                //   if-branches inverted, nested calls handled transitively
```

That `Adjoint` line is one of Qora's representative features: a request to "undo this computation" becomes one line of code, and the compiler builds the actual reverse circuit.
It is also the starting point for [automatic uncomputation](docs/TODO.md).

**Namespaces and imports** let large programs be split across multiple files.
`import` uses a quoted file path, like Python, including the extension.

`gates_lib.qor`

```qora
namespace GatesLib
{
    operation Bell(Qubit[] q)
    {
        H(q[0]);
        CNOT(q[0], q[1]);
    }
}
```

`rotations.qor`

```qora
namespace Rotations
{
    operation Tilt(Qubit q)
    {
        Ry(pi/4, q);
        Rz(pi/8, q);
    }
}
```

`main.qor`

```qora
import "gates_lib.qor";         // load gates_lib.qor from the same directory
import "rotations.qor";         // load rotations.qor from the same directory

namespace App
{
    open GatesLib;
    open Rotations;

    operation Prepare(Qubit[] q)
    {
        Bell(q);                // unqualified through open
        Tilt(q[0]);             // unqualified through open
    }
}

operation Main()
{
    use q = Qubit[2];
    App.Prepare(q);             // qualified call into App
    Rotations.Tilt(q[1]);       // direct qualified call into another imported namespace
}
```

Name resolution is similar to the pattern used by familiar programming languages with namespaces.
Qora looks in the current namespace, then the global namespace, then the `open`ed namespaces.
If two `open`ed namespaces contain the same name, Qora does not silently pick one; it reports a compile error with the candidates.

`import "lib/gates.qor";` loads `lib/gates.qor` relative to the file that contains the import.
The path is written exactly inside quotes, including the extension.
Each canonical path is attempted only once even if multiple import paths reach it.
A cyclic back-edge therefore reaches a path already registered in `loaded` and is skipped without an error; QSEM021 is retained only as a retired/reserved code and is no longer emitted.

Built-in gate names follow the same rule, and built-in gates live in the implicit `Qora.Intrinsic` namespace.
Because of that, a namespace may define an operation with a name such as `Rx`.
If a call could mean both a user-defined operation and a built-in gate, Qora reports an explicit ambiguity error.
In that case, write the user-defined operation as `L.Rx(q)` and the built-in gate as `Qora.Intrinsic.Rx(pi/2, q)`.

If there is no user candidate with the same name in scope, a bare gate name means the built-in gate.
That lets you use the gate names from the documentation directly in your program.

**When you get it wrong, the compiler says why** — every violation in one compile, before any output:

```
[QSEM001] in `Main`: `Adjoint Meas` cannot be compiled: it measures a qubit,
          and measurement is irreversible
[QSEM006] in `Main`: `Bell` expects 1 argument(s) but got 2
[QSEM016] in `Main`: index `q[5]` is out of range; `q` has 2 qubit(s) (valid: 0..1)
[QSEM018] in `Go`: `Flip` is ambiguous here: it could be `LibA.Flip` or `LibB.Flip`
          — qualify the call (e.g. `LibA.Flip(...)`)
```

**Running your programs.** Qora programs with user-defined operations, loops, classical variables, and `while` can run today on Amazon Braket's local simulator:

```bash
pip install amazon-braket-sdk            # Python 3.10–3.13
python tools/run_braket.py yourfile.qor --shots 1000
```

Both execution paths are also used for release validation with measured statistics.
`tools/braket_validation.py` confirms that the full language executes, including whole-operation `Adjoint` returning states to `|0…0⟩`, and `tools/qiskit_validation.py` validates the flat subset accepted by Qiskit's `qasm3.loads(...)`.
Qiskit's importer does not yet read `def`, `int`, or variable loop indices, so a flattening emission mode is planned to bridge Qiskit and real hardware.

## How the compiler works

The canonical stage-by-stage version is  **[the Qora Compiler Datasheet](https://aj-comp.github.io/Qora/architecture.html)**.

A compiler is a program that translates the code you wrote into a standard form a quantum computer can
actually run — here, **OpenQASM 3**. Qora doesn't do that in one leap; it splits the work into small steps,
each doing exactly one job.

The very first thing it does is move your code into a form that's convenient to work on internally, called
the **IR** (intermediate representation) — think of it as the compiler's own tidy notebook copy of your
program. Because it reshapes this IR instead of re-reading your text, every later step stays simple and hard
to get wrong.

It then checks names, catches rule violations, and applies the transformations it needs on that IR, and
finally prints the result out as OpenQASM text. Each row below is one of those steps:

| Stage | Pass | Input | Output | What it does |
|---|---|---|---|---|
| Parse | *Janglim engine* | source text | AST | Turns the source code you wrote into a syntax tree the compiler can work with — shape only, no meaning yet. |
| Lower | `QoraLowering` | AST | `QProgram` (IR) | Rebuilds that parser tree as Qora's own data structures (the IR); every later step works on this. |
| Expand | `ModuleLoader` | `QProgram` (IR) | `QProgram` (IR) | Reads every `.qor` file pulled in by `import` and merges them into one program, so later steps see a single file. |
| Desugar measure | `MeasureConditionLowering` | `QProgram` (IR) | `QProgram` (IR) | Rewrites a measurement taken directly inside a condition — `if (M(q[0]) == 1)` — into "measure into a `bit` first, then test that `bit`", because OpenQASM can't measure inside a condition. |
| Resolve | `Resolver` | `QProgram` (IR) | `QProgram` (IR) | Works out which operation a short call like `Bell(q)` really means, and pins it to its full name (e.g. `GatesLib.Bell`). |
| Validate | `QoraValidator` + `SymbolTableBuilder` | `QProgram` (IR) | `QProgram` (IR) or errors | Checks the whole program is something a quantum computer can actually do; if anything is wrong it reports every error and stops here (no OpenQASM is produced). |
| Monomorphize | `Monomorphizer` | `QProgram` (IR) | `QProgram` (IR) | Specializes an operation with a `Qubit[]` parameter for the concrete register size at each call and replaces uses such as `q.Count` before emission — like a C++ template or a Rust generic. |
| Flatten conjugation | `ConjugationLowering` | `QProgram` (IR) | `QProgram` (IR) | Unfolds a within/apply (compute · use · uncompute) block into plain gates, auto-adding the reverse of `within` at the end to clean up the temporary qubit. |
| Materialize | `AdjointMaterializer` | `QProgram` (IR) | `QProgram` (IR) | Turns `Adjoint Foo` ("run Foo backwards") into a real inverse operation `Foo__adj` — its body reversed and each gate inverted. |
| Mangle | `NameMangler` | `QProgram` (IR) | `QProgram` (IR) | Renames anything that would clash with an OpenQASM keyword or gate name to a safe form (adds `_` only on a real clash), and flattens namespace dots (`MyLib.Bell` → `MyLib_Bell`). |
| Check | `ReferentialCheck` | `QProgram` (IR) | `QProgram` (IR) or errors | A final safety net: confirms every name the program uses points at something really declared; a mismatch means a compiler bug, so it fails loudly. |
| Target-lower | `OpenQasmLowering` | `QProgram` (IR) | `QProgram` (IR) | Adjusts the few things that are valid in Qora but not in OpenQASM — for example, a `const` holding a runtime value becomes a plain variable. |
| Emit | `QasmEmitter` | `QProgram` (IR) | OpenQASM text | Prints the now-finished IR out as OpenQASM 3 text; it makes no more decisions — a pure printer. |

- The **typed IR** (`src/Qora.Core/Ir/`) is owned end-to-end by the compiler; the parse engine stops
  at the lowering boundary.
- The **module loader** merges the whole `import` graph into one program.<br>The **resolver** then pins operation call targets to fully-qualified names (FQNs), so every later call site points at exactly one operation.<br>As in other programming languages with namespaces, ambiguity is an error; see the [design document](docs/namespaces-design.md) for details.
- The **validator** (QSEM001–025) is collect-all.<br>One compile reports argument kinds and counts, register sizes, index bounds, reserved names, recursion, misplaced `use`, unknown or used-before-declared identifiers, irreversible `Adjoint` targets, and more.<br>The unified symbol table used by the validator is also exposed in the stages view.
- The **name mangler** flattens namespace dots (`MyLib.Bell` → `MyLib_Bell`) and appends `_` only when
  an emitted identifier would otherwise collide with a keyword, stdgates gate, or another emitted name.
  Errors and the stages view keep your original/FQN names; only the QASM sees the collision-safe form.
- The **adjoint materializer** uses a pure IR→IR inverter to turn whole-operation `Adjoint` calls into
  synthesized inverse operations before names are mangled. That same inverter already backs
  **conjugation lowering**, which flattens a `within`/`apply` (compute · apply · uncompute) node into gates —
  the first landed piece on the road to automatic uncomputation; what remains is the analysis that injects it.
