9. 차단의 두 종류 — 원리적 한계와 보수적 거부

에러의 지도 — 물리의 벽과 구현의 여지

← 8. 담요 문제 — 질의 폭과 문장 폭 코스 목차 10. 왜 정적 컴파일인가 — 두 개의 런타임 →

컴파일 에러로 이어지는 차단에는 성격이 다른 두 종류가 있습니다. 원리적 한계는 어떤 컴파일러도 넘을 수 없는 물리의 벽입니다. 예컨대 앤실라에 H를 적용해 줄을 두 갈래로 쪼갠 뒤 생존자가 그 갈래를 기록하면, 생존자를 건드리지 않고 앤실라를 |0⟩로 되돌리는 청소가 아예 존재하지 않게 되므로 어느 언어로 옮겨 적어도 같은 지점에서 거부됩니다. 반면 보수적 거부는 현재 구현이 아직 다루지 못해 안전한 쪽으로 물러선 판정이며, 향후 완화가 계획되어 있습니다. 예컨대 벌거벗은 루프는 계산 자체를 완벽히 되돌릴 수 있는데도 지금의 주입기가 문장 단위로만 역을 만들기 때문에 차단되고, 연산으로 한 겹 감싸면 오늘도 통과합니다. 여덟 가지 차단을 성격별로 정리하면 다음과 같습니다.

차단성격설명
H·Rx·Ry 스크래치🧱 원리적 한계줄을 쪼갠 뒤 생존자가 기록하면 도로 접을 수 없습니다 — 모든 언어가 동일하게 거부합니다
얽힌 타깃 위의 Y·CY🧱 원리적 한계5장 — 계약을 지키는 청소가 존재하지 않습니다
측정된 값🧱 원리적 한계붕괴에는 유니터리 역이 없습니다 (측정된 앤실라는 애초에 청소 대상이 아니라 출력)
런타임 의존 값 (동적 인덱스·경계)🧱 정적 컴파일의 숙명컴파일 타임에 회로를 확정하는 모든 언어의 공통 한계 — 10장
벌거벗은 루프 (몸통이 값 편집뿐)🔧 보수적 거부복합문 통째 역변환으로 완화 예정 (지금은 연산으로 감싸면 통과 — 7장)
정적 담요 (리터럴 경계 루프·브로드캐스트)🔧 보수적 거부정적 전개로 원소별 정밀 기록이 가능해지면 자동 청소로 전환됩니다
확정 기저값 위의 Y🔧 보수적 거부얽히지 않았음을 그래프로 증명하면 재허용 가능합니다 (5장)
측정 bit 조건의 if 안 쓰기🔧+🧱 혼합역변환 자체는 가능하나 "조건이 변하지 않았다"는 증명이 추가로 필요합니다

마지막 행처럼 두 성격이 겹치는 경우도 있습니다. 측정 bit를 조건으로 쓴 if 안의 쓰기는 역변환 자체는 만들 수 있지만, 청소 시점까지 그 조건 값이 변하지 않았다는 증명이 추가로 필요합니다.

🧱 원리적 한계는 언어가 사용자에게 물어야 마땅한 질문입니다 · 🔧 보수적 거부는 완화될수록 에러가 자동 청소로 바뀝니다

대표 사례 ① — 원리적 한계: H 스크래치

2장의 장부로 읽으면 벽이 정확히 어디에 있는지 보입니다. H는 앤실라의 줄 하나를 두 갈래로 쪼개고, 생존자가 CNOT으로 그 갈래를 기록하는 순간 두 갈래는 서로 다른 세계가 됩니다. 

use q = Qubit[1];  use t = Qubit[1];
H(t[0]);            // t의 줄 1개를 2갈래로 쪼갬
CNOT(t[0], q[0]);   // 생존자 q가 갈래를 기록
bit m = M(q[0]);    // q는 출력으로 승격, t는 청소 후보 → ❌ 차단

이 시점부터는 t만 만지는 어떤 연산도 — H를 한 번 더 적용해도 — 두 갈래를 하나로 접어 |0⟩로 되돌리지 못합니다. 접으려면 q에 남은 기록까지 거꾸로 지워야 하는데, 그러면 생존자에게 배달된 값 자체가 사라져 4장의 계약이 깨집니다. 계약을 지키는 청소가 존재하지 않으므로 차단이 유일한 답이고, 이 벽은 Qora가 아니라 물리가 세운 것이어서 어느 언어로 가져가도 같은 지점에서 같은 거부를 만납니다.

대표 사례 ② — 보수적 거부: 벌거벗은 루프

보수적 거부는 벽이 아니라 도달 범위의 문제입니다. 7장에서 보았듯 지금의 주입기는 문장 단위로만 역을 만들 수 있고 카탈로그에 "for의 역"이라는 항목이 없어서, 몸통이 값 편집뿐인 루프를 정직하게 거부합니다. 그러나 같은 계산을 연산으로 한 겹 감싸 문장 하나로 만들면 오늘도 safe 판정을 받습니다.

벌거벗은 루프 — ❌ 지금은 차단

use a = Qubit[3];  use out = Qubit[1];
for i in 0..2 { X(a[i]); }
CNOT(a[0], out[0]);
bit c = M(out[0]);
// a 청소 불가 (ContainedWrite)

연산으로 감싼 같은 계산 — ✅ safe

operation Flip(Qubit[3] p){
  for i in 0..2 { X(p[i]); }
}
use a = Qubit[3];  use out = Qubit[1];
Flip(a);
CNOT(a[0], out[0]);
Adjoint Flip(a);   // 자동 주입
bit c = M(out[0]);  // c = 1 (확률 100%)

표의 🔧 항목들은 모두 이런 성격입니다. 복합문 통째 역변환이 들어오면 위의 벌거벗은 루프가 감싸지 않아도 통과하고, 정적 전개가 들어오면 담요 차단(8장)이, 얽히지 않았다는 그래프 증명이 들어오면 확정 기저값 위의 Y 차단(5장)이 각각 자동 청소로 바뀝니다. 즉 보수적 거부는 오늘 쓸 수 있는 우회가 있고, 내일은 그 우회조차 필요 없어지는 에러입니다.

에러를 내는 것 자체가 설계 결함이 아닌 이유도 여기서 나옵니다. 컴파일러에게 남은 대안은 둘뿐입니다. 청소를 조용히 건너뛰면 "측정하지 않은 use 레지스터는 |0⟩로 반납된다"는 언어 규칙이 거짓이 되고, 억지로 역연산을 주입하면 5장에서 본 것처럼 생존자의 결과가 바뀝니다. 자동으로 보증할 수 없는 것을 소리 내어 알리고, 8장의 수정 경로처럼 사용자에게 선택지를 돌려주는 것이 유일하게 정직한 동작입니다.
