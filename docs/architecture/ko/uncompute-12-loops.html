12. 루프와 호출 — 왜 판정이 달라지는가

문장 구조가 판정을 가르는 이유

← 11. 주입기 — 문장 붙여넣기 기계 코스 목차 13. 담요 문제 — 질의 폭과 문장 폭 →

11장의 주입기는 앤실라를 편집한 문장들의 목록을 가져와, 각 문장의 역을 역순으로 사망점에 붙여넣는 기계였습니다.
이 장에서는 그 붙여넣기 절차를 루프가 들어간 계산에 적용하면 무슨 일이 생기는지 확인합니다.
다루는 계산은 하나뿐입니다.
세 큐빗짜리 앤실라의 원소마다 X 게이트를 한 번씩 적용해 |000⟩을 |111⟩로 만드는 계산입니다.
같은 계산이라도 문장 구조를 어떻게 잡는지에 따라 절차의 운명이 갈리므로, 먼저 절차가 끝까지 성립하는 구조를 하나 세워 두고, 그다음 그 구조에서 딱 한 줄만 바꿔서 어디가 무너지는지 보겠습니다.

1단계 — 루프를 연산으로 감싸 호출하면 붙여넣기가 그대로 성립합니다

세 번의 X 게이트를 Flip이라는 연산의 몸통에 넣고, Main에서는 그 연산을 문장 하나로 호출합니다.
앤실라 a는 |111⟩이 된 뒤 첫 원소의 값을 out에 복사하고, 프로그램은 out만 측정합니다. 

operation Flip(p: Qubit[]){
    for i in 0..2 { X(p[i]); }   // 루프는 연산 안에 삽니다
}
operation Main(){
    use a   = Qubit[3];
    use out = Qubit[1];
    Flip(a);                     // a를 편집한 유일한 문장 — a는 |111⟩
    CNOT(a[0], out[0]);          // a의 값을 out에 복사한다. 이후 a는 쓰이지 않는다
    var c: bit = M(out[0]);           // c = 1 (확률 100%)
}

11장의 절차를 그대로 밟아 보겠습니다.
a를 편집한 문장의 목록은 호출 Flip(a) 하나이고, 이 문장은 텍스트로도 하나, 실행도 정확히 한 번입니다.
역 카탈로그에서 호출 Foo(...)의 역은 Adjoint Foo(...)이므로, 사망점인 CNOT 직후에 Adjoint Flip(a)가 붙습니다.
붙여넣기가 끝난 Main의 몸통은 다음 텍스트가 되고, 모든 문장이 그 자리에서 성립합니다. 

    Flip(a);                     // a는 |111⟩
    CNOT(a[0], out[0]);          // 사망점
    Adjoint Flip(a);             // ← 주입된 역 — 문장 하나, 실행 한 번, a는 |000⟩로 복귀
    var c: bit = M(out[0]);           // c = 1 (확률 100%) — 주입 전과 동일

a는 safe 판정을 받고, 이 주입으로 |000⟩까지 되돌아가 반납됩니다.
주입된 문장 하나가 세 번의 X 게이트를 전부 되돌릴 수 있는 근거는 유니터리의 성질입니다.
Flip의 몸통이 차례로 적용하는 게이트를 U₁, U₂, U₃이라 하면, 호출 문장 Flip(a)가 양자 상태에 적용하는 것은 그 곱으로 정의되는 하나의 유니터리 U₃U₂U₁입니다.
유니터리 곱의 역은 각 요소의 역을 역순으로 곱한 (U₃U₂U₁)† = U₁†U₂†U₃†이므로, Adjoint Flip(a)는 마지막 게이트의 역부터 첫 게이트의 역까지를 차례로 실행하는 연산이 됩니다.
이 순서 뒤집기는 사용자가 손으로 할 일이 아니라, Adjoint 몸통을 만들 때 컴파일러가 처리하는 일입니다.
양쪽이 실행하는 게이트를 단계별로 나란히 놓으면 다음과 같습니다.

단계Flip(a)가 실행하는 게이트Adjoint Flip(a)가 실행하는 게이트
1X(a[0])X(a[2])
2X(a[1])X(a[1])
3X(a[2])X(a[0])

X 게이트의 역은 X 자신이므로 게이트 이름은 그대로이고, 실행 순서만 거울처럼 뒤집힙니다.
|000⟩에서 출발한 a는 Flip(a) 뒤에 |111⟩이 되고, Adjoint Flip(a) 뒤에 다시 |000⟩으로 돌아옵니다.
out은 주입보다 앞선 CNOT 게이트에서 이미 |1⟩로 바뀌었으므로, 측정 결과는 주입 전과 똑같이 100% 확률의 1입니다.
절차가 이렇게 끝까지 성립한 근거를 정리하면 두 가지입니다.
a를 편집한 문장이 텍스트 하나에 실행 한 번으로 대응했고, 그 문장 종류인 호출의 역이 역 카탈로그에 있었습니다.
그럼 이 두 근거가 사라지면 절차는 정확히 어디서 무너질까요.

2단계 — 같은 루프를 35에 직접 쓰면 어디가 막히는가

무너지는 지점을 보려고 방금의 프로그램에서 딱 한 줄만 바꾸겠습니다.
호출 Flip(a)가 있던 자리에 Flip의 몸통에 있던 루프를 그대로 옮겨 적는 것입니다.
이렇게 연산에 감싸이지 않고 호출자 몸통에 직접 쓰인 루프를 벌거벗은 루프라고 부르겠습니다. 

operation Main(){
    use a   = Qubit[3];
    use out = Qubit[1];
    for i in 0..2 { X(a[i]); }   // ← 바뀐 한 줄: 호출 대신 벌거벗은 루프
    CNOT(a[0], out[0]);          // 사망점 — 계산 내용은 1단계와 완전히 같음
    var c: bit = M(out[0]);
}

계산의 내용은 조금도 변하지 않았습니다.
a는 여전히 |111⟩이 되고, out은 여전히 |1⟩로 바뀝니다.
그런데 절차의 첫 단계인 목록 만들기부터 사정이 달라집니다.
a를 편집한 문장으로 목록에 잡히는 것은 루프 몸통의 X(a[i])입니다.
for 자체는 목록에 들어갈 수 있는 문장 종류가 아닌데, 역 카탈로그에 게이트 적용의 역과 호출의 역은 있어도 "for의 역"이라는 항목은 없기 때문입니다.
그리고 목록에 잡힌 X(a[i])는 텍스트로는 한 번 존재하면서 실제로는 i = 0, 1, 2에 대해 세 번 실행되었습니다.
그래도 절차를 억지로 끝까지 밀어붙이면 어떤 텍스트가 나오는지 보겠습니다.
X 게이트는 자기 역이므로 목록의 유일한 항목 X(a[i])의 역은 X(a[i]) 그대로이고, 이것을 사망점인 CNOT 직후에 붙여넣으면 다음이 됩니다. 

    for i in 0..2 { X(a[i]); }   // i는 이 중괄호 안에서만 존재하는 이름
    CNOT(a[0], out[0]);          // 사망점
    X(a[i]);                     // ← 붙여넣기 시도 — i가 루프 밖에 없음 (성립하지 않는 텍스트)
    var c: bit = M(out[0]);           // 게다가 문장 1개의 역 1개 ≠ 실행 3번

실패가 두 겹으로 겹쳐 있습니다.
첫째, 붙여넣은 문장 속의 i는 루프 안에서만 존재하는 이름이라서, 루프 밖에 놓인 X(a[i])는 아예 성립하지 않는 텍스트입니다.
둘째, 이름 문제를 눈감아 주더라도 역 문장 하나로는 세 번의 실행을 되돌릴 수 없습니다.
1단계에서 절차를 지탱했던 "문장 하나 = 실행 한 번"의 대응이 컨테이너 안에서는 깨지기 때문입니다.
그래서 컴파일러는 이런 텍스트를 만들어 보는 대신, 컨테이너(if, for) 안에 들어 있는 쓰기를 만나면 그 앤실라를 ContainedWrite로 분류해 자동 청소를 차단합니다.
성립하지 않는 정리 코드를 조용히 만드느니 이 쓰기를 정직하게 거부하는 것입니다.

그런데 여기서 이상한 점이 하나 남습니다.
1단계의 프로그램에도 똑같은 루프가 분명히 있었습니다.
Flip의 몸통 안에서 X(p[i])는 마찬가지로 세 번 실행되었는데, 왜 거기서는 아무것도 차단되지 않았을까요.

같은 루프가 1단계에서는 왜 무사했는가 — 호출은 통째 역변환 단위

답은 루프가 사는 곳에 있습니다.
1단계에서 루프는 Main의 몸통이 아니라 호출된 연산 Flip의 몸통 안에 살았고, Main에서 a를 편집한 문장은 컨테이너 밖의 호출 하나였습니다.
주입기는 Adjoint Flip(a)라는 문장 하나만 붙여넣으면 되고, 세 번의 반복을 되돌리는 일은 그 문장이 적용하는 역 유니터리 U₁†U₂†U₃† 안에서 일어나므로, 붙여넣기 절차가 반복의 개수를 셀 필요가 없습니다.
판정이 나뉜 지점이 정확히 여기입니다.

호출은 통째 역변환 단위입니다

호출을 원자 단위로 뒤집으면 호출된 연산(callee) 내부의 반복과 구조는 뒤집힌 단위 안에서 그대로 미러링됩니다.
ContainedWrite가 감시하는 것은 호출자 몸통에서 쓰기를 직접 감싼 컨테이너뿐이므로, callee 내부의 루프는 차단 사유가 되지 않습니다.
벌거벗은 루프를 지금 통과시키는 실용적 요령도 여기서 나옵니다.
루프를 연산으로 한 겹 감싸서 호출 문장 하나로 만들면 됩니다.
벌거벗은 루프 자체를 통째로 뒤집는 일반화, 곧 몸통이 전부 값 편집인 복합문을 하나의 단위로 역변환하는 확장은 계획된 개선입니다(14장).

한눈에 다시 보기 — 같은 계산, 두 문장 구조

두 구조에서 벌어진 일을 나란히 놓고 요약하면 다음과 같습니다.

연산으로 감싼 루프 — ✅ safe

operation Flip(p: Qubit[]){
  for i in 0..2 { X(p[i]); }
}
use a = Qubit[3];
Flip(a);
Adjoint Flip(a);  // 붙여넣기 성립

Flip(a)는 어엿한 문장 하나이고 실행도 정확히 한 번입니다.
그 역 Adjoint Flip(a)도 문장 하나이며, 내부의 루프는 그 안에서 알아서 거꾸로 돕니다 (X(a[2]); X(a[1]); X(a[0])).

벌거벗은 루프 — ❌ 차단 (ContainedWrite)

use a = Qubit[3];
for i in 0..2 { X(a[i]); }
// 붙여넣기 시도:
X(a[i]);  // ← i가 루프 밖에 없음
//    (성립하지 않는 텍스트)
// 게다가 문장 1개 ≠ 실행 3번

for는 문장이 아니라 문장을 감싼 비계입니다.
역 카탈로그에는 "for의 역"이라는 항목이 없고, 알맹이인 X(a[i])는 비계 밖으로 꺼내는 순간 i를 잃어 성립하지 않는 텍스트가 됩니다.
그래서 컴파일러는 이 쓰기를 정직하게 거부합니다.

감싸기 요령의 대가 — 담요 쓰기

남은 질문은 감싸기 요령이 공짜인가 하는 것입니다.
대가가 하나 따라옵니다.
Flip(a)처럼 레지스터를 통째로 넘긴 호출은 문장 하나가 레지스터 전체를 한 덩어리로 기록하는 담요 쓰기가 됩니다.
담요 쓰기 자체는 잘못이 아니지만, 원소 하나만 묻는 질의와 만나면 폭이 어긋나는 문제가 생깁니다.
이 폭의 불일치는 13장에서 다룹니다.
