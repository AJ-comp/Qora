1장에서 자동 uncomputation의 규칙을 세웠습니다.
측정하지 않은 use 레지스터는 스코프가 끝날 때 |0⟩로 되돌려 반납되어야 한다는 규칙입니다.
그런데 이 규칙을 지키려면 컴파일러는 먼저 어떤 큐빗을 되돌릴지부터 정해야 하고, 프로그램의 모든 큐빗을 되돌려서는 안 됩니다.
파라미터로 받은 큐빗은 호출한 쪽의 소유라 함부로 건드릴 수 없고, 측정한 큐빗은 그 값이 곧 프로그램의 결과라서 되돌리면 결과를 지우는 셈이 됩니다.
그리고 되돌려야 할 큐빗이라도, 계산 이력에 따라 자동으로 되돌릴 수 있는 경우와 없는 경우로 나뉩니다.
그래서 컴파일러는 큐빗마다 아래와 같은 세 가지 질문을 차례로 던집니다.

단계질문판별 기준API
0. 앤실라인가?누구의 큐빗인가use로 만들었으면 앤실라입니다. 파라미터는 호출자의 소유이므로 제외됩니다.IsAncilla
1. 청소 후보인가?측정되었는가측정되지 않은 앤실라만 청소 후보입니다. 측정된 앤실라는 되돌릴 역연산이 없어 청소할 수 없으므로 제외됩니다.IsCleanupCandidate
2. 청소 가능한가?역사가 깨끗한가계산 이력이 4개 조항을 통과해야 합니다 (9장–13장).UncomputeSafety

각 질문이 실제로 어떤 큐빗을 필터링하는지, 0번과 1번을 예제로 하나씩 확인하겠습니다.
(2번 질문은 9장부터 13장까지가 통째로 다룹니다.)

질문 0의 제외 사례 — 피호출 연산이 파라미터 큐빗을 되돌리면, 호출자가 그 큐빗에 준비한 상태가 사라집니다

번호 ①~⑤는 이 프로그램이 실제로 실행되는 순서입니다.
Main은 자신이 use로 만든 큐빗 q[0]을 Prepare에 인자로 넘겨 호출합니다. ① 주석
Prepare는 그 큐빗을 p라는 이름으로 받고, 이 p가 곧 Main의 q[0]입니다. ② 주석
Prepare는 그 p에 X 게이트를 적용해 p를 |1⟩ 상태로 만듭니다. ③ 주석
만약 컴파일러가 p를 청소 대상으로 오인하면, Prepare의 스코프가 끝날 때 p를 |0⟩로 되돌리는 역연산 X(p)를 삽입합니다. ④ 주석
다시 Main으로 돌아와 q[0]을 측정해 r에 담는데, 앞서 삽입된 역연산이 q[0]을 |0⟩로 되돌려 r이 기대한 1이 아니라 0이 됩니다. ⑤ 주석 

operation Prepare(Qubit p){    // ② Prepare가 그 큐빗을 p라는 이름으로 받는다 (p가 곧 Main의 q[0])
    X(p);                      // ③ p에 X 게이트를 적용해 p = |1⟩
    X(p);                      // ④ 청소 대상으로 오인하면 스코프 끝에 삽입되는 역연산 → p = |0⟩ (X가 취소됨)
}
operation Main(){
    use q = Qubit[1];
    Prepare(q[0]);             // ① Main이 use로 만든 q[0]을 Prepare에 넘겨 호출한다
    bit r = M(q[0]);           // ⑤ q[0]을 측정해 r에 담는다. 위 역연산 탓에 r = 1이 아니라 0
}

그래서 질문 0은 큐빗의 소유권을 기준으로 필터링합니다.
파라미터 큐빗은 그 소유자인 호출자만 정리할 수 있고, 피호출 연산은 자신이 use로 생성한 큐빗만 정리 대상으로 삼습니다.

질문 1의 제외 사례 — 연산이 측정한 큐빗은 정리 코드를 만들 수 없습니다

측정은 큐빗의 중첩 상태를 하나의 기저 값으로 붕괴시키는 연산인데, 이 붕괴는 유니터리 연산이 아니라서 역연산이 존재하지 않습니다.
자동 청소는 유니터리 게이트의 역연산을 거꾸로 적용해 큐빗을 |0⟩로 되돌리는 방식이므로, 되돌릴 역연산이 없는 측정된 큐빗은 원리적으로 청소할 수 없습니다.
8장의 상태벡터로 옮기면, 측정으로 삭제된 항의 내용은 어디에도 남지 않아 복원할 방법이 없다는 뜻입니다. 

use q = Qubit[1];
H(q[0]);
bit r = M(q[0]);     // q를 측정해 그 값을 r에 저장한다. r이 이 프로그램의 결과다

그래서 질문 1은 큐빗이 측정된 적이 있는지를 확인하고, 위 q처럼 측정된 큐빗은 청소 후보에서 제외합니다.

측정은 앤실라 여부를 바꾸지 않습니다 — 역할을 바꿉니다

측정된 use 레지스터도 앤실라로 태어났다는 사실은 변하지 않습니다.
다만 측정으로 그 값이 bit 변수에 읽히면, 그 큐빗은 계산에 쓰고 버리는 작업공간이 아니라 프로그램이 내놓는 결과가 됩니다.
측정으로 앤실라가 이렇게 출력이 되는 것을 출력으로 승격이라 부릅니다.
그래서 같은 큐빗도 상황에 따라 세 가지 용어로 구분해 부릅니다.
태생을 가리키는 말이 앤실라, 측정되지 않아 정리해야 하는 앤실라가 청소 후보, 판정까지 통과해 자동 정리가 확정된 후보가 safe입니다.

예제 — 세 질문의 답을 한 프로그램에서 한꺼번에

이 판별은 상상 속 절차가 아니라 지금 실행해 볼 수 있는 기능입니다.
컴파일러의 --stages 리포트가 큐빗마다 세 질문의 답을 그대로 출력합니다.
파라미터 하나, 측정되는 레지스터 하나, 청소 후보 하나가 모두 등장하는 작은 프로그램으로 확인해 보겠습니다. 

operation Flip(Qubit p){ X(p); }
operation Main(){
    use a   = Qubit[1];          // a를 만든다
    use out = Qubit[1];          // out을 만든다
    Flip(a[0]);                  // a에 X를 적용한다 (a = |1⟩)
    CNOT(a[0], out[0]);          // a의 값을 out에 복사한다. 이후 a는 쓰이지 않는다
    bit c = M(out[0]);           // out을 측정해 그 값을 c에 저장한다
}
// --stages 의 uncompute 뷰:
p:   input (parameter) — caller-owned, not an ancilla
out: ancilla, measured — promoted to output, not a cleanup candidate
a:   ancilla, cleanup candidate — safe to uncompute

이 프로그램에는 큐빗이 p, out, a 셋 있습니다.
아래 표의 각 행은 그중 한 큐빗이고, 가운데 칸은 그 큐빗이 세 질문(0 소유권 · 1 측정 여부 · 2 복원 가능성) 가운데 어디서 제외되었는지, 또는 셋 다 통과했는지를 나타냅니다.

큐빗판정사유
p질문 0에서 제외Flip이 use로 생성한 큐빗이 아니라 파라미터로 전달받은 큐빗이므로 앤실라에 해당하지 않습니다.
out질문 1에서 제외use로 생성한 앤실라이기는 하나, M(out[0])으로 측정되어 그 값이 고전 비트 c에 기록되었으므로 출력으로 승격됩니다.
a세 질문 모두 충족use로 생성하였고(질문 0 충족), 측정된 이력이 없으며(질문 1 충족), X 하나만으로 값을 구성하여 복원이 가능합니다(질문 2 충족). 따라서 청소 후보이자 safe로 판정됩니다.

a의 값은 CNOT으로 out에 복사된 뒤로는 아무도 a 자체를 읽지 않습니다.
클래식 코드의 t = ...; result = f(t);에서 t가 임시 변수인 것과 같은 상황입니다.
이런 큐빗이 청소 후보가 되고, 컴파일러는 a가 마지막으로 쓰인 CNOT 직후에 정리 코드를 자동으로 넣습니다.
그 정리 코드가 정확히 무엇이고 어떻게 만들어지는지는 11장에서 다룹니다.
