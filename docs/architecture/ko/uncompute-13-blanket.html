13. 담요 문제 — 질의 폭과 문장 폭

레지스터 통짜 쓰기와 원소 질문의 충돌

← 12. 루프와 호출 — 왜 판정이 달라지는가 코스 목차 14. 차단의 두 종류 — 원리적 한계와 보수적 거부 →

12장에서 루프를 연산으로 감싸 호출 문장 하나로 만드는 요령을 익히면서, 그 문장이 레지스터 전체를 한 덩어리로 기록하게 된다는 대가도 함께 보았습니다.
이렇게 한 문장이 레지스터 전체를 한 덩어리로 쓰는 것을 담요 쓰기라고 부릅니다.
브로드캐스트 X(a), 레지스터를 통째로 넘긴 호출, 인덱스가 컴파일 타임에 정해지지 않는 루프가 모두 여기에 해당합니다.
담요 쓰기 자체는 잘못이 아닙니다.
문제는 담요 쓰기가 원소 단위의 질문과 만날 때 생기며, 그 본질은 폭의 불일치 하나입니다.

여기서 폭은 두 종류입니다.
질의 폭이란 컴파일러가 "스코프가 끝나기 전에 |0⟩ 상태로 되돌릴 수 있는가"라고 묻는 대상의 크기를 말합니다.
레지스터 전체를 묻는 질의도 있고, a[1]처럼 원소 하나만 묻는 질의도 있습니다.
문장 폭이란 한 문장이 적용하는 유니터리가 실제로 작용하는 큐빗의 범위를 말합니다.
예를 들어 X(a)의 문장 폭은 a[0]과 a[1] 전부이고, X(a[0])의 문장 폭은 a[0] 하나입니다.
역연산은 문장이 적용한 유니터리의 역을 같은 큐빗 범위에 통째로 적용하는 것이므로, 언제나 문장 폭 그대로만 만들어집니다.

역연산은 문장 전체 단위로만 만들 수 있습니다.
따라서 청소해야 할 것은 원소 하나인데 그 원소를 편집한 문장이 레지스터 전체를 편집했다면, 그 문장의 역연산을 원소 하나로 축소할 방법이 없습니다.

기준선 — 전원이 스크래치인 담요 쓰기는 왜 문제없이 통과하는가

규칙만 보면 담요 쓰기는 늘 위험해 보이지만, 두 폭이 일치하는 동안에는 아무 문제도 없습니다.
먼저 그 기준선을 세워 두겠습니다.
아래 프로그램은 두 원소짜리 앤실라 a를 담요 호출 하나로 기록하고, 아무도 측정하지 않습니다. 

operation Bcast(Qubit[2] p){
    for i in 0..1 { X(p[i]); }
}
operation Main(){
    use a = Qubit[2];
    Bcast(a);            // 담요 쓰기 — 문장 폭 = a[0]·a[1] 전부
    // 아무도 a를 측정하지 않습니다 — 전원이 스크래치
}

a의 두 원소는 모두 측정되지 않았으므로 레지스터 전원이 청소 후보이고, 질의 폭은 레지스터 전체입니다.
a를 기록한 문장은 Bcast(a) 하나뿐이고, 그 역연산 Adjoint Bcast(a)의 문장 폭도 레지스터 전체입니다.
질의 폭과 문장 폭이 정확히 일치하므로 썰어낼 것이 없고, 11장의 주입기가 사망점에 Adjoint Bcast(a) 한 문장을 붙여넣는 것으로 정리가 끝납니다.
상태로 확인해 보면, Bcast(a)는 a의 기저 상태 |00⟩을 |11⟩로 보내고 Adjoint Bcast(a)는 |11⟩을 도로 |00⟩으로 보내므로, a는 정확히 |00⟩ 상태로 반납됩니다.
리포트도 같은 결론을 출력합니다. 

// --stages 의 uncompute 뷰:
a: ancilla, cleanup candidate — safe to uncompute

한 줄을 추가하면 — 25만 측정해 운명을 가릅니다

기준선이 통과한 이유는 두 원소의 운명이 같아서 질의 폭이 레지스터 전체였기 때문입니다.
그렇다면 두 원소의 처리가 나뉘는 순간에는 무슨 일이 벌어질까요.
기준선 프로그램에 딱 한 줄, a[0]만 측정하는 문장을 끼워 넣겠습니다. 

operation Bcast(Qubit[2] p){
    for i in 0..1 { X(p[i]); }
}
operation Main(){
    use a = Qubit[2];
    Bcast(a);                 // 담요 쓰기 — 문장 폭 = a[0]·a[1] 전부
    bit c = M(a[0]);          // ← 추가된 한 줄: a[0]만 측정
    // c = 1 (확률 100%) — Bcast가 a[0]을 |1⟩로 만들어 두었기 때문입니다
}

추가된 측정 하나가 두 원소의 처리를 서로 다르게 만들었습니다.
a[0]은 측정되어 그 값이 고전 비트 c로 읽혔으므로 출력으로 승격되어 청소 대상에서 빠집니다.
a[1]은 측정되지 않았으므로 여전히 청소 후보이고, 질의 폭은 이제 원소 하나인 a[1]로 줄었습니다.
그런데 a[1]을 기록한 문장은 기준선과 똑같은 담요 호출 Bcast(a)이고, 그 역연산의 문장 폭은 여전히 레지스터 전체입니다.
기준선에서 정확히 일치했던 두 폭이 처음으로 어긋났습니다.

어긋난 폭은 어떻게 처리되는가 — 리포트가 지목하는 원소

치워야 하는 것은 a[1] 하나뿐인데, 주입기가 붙여넣을 수 있는 역은 문장 전체의 역인 Adjoint Bcast(a)뿐입니다.
그 문장을 붙여넣으면 이미 측정되어 출력으로 승격된 a[0]까지 함께 다시 써 버립니다.
그래서 원소 폭 질의가 담요 쓰기를 만나면 컴파일러는 그 원소를 차단합니다.
위 프로그램을 컴파일하면 다음과 같은 판정이 나옵니다. 

a: ancilla, measured — promoted to output, not a cleanup candidate
   — per element: a[1] blocked: statement-wide write of `a` — its adjoint
     cannot be sliced to `a[1]` and would rewrite the sibling elements ...

리포트는 레지스터 요약과 함께 원소 단위의 진실도 보여줍니다.
막힌 원소에는 막힌 이유가 붙고, 깨끗한 원소에는 후보 표시가 붙습니다.
첫 항의 레지스터 요약만 보면 측정과 승격으로 끝난 것 같지만, per element 항이 a[1]은 담요 쓰기 때문에 막혔다는 사실과 그 이유까지 정확히 지목합니다.

이제 두 경우가 모두 눈앞에 있으니, 폭의 언어로 나란히 놓고 정리하겠습니다.

경우질의 폭역연산의 문장 폭판정
담요 + 전원 스크래치레지스터 전체 (a[0]·a[1])레지스터 전체✅ safe — Adjoint Bcast(a) 한 번으로 정리
담요 + 운명 혼합원소 하나 (a[1])레지스터 전체❌ a[1] 차단 — 역을 원소 하나로 썰 수 없음

그럼 어떻게 고쳐야 할까요 — 네 개의 수정 경로

리포트가 원인을 폭의 불일치로 지목했으니, 고치는 길도 폭을 기준으로 정리됩니다.
한 레지스터 안에서 원소마다 처리가 나뉘었는데 이력에 담요 쓰기가 남아 있다면, 다음 네 개의 경로 중 하나로 코드를 고칠 수 있습니다.
경로마다 위의 막힌 프로그램에 실제로 적용해 결과를 확인하겠습니다.

수정 ① — 구조 변경 (권장): 문장 폭을 원소 하나로 줄입니다

첫 번째 길은 원인 자체를 없애는 길입니다.
담요 호출 하나를 원소별 호출 두 개로 바꾸면 각 문장의 폭이 원소 하나로 줄어듭니다.
그러면 a[1]의 이력은 원소 폭 문장만으로 이루어지므로, 원소 폭 질의와 폭이 일치해 컴파일러가 자동으로 치울 수 있게 됩니다. 

operation FlipOne(Qubit p){ X(p); }
operation Main(){
    use a = Qubit[2];
    FlipOne(a[0]);           // 문장 폭 = a[0] 하나
    FlipOne(a[1]);           // 문장 폭 = a[1] 하나
    bit c = M(a[0]);         // c = 1 (확률 100%) — a[0]만 측정 → 출력으로 승격
    // a[1]: 이력이 원소 폭뿐이라 청소 후보로 판정되고,
    // 컴파일러가 마지막 사용 직후에 역연산을 자동 삽입합니다
}

같은 효과를 내는 다른 구조 변경으로, 운명이 다른 원소들을 처음부터 서로 다른 레지스터로 선언하는 방법도 있습니다.
어느 쪽이든 이력이 정밀해져서 컴파일러가 자동으로 치워 주므로, 이 길이 권장 경로입니다.

수정 ② — 남은 원소도 측정합니다: 질의 자체를 없앱니다

두 번째 길은 폭을 맞추는 대신 질의 자체를 없애는 길입니다.
a[1]도 측정하면 그 값이 고전 비트로 읽혀 출력으로 승격되고, 반납 의무가 사라집니다.
승격된 원소는 청소 후보가 아니므로 |0⟩ 상태로 되돌릴 수 있는지 묻는 질의가 애초에 발생하지 않습니다. 

// operation Bcast는 기준선과 동일합니다
operation Main(){
    use a = Qubit[2];
    Bcast(a);
    bit c = M(a[0]);          // c = 1 (확률 100%)
    bit d = M(a[1]);          // ← 추가: a[1]도 측정 → 출력으로 승격
    // d = 1 (확률 100%) — Bcast가 a[1]도 |1⟩로 만들어 두었기 때문입니다
}

이제 두 원소가 모두 출력으로 승격되어 레지스터에 청소 후보가 없고, 차단할 대상도 없습니다.
다만 측정은 프로그램의 의미를 바꾸는 실제 변경이므로, 그 값이 정말 결과로 필요할 때 선택하는 길입니다.

수정 ③ — 명시적 Reset: 비가역 지우기를 코드로 선언합니다

세 번째 길은 남은 원소를 Reset으로 강제로 |0⟩ 상태로 만드는 길입니다.
8장에서 본 대로 Reset 게이트는 값을 강제로 0으로 만들고 이력을 지우는 비가역 연산입니다. 

// operation Bcast는 기준선과 동일합니다
operation Main(){
    use a = Qubit[2];
    Bcast(a);
    bit c = M(a[0]);          // c = 1 (확률 100%)
    Reset(a[1]);              // ← 추가: 비가역 지우기를 코드로 선언
    // a[1]은 |0⟩로 — 이 프로그램에서는 얽힘이 없어 무해합니다
}

이 프로그램에서는 Bcast(a)가 만든 상태가 기저 상태 |11⟩ 하나뿐이라 a[1]이 어느 큐빗과도 얽혀 있지 않고, 따라서 Reset(a[1])은 a[1]만 |0⟩로 되돌리며 다른 큐빗은 건드리지 않습니다.
그러나 얽힌 큐빗에 Reset 게이트를 적용하면 측정과 똑같이 상태벡터가 하나의 기저 상태로 붕괴하여 얽혀 있던 다른 큐빗의 상태까지 함께 바뀌므로, 얽힘이 없음을 스스로 확신할 수 있을 때만 쓰는 길입니다.

수정 ④ — 직접 역연산: 이력을 손으로 되돌립니다

네 번째 길은 남은 원소의 이력을 손으로 되돌리는 길입니다.
Bcast(a)가 a[1]에 적용한 편집은 X 하나이고 X 게이트는 자기 자신이 역이므로, X(a[1]) 한 문장이 a[1]을 정확히 |0⟩로 되돌립니다. 

// operation Bcast는 기준선과 동일합니다
operation Main(){
    use a = Qubit[2];
    Bcast(a);
    bit c = M(a[0]);          // c = 1 (확률 100%)
    X(a[1]);                  // ← 추가: 손으로 쓴 역연산 — a[1]은 |0⟩로
}

이 길은 유니터리만 사용하므로, Reset 게이트와 달리 얽힘이 있어도 물리적으로 완전한 수정입니다.
컴파일러가 자동으로 하려던 일을 사용자가 손으로 대신한 것과 같습니다.

③과 ④는 물리적으로는 반납을 끝내지만, 컴파일러가 그 Reset 게이트가나 손 역연산을 "반납 완료"로 인정하는 규칙은 아직 없습니다.
따라서 지금 판정까지 바꾸는 수정은 ①과 ②이며, ③·④를 반납 완료로 인정하는 명시 선언문은 향후 제공됩니다.

네 경로를 한눈에 정리하면 다음과 같습니다.

수정코드의미
① 구조 변경 (권장)FlipOne(a[0]); FlipOne(a[1]); 원소별 호출, 또는 운명별로 레지스터 분리 선언이력이 정밀해져 컴파일러가 자동으로 치웁니다
② 측정bit d = M(a[1]);출력으로 승격되어 반납 의무가 사라집니다
③ 명시적 ResetReset(a[1]);비가역 지우기를 코드로 선언합니다 (얽힌 큐빗에서는 다른 큐빗까지 오염될 수 있으니 주의; 컴파일러 판정은 아직 바뀌지 않습니다)
④ 직접 역연산X(a[1]);물리적으로 완전한 수정입니다 (컴파일러 판정은 아직 바뀌지 않습니다)

그렇다면 이 담요 차단은 앤실라에 H 게이트를 적용해 벌어지는 차단과 같은 성격일까요.
같은 컴파일 에러라도 어느 언어도 넘을 수 없는 물리의 벽과, 현재 구현이 안전한 쪽으로 물러선 판정은 다르게 읽어야 합니다.
차단들을 이 기준으로 가려내는 일이 14장의 주제입니다.
