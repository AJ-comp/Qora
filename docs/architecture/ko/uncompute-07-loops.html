7. 루프와 호출 — 왜 판정이 갈리는가

문장 구조가 판정을 가르는 이유

← 6. 주입기 — 문장 붙여넣기 기계 코스 목차 8. 담요 문제 — 질의 폭과 문장 폭 →

같은 계산도 문장 구조에 따라 판정이 갈립니다. 6장의 "붙여넣기" 절차에 실제로 넣어 보면 이유가 보입니다.

벌거벗은 루프 — ❌ 차단 (ContainedWrite)

use a = Qubit[3];
for i in 0..2 { X(a[i]); }
// 붙여넣기 시도:
X(a[i]);  // ← i가 루프 밖에 없음
//    (성립하지 않는 텍스트)
// 게다가 문장 1개 ≠ 실행 3번

for는 "문장"이 아니라 문장을 감싼 비계입니다. 카탈로그에 "for의 역"이라는 항목이 없고, 알맹이 X(a[i])는 비계 밖으로 꺼내는 순간 의미를 잃습니다. 그래서 컴파일러는 정직하게 거부합니다.

연산으로 감싼 같은 루프 — ✅ safe

operation Flip(Qubit[3] p){
  for i in 0..2 { X(p[i]); }
}
use a = Qubit[3];
Flip(a);
Adjoint Flip(a);  // 붙여넣기 성립

Flip(a)는 어엿한 문장 하나이고 실행도 정확히 한 번입니다. 그 역 Adjoint Flip(a)도 문장 하나이며, 내부의 루프는 그 안에서 알아서 거꾸로 돕니다 (X(a[2]); X(a[1]); X(a[0])).

호출은 통째 역변환 단위입니다

호출을 원자 단위로 뒤집으면 callee 내부의 반복·구조가 뒤집힌 단위 안에서 그대로 미러링됩니다. 그래서 callee 내부의 루프는 차단 사유가 되지 않습니다 — ContainedWrite가 감시하는 것은 호출자 몸통에서 쓰기를 직접 감싼 컨테이너뿐입니다. 벌거벗은 루프를 지금 통과시키는 실용적 요령도 여기서 나옵니다: 연산으로 한 겹 감싸면 됩니다. 벌거벗은 루프 자체를 통째로 뒤집는 일반화(몸통이 전부 값 편집인 복합문을 하나의 단위로 역변환)는 계획된 개선입니다(9장).
