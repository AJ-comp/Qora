1. 앤실라와 uncomputation — 작업 큐빗은 왜 되돌려야 하는가

use로 만든 작업 큐빗과 자동 되돌리기

코스 목차 2. 컴파일러는 어떤 큐빗을 정리하는가 — 판별 사다리 →

자동 uncomputation은 컴파일러가 다 쓴 작업 큐빗을 |0⟩로 되돌려 주는 기능입니다.
그 출발점은 "작업 큐빗"이 무엇이고 왜 되돌려야 하는지를 아는 것입니다.
이 장에서는 작업 큐빗(앤실라)이 무엇인지 정의하고, 계산이 끝난 뒤 방치하면 왜 측정 결과가 망가지는지, 그래서 왜 |0⟩로 되돌려야 하는지를 봅니다.

앤실라(ancilla)는 계산 도중에만 잠깐 쓰이는 보조 큐빗으로, 클래식 프로그램에서 최종 결과를 만들기 위해 잠시 사용하는 임시 지역변수와 같은 역할을 합니다.
Qora에서는 use 문으로 만든 큐빗 레지스터가 앤실라에 해당하며, 이 레지스터는 해당 연산에만 속하고 항상 |0⟩ 상태로 시작합니다.
따라서 출발점만 놓고 보면 "0으로 초기화된 지역변수"와 정확히 같습니다.
언어마다 이름은 다르지만, Q#의 use, Quipper의 with_ancilla, Qiskit의 AncillaRegister는 모두 같은 개념을 가리킵니다. 

// 클래식:
int t = 0;              // 0으로 초기화된 지역 임시 변수

// 양자:
use t = Qubit[1];       // 이 연산이 소유하고 |0⟩로 시작하는 보조 큐빗 = 앤실라

클래식 지역변수와 다른 점은 단 하나입니다.
클래식 변수는 스코프가 끝나면 그냥 버리면 그만이지만, 양자 앤실라는 계산 도중 다른 큐빗과 얽히기 때문에 그냥 버릴 수 없습니다.
얽힌 채로 방치된 앤실라는 살아남은 큐빗들의 간섭을 망가뜨려서, 프로그램의 최종 결과 자체를 바꿔 버립니다.

예제 — 방치된 앤실라가 결과를 실제로 바꾸는 최소 사례

1단계 — 얽힘이 없을 때는 아무 문제가 없습니다

먼저 앤실라가 아예 없는 프로그램부터 보겠습니다.
H 게이트를 두 번 연달아 적용하면 상태는 원래대로 돌아옵니다.
첫 H 게이트가 |0⟩을 두 기저 상태(q가 0인 기저 상태와 1인 기저 상태)의 중첩으로 만들고, 두 번째 H 게이트에서는 |0⟩ 성분으로 가는 진폭 기여가 보강되는 반면 |1⟩ 성분으로 가는 진폭 기여는 상쇄되기 때문입니다. 

use q = Qubit[1];
H(q[0]);           // |0⟩을 두 기저 상태의 중첩으로 만듦
H(q[0]);           // |0⟩ 진폭 기여는 보강, |1⟩ 진폭 기여는 상쇄 (간섭)
bit m = M(q[0]);
// m = 0 (확률 100%) — 언제 돌려도 같은 결과

측정 결과는 언제나 0입니다.
이처럼 같은 기저 상태에 도착한 진폭 기여가 보강되거나 상쇄되는 간섭은 양자 계산의 동력이고, 이 프로그램은 그것이 정상 작동하는 기준선입니다.

2단계 — 얽힌 앤실라를 방치하면 같은 프로그램이 무너집니다

이제 위 프로그램에 딱 한 줄만 끼워 넣겠습니다.
두 H 사이에서 앤실라 t가 q의 값을 기록하게 하고(CNOT), t는 그 뒤로 읽지도 측정하지도 않고 그냥 내버려 둡니다. 

use q = Qubit[1];  use t = Qubit[1];
H(q[0]);
CNOT(q[0], t[0]);  // ← 추가된 한 줄: t가 q의 값을 기록 (얽힘)
H(q[0]);           // t는 그대로 방치
bit m = M(q[0]);
// m = 0 또는 1 (각 50%) — 결정론이 동전 던지기로

결과가 "항상 0"에서 "반반 무작위"로 변했습니다.
t를 읽은 적도 측정한 적도 없는데, 존재만으로 q의 결과를 바꿔 놓은 것입니다.
이유는 1단계의 간섭 조건이 깨졌기 때문입니다.
두 번째 H 게이트가 결과를 0으로 되돌리려면 q의 두 기저 상태 항에서 나온 진폭 기여가 같은 기저 상태에서 다시 만나 간섭해야 합니다.
하지만 t가 각 기저 상태 항에서 q의 값을 기록해 버립니다.
그 결과 q가 0인 기저 상태의 t는 0이고 q가 1인 기저 상태의 t는 1이므로, 두 기저 상태 항에서 나온 진폭 기여는 서로 다른 전체 기저 상태에 놓여 같은 기저 상태로 모이지 못하고 서로 간섭하지도 못합니다.
진폭 기여가 간섭하는 과정을 정확히 읽는 방법은 7장에서 다룹니다.

해결 — 버리기 전에 기록을 지우면 됩니다

문제의 원인이 t에 남은 기록이었으니, 해결책은 버리기 전에 그 기록을 지우는 것입니다.
2단계 프로그램의 두 번째 H 앞에 CNOT(q[0], t[0])를 한 번 더 적용하면 됩니다.
CNOT 게이트는 자기 자신이 역연산이므로, 두 번째 CNOT 게이트가 첫 번째 CNOT 게이트가 남긴 기록을 정확히 되돌립니다. 

use q = Qubit[1];  use t = Qubit[1];
H(q[0]);
CNOT(q[0], t[0]);  // t가 q의 값을 기록 (얽힘)
CNOT(q[0], t[0]);  // ← 추가된 정리: t의 기록을 지움 (얽힘 해제)
H(q[0]);
bit m = M(q[0]);
// m = 0 (확률 100%) — 1단계와 똑같이 복구

두 번째 CNOT 게이트가 실행되는 순간의 상태벡터를 따라가면, t가 |0⟩로 돌아가면서 얽힘이 풀리는 것을 확인할 수 있습니다.

시점상태벡터 (|q,t⟩ 전개)일어난 일
첫 CNOT 후|0,0⟩ ×1/√2 · |1,1⟩ ×1/√2t가 q를 기록 — 기저 상태마다 t 값이 다름 (얽힘)
정리 CNOT 후|0,0⟩ ×1/√2 · |1,0⟩ ×1/√2q=1인 기저 상태의 t만 도로 뒤집힘 — 모든 기저 상태에서 t=0 (얽힘 해제)
두 번째 H 후|0,0⟩ ×1두 기저 상태의 q 값이 같아져 간섭이 복원됨 — m = 0 확정

정리 CNOT 덕분에 모든 기저 상태에서 t가 0이 되어, 두 기저 상태 항의 t 값이 다시 같아졌습니다.
그래서 두 번째 H 게이트가 다시 두 기저 상태 항의 진폭 기여를 같은 기저 상태로 모아 상쇄시킬 수 있게 되고, 측정 결과는 1단계와 똑같이 100% 확률로 0이 됩니다.
이렇게 앤실라를 |0⟩로 되돌리는 일을 uncomputation이라 부르며, Qora는 이 정리 코드를 사용자가 작성하지 않아도 컴파일러가 자동으로 삽입합니다.
그래서 언어 규칙도 하나로 정해집니다.
측정하지 않은 use 레지스터는 스코프가 끝날 때 |0⟩로 반납되어야 합니다.
