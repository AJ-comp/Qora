14. 차단의 두 종류 — 원리적 한계와 보수적 거부

에러의 지도 — 물리의 벽과 구현의 여지

← 13. 담요 문제 — 질의 폭과 문장 폭 코스 목차 15. 왜 정적 컴파일인가 — 두 개의 런타임 →

지금까지 10장의 Y 차단, 12장의 ContainedWrite, 13장의 statement-wide 차단이라는 세 개의 컴파일 에러를 만났습니다.
그런데 이 에러들은 같은 무게가 아니었습니다.
10장은 "계약을 지킬 방법이 존재하지 않는다"는 단계별 계산으로 끝났고, 12장은 "연산으로 한 겹 감싸면 오늘도 통과한다"는 요령으로 끝났습니다.
존재하지 않는 것과 아직 못 하는 것은 사용자에게 전혀 다른 행동을 요구하므로, 에러를 받았을 때 가장 먼저 물어야 할 질문은 이 차단이 둘 중 어느 쪽인가 하는 것입니다.
판별 기준은 9장에서 정의한 계약(앤실라 복귀, 나머지 보존) 그 자체이며, 두 종류는 계약의 말로 다음과 같이 정의됩니다.

종류정의
🧱 원리적 한계앤실라를 |0⟩ 기저 상태로 되돌리는 동시에, 나머지 모든 큐빗의 양자 상태를 각 기저 상태의 진폭이 부호와 위상까지 일치하도록 보존하는 유니터리 연산이 수학적으로 존재하지 않는 차단입니다.
계약을 만족하는 회로 자체가 없으므로, 구현을 아무리 고쳐도, 어느 언어로 옮겨 적어도 이 차단은 사라지지 않습니다.
🔧 보수적 거부계약을 만족하는 유니터리 연산은 존재하지만, 현재 구현이 그 연산을 회로로 구성하는 절차를 아직 갖추지 못해 안전한 쪽인 차단으로 물러선 판정입니다.
차단의 근거가 물리가 아니라 구현의 도달 범위이므로, 원리상으로는 에러 없이 자동 청소될 수 있는 코드입니다.

상태벡터(3장)의 말로 옮기면 다음과 같습니다.
원리적 한계는 생존자들의 항 구성과 진폭을 그대로 둔 채 앤실라의 기저 값만 모든 항에서 0으로 만드는 편집이 아예 존재하지 않는 경우입니다.
보수적 거부는 그런 편집이 존재하는데, 지금의 주입기가 그 편집을 받아 적을 항 모르는 경우입니다.

🧱 원리적 한계는 계약을 만족하는 유니터리 연산이 존재하지 않는 차단이고, 🔧 보수적 거부는 그 유니터리 연산이 존재하지만 현재 구현이 아직 회로로 구성하지 못하는 차단입니다.

"존재하지 않는다"는 무엇으로 확인하는가 — 대표 사례 ①: H 스크래치

유니터리 연산이 존재하지 않는다는 것은 강한 주장이므로, 대표 사례 하나에서 직접 확인합니다.
앤실라 t에 H 게이트를 적용해 상태벡터의 한 항을 두 기저 상태로 나누고, 생존자 q가 CNOT 게이트로 각 기저 상태의 t 값을 기록하는 프로그램입니다. 

use q = Qubit[1];  use t = Qubit[1];
H(t[0]);            // t의 한 줄을 두 기저 상태로 나눔
CNOT(t[0], q[0]);   // 생존자 q가 각 기저 상태의 t 값을 기록
bit m = M(q[0]);    // q는 출력으로 승격, t는 청소 후보 → ❌ 차단

CNOT 직후의 상태벡터는 항마다 t 값이 다른, 곱으로 분해할 수 없는 얽힌 상태벡터입니다.
청소가 존재하려면 t만 만지는 연산으로 이 상태벡터를 "모든 항에서 t가 0"인 상태벡터로 바꿔야 합니다.
가장 유력한 후보인 "H 게이트를 한 번 더"를 실제로 적용해 보겠습니다.

청소 시도 전 — CNOT 직후

항t 값q 값진폭
1행001/√2
2행111/√2

H(t[0])를 한 번 더 적용한 후

항t 값q 값진폭
1행00+1/2
2행10+1/2
3행01+1/2
4행11−1/2

두 기저 상태에서 나온 기여가 같은 상태벡터 항에 모이기는커녕 항이 넷으로 늘었고, t의 기저 값에는 여전히 0과 1이 섞여 있습니다.
실패하는 것은 H 게이트만이 아닙니다.
한 큐빗에만 작용하는 유니터리 연산은 그 큐빗과 나머지 사이의 얽힘을 만들지도 없애지도 못하므로, 얽힌 t를 t만 만져서 모든 항에서 0인 상태로 되돌리는 회로는 존재하지 않습니다.
접으려면 q에 남은 기록까지 거꾸로 지워야 하는데, 그러면 생존자에게 복사된 값 자체가 사라져 계약의 2번 조건(나머지 보존)이 깨집니다.
계약을 지키는 청소가 존재하지 않으므로 차단이 유일한 답이고, 이 벽은 Qora가 아니라 물리가 세운 것이어서 어느 언어로 옮겨 적어도 같은 지점에서 같은 거부를 만납니다.

그럼 반대쪽은 어떤 모습인가 — 대표 사례 ②: 벌거벗은 루프

대표 사례 ①에서는 청소 회로 자체가 존재하지 않았습니다.
보수적 거부는 정반대의 상황에서 나옵니다.
벌거벗은 루프가 앤실라를 |111⟩로 만들었다면 X 세 번으로 완벽하게 |000⟩로 되돌아가고, 그 과정에서 진폭은 어느 시점에도 변하지 않으므로 계약을 만족하는 유니터리 연산이 분명히 존재합니다.
차단의 원인은 물리가 아니라 주입기의 도달 범위입니다.
11장의 주입기는 문장 단위로만 역을 만드는데 역 카탈로그에 "for의 역"이라는 항목이 없어서, 몸통이 값 편집뿐인 루프를 정직하게 거부합니다.
12장에서 각각 따로 확인한 두 결과를 나란히 놓으면, 계산은 같고 문장의 구조만 다르다는 것이 보입니다.

벌거벗은 루프 — ❌ 지금은 차단

use a = Qubit[3];  use out = Qubit[1];
for i in 0..2 { X(a[i]); }
CNOT(a[0], out[0]);
bit c = M(out[0]);
// a 청소 불가 (ContainedWrite)

연산으로 감싼 같은 계산 — ✅ safe

operation Flip(Qubit[3] p){
  for i in 0..2 { X(p[i]); }
}
use a = Qubit[3];  use out = Qubit[1];
Flip(a);
CNOT(a[0], out[0]);
Adjoint Flip(a);   // 자동 주입
bit c = M(out[0]);  // c = 1 (확률 100%)

오른쪽이 통과하는 이유는 Flip(a)가 역 카탈로그에 항목이 있는 호출 문장 하나이기 때문이고, 왼쪽이 막히는 이유는 같은 계산이 카탈로그에 없는 모양으로 적혀 있기 때문입니다.
문장 모양의 문제이므로 구현이 자라면 사라지는 차단입니다.
몸통이 전부 값 편집인 복합문을 하나의 단위로 역변환하는 확장이 들어오면, 왼쪽 코드도 감싸지 않은 채 통과합니다.

그럼 지금까지 만난 차단은 각각 어느 쪽인가 — 여덟 가지 차단의 지도

존재하지 않는 경우와 존재하는데 아직 못 하는 경우를 하나씩 확인했으니, 이제 여덟 가지 차단 전부를 같은 기준으로 분류할 수 있습니다.
각 행에는 그 판정을 손으로 따라갈 수 있는 장 또는 예제를 근거로 달았습니다.

차단성격근거
H·Rx·Ry 스크래치🧱 원리적 한계항을 쪼갠 값을 생존자가 기록한 순간부터 계약을 만족하는 청소가 존재하지 않습니다 (위 대표 사례 ①, 게이트 분류는 9장)
얽힌 타깃 위의 Y·CY🧱 원리적 한계역연산을 주입하면 생존자의 측정 결과가 100% 확률의 1에서 100% 확률의 0으로 바뀝니다 (10장의 단계별 상태벡터)
측정된 값🧱 원리적 한계붕괴에는 유니터리 역이 없고, 측정된 앤실라는 출력으로 승격되어 애초에 청소 후보가 아닙니다 (2장의 승격 규칙)
런타임 의존 값 (동적 인덱스·경계)🧱 정적 컴파일의 숙명컴파일 타임에 회로를 확정하는 모든 언어가 공유하는 한계입니다 (확정이 왜 물리적 강제인지는 15장)
벌거벗은 루프 (몸통이 값 편집뿐)🔧 보수적 거부지금은 연산으로 감싸면 통과하고, 복합문 통째 역변환으로 완화되면 감싸기 없이 통과합니다 (위 대표 사례 ②, 12장)
정적 담요 (리터럴 경계 루프·브로드캐스트)🔧 보수적 거부13장의 혼합 운명 예제가 이 차단이며, 정적 전개로 원소별 정밀 기록이 가능해지면 자동 청소로 전환됩니다
확정 기저값 위의 Y🔧 보수적 거부얽히지 않은 타깃 위의 Y 게이트가 무해함은 10장의 한 항짜리 예제에서 확인했으며, 얽히지 않았음을 그래프로 증명하면 재허용 가능합니다
측정 bit 조건의 if 안 쓰기🔧+🧱 혼합역 문장 자체는 만들 수 있지만 "조건이 변하지 않았다"는 증명이 추가로 필요합니다 (바로 아래 예제)

마지막 행만 두 성격이 겹치므로, 가장 작은 예제로 겹침의 정체를 확인합니다.
측정 bit를 조건으로 쓴 if 안의 쓰기는, 같은 조건을 단 같은 문장을 한 번 더 붙여넣는 것으로 되돌릴 수 있습니다. 

use a = Qubit[1];  use q = Qubit[1];
H(q[0]);
bit m = M(q[0]);            // m: 0/1 각 50% — q는 출력으로 승격
if (m == 1) { X(a[0]); }    // a의 쓰기가 측정 bit m에 걸려 있음
if (m == 1) { X(a[0]); }    // ← 역 문장 (X는 자기 자신이 역)
// m = 0이면 두 문장 다 실행되지 않고, m = 1이면 X가 두 번 실행됩니다 — 두 경우 모두 a는 |0⟩

역 문장 자체는 이렇게 만들 수 있습니다.
그러나 이 역이 옳으려면 쓰기 시점부터 청소 시점까지 조건 값 m이 변하지 않았다는 증명이 추가로 필요하고, 그 증명이 갖춰지기 전까지 컴파일러는 안전한 쪽인 차단으로 물러섭니다.
역변환의 존재와 증명의 요구가 한 차단 안에 겹쳐 있으므로, 이 행만 🔧+🧱 혼합으로 분류됩니다.

여덟 항의 지도를 보고 나면 — 에러를 내는 것이 왜 정직한 동작인가

지도를 완성하고 나면, 이렇게 많은 경우를 차단하는 설계 자체가 결함은 아닌가 하는 의문이 남습니다.
답은 에러 대신 할 수 있는 일을 나란히 놓으면 나옵니다.
컴파일러에게 남은 대안은 둘뿐입니다.
청소를 조용히 건너뛰면 "측정하지 않은 use 레지스터는 |0⟩로 반납된다"는 언어 규칙이 거짓이 되고, 억지로 역연산을 주입하면 10장에서 본 것처럼 생존자의 결과가 바뀝니다.
자동으로 보증할 수 없는 것을 소리 내어 알리고, 13장의 수정 경로처럼 사용자에게 선택지를 돌려주는 것이 유일하게 정직한 동작입니다.
