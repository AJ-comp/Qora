10. 왜 정적 컴파일인가 — 두 개의 런타임

호스트와 QPU — 인터프리터가 들어갈 수 없는 곳

← 9. 차단의 두 종류 — 원리적 한계와 보수적 거부 코스 목차 11. 언어 지형 속의 Qora →

양자 프로그램의 "런타임"은 하나가 아니라 둘이며, 이 분리가 언어 설계 전체를 결정합니다. 하나는 프로그램을 준비하고 최종 결과를 받아 보는 호스트로, Python 인터프리터 같은 보통의 소프트웨어가 사는 곳입니다. 다른 하나는 QPU로, 큐빗이 양자 상태를 유지하는 짧은 시간, 즉 코히어런스 창 안에서만 회로를 실행할 수 있는 장치입니다. 두 세계는 물리적으로 다른 기계일 뿐 아니라, 무엇보다 시간의 단위가 다릅니다.

호스트 런타임QPU 런타임
장소노트북/서버 CPU — Python 등이 사는 곳코히어런스 창 안 — 회로가 실행되는 곳
시간 규모밀리초마이크로초
인터프리터 입장가능물리적으로 불가 (호스트 왕복은 창보다 천 배 이상 느림)

예제 — 측정 결과에 따라 같은 창 안에서 위상을 고쳐야 하는 프로그램

이 시간 차이가 실제로 무엇을 의미하는지, 가장 작은 프로그램으로 따라가 보겠습니다. 아래 프로그램에서 q[1]은 중첩 상태로 준비되어, q[0]의 측정 결과에 따라 위상 보정(Z)을 기다립니다. 보정이 제때 적용되기만 하면 마지막 측정 결과 r는 언제나 m과 같습니다. 

operation Main(){
    use q = Qubit[2];
    H(q[1]);                    // q[1]을 중첩으로 준비 — 기다리는 동안 "상할 수 있는" 상태
    H(q[0]);
    bit m = M(q[0]);            // 측정 — 여기서 시계가 돌기 시작합니다
    if (m == 1) { Z(q[1]); }    // 결과가 1이면 같은 코히어런스 창 안에서 위상 보정
    H(q[1]);                    // 보정이 제때 됐다면 중첩이 확정값으로 접힘
    bit r = M(q[1]);            // 제때 실행되면 항상 r == m
}

핵심은 q[1]이 기다리는 동안 들고 있는 것이 중첩의 위상 관계라는 점입니다. 위상 관계는 코히어런스 시간 안에서만 유지되고, 시간이 지나면 무작위로 흐트러집니다. 측정이 끝난 순간부터, if 한 줄을 누가 실행하느냐에 따라 시간을 따라가 보겠습니다.

경과 시간호스트의 인터프리터가 if를 실행컴파일되어 QPU 옆 제어장치에 배치
0측정 결과 m이 QPU 옆 제어 전자장치에 잡힘동일
수 µsm을 호스트로 전송하는 중 — 그동안 q[1]이 들고 있는 위상 관계가 코히어런스 창과 함께 소모됨미리 배치된 조건 로직이 m을 확인하고 즉시 Z 펄스 발사 — 같은 창 안에서 보정 완료, r == m
수 ms인터프리터가 if (m == 1)을 평가하고 Z 명령을 회신 — 그러나 마이크로초 단위의 창은 이미 닫힌 지 오래이고, q[1]의 위상 관계는 사라진 뒤라 r는 m과 무관한 동전 던지기가 됨이미 실행이 끝난 뒤
밀리초는 마이크로초의 천 배입니다

호스트 왕복 한 번이 밀리초 단위라면, 그 답이 돌아오는 동안 마이크로초 단위의 코히어런스 창은 천 번 넘게 열렸다 닫히고도 남습니다. 큐빗은 호스트의 답을 기다려 주지 않고 먼저 죽습니다.

그래서 위 예제의 if처럼 측정 결과에 곧바로 반응하는 회로 내 반응 로직은, 컴파일되어 QPU 옆 제어장치에 미리 배치되어 있어야만 실행될 수 있습니다. Qora가 이 if를 호스트 코드로 남겨 두지 않고 회로 명령의 일부로 컴파일해 내보내는 이유가 여기에 있습니다. 이 요구는 하드웨어가 커질수록 강해집니다. 오류정정 시대의 하드웨어는 모든 논리 연산에 이런 반응, 즉 실시간 신드롬 디코딩과 조건부 정정을 요구하므로, 컴파일은 선택이 아니라 물리적 강제가 됩니다.

시뮬레이터에서는 왜 이 문제가 보이지 않는가

시뮬레이터 위에서는 두 런타임이 같은 프로세스입니다. 측정 결과도 그냥 메모리에 있는 값이라서, 위 예제의 if를 호스트 언어의 if로 실행해도 아무런 시간 압박이 없습니다. 인터프리터 기반 언어의 우아함은 정확히 이 조건 위에서 성립하고, 실기계로 가는 순간 그 동일성이 깨집니다. 동적 인덱스나 런타임 경계처럼 "컴파일 타임에 모르는 값"의 문제 (8·9장)는 바로 이때 태어납니다. 큐빗 수를 컴파일 타임에 확정해야 하는 것도 같은 이유에서 나옵니다. 실기계의 큐빗은 칩에 고정된 자원이라서, 컴파일러가 모든 레지스터의 물리 배치를 미리 정해 두어야 하기 때문입니다.
