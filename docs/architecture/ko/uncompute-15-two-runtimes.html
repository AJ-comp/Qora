15. 왜 정적 컴파일인가 — 두 개의 런타임

호스트와 QPU — 인터프리터가 들어갈 수 없는 곳

← 14. 차단의 두 종류 — 원리적 한계와 보수적 거부 코스 목차 16. 언어 지형 속의 Qora →

14장의 차단 지도에는 성격이 다른 행이 하나 있었습니다.
동적 인덱스와 런타임 경계의 차단은 물리의 벽도 구현의 미비도 아니라, 컴파일 타임에 회로를 확정하는 모든 언어가 공유하는 정적 컴파일의 숙명으로 분류되었습니다.
그렇다면 질문이 하나 남습니다.
왜 그 숙명을 짊어지면서까지 회로를 컴파일 타임에 확정해야 할까요.
답은 양자 프로그램이 실행되는 환경 자체에 있습니다.
양자 프로그램의 "런타임"은 하나가 아니라 둘이며, 이 분리가 언어 설계 전체를 결정합니다.

하나는 호스트입니다.
프로그램을 준비하고 최종 결과를 받아 보는 보통의 컴퓨터로, Python 인터프리터 같은 일반 소프트웨어가 사는 곳입니다.
다른 하나는 QPU입니다.
큐빗이 양자 상태, 즉 중첩을 이루는 각 기저 상태의 진폭과 그 사이의 위상 관계를 유지하는 동안에만 회로를 실행할 수 있는 장치입니다.
이 유지 시간의 구간을 코히어런스 창이라 부르며, 창이 닫히면 위상 관계가 무작위로 흐트러져 중첩에 기대던 계산이 더 이상 성립하지 않습니다.
두 런타임은 물리적으로 다른 기계일 뿐 아니라, 무엇보다 시간의 단위가 다릅니다.

호스트 런타임QPU 런타임
장소노트북/서버 CPU — Python 등이 사는 곳코히어런스 창 안 — 회로가 실행되는 곳
시간 규모밀리초마이크로초

예제 — 측정 결과에 따라 같은 창 안에서 위상을 고쳐야 하는 프로그램

밀리초와 마이크로초의 차이가 실제로 무엇을 부수는지, 가장 작은 프로그램으로 따라가 보겠습니다.
아래 프로그램에서 q[1]은 H 게이트로 중첩 상태 (|0⟩+|1⟩)/√2 로 준비되어, q[0]의 측정 결과에 따라 위상 보정(Z)을 기다립니다.
보정이 제때 적용되기만 하면 마지막 측정 결과 r는 언제나 m과 같습니다. 

operation Main(){
    use q = Qubit[2];
    H(q[1]);                    // q[1]을 중첩으로 준비 — 기다리는 동안 "상할 수 있는" 상태
    H(q[0]);
    bit m = M(q[0]);            // 측정 — 여기서 시계가 돌기 시작합니다
    if (m == 1) { Z(q[1]); }    // 결과가 1이면 같은 코히어런스 창 안에서 위상 보정
    H(q[1]);                    // 보정이 제때 됐다면 중첩이 확정값으로 접힘
    bit r = M(q[1]);            // 제때 실행되면 항상 r == m
}

보정이 제때 적용되었을 때 왜 항상 r == m이 되는지는, m이 0인 경우와 1인 경우를 각각 따라가면 직접 확인할 수 있습니다.

m보정보정 직후 q[1]의 상태H(q[1]) 후r
0없음(|0⟩+|1⟩)/√2|0⟩0 (확률 100%)
1Z 적용 — 부호 반전(|0⟩−|1⟩)/√2|1⟩1 (확률 100%)

핵심은 q[1]이 기다리는 동안 들고 있는 것이 두 진폭 사이의 위상 관계, 즉 표의 두 행을 가르는 바로 그 + / − 부호라는 점입니다.
위상 관계는 코히어런스 창 안에서만 유지되고, 창이 닫히면 무작위로 흐트러집니다.
그러므로 이 프로그램의 운명은 if 한 줄이 어느 시계 위에서 실행되느냐에 달려 있습니다.
두 경우를 차례로 보겠습니다.

1단계 — 기준선: 14가 컴파일되어 QPU 옆 제어장치에 배치된 경우

Qora가 이 프로그램을 컴파일하면 if (m == 1) { Z(q[1]); }는 회로 명령의 일부가 되어, 실행이 시작되기 전에 QPU 바로 옆의 제어장치에 미리 적재됩니다.
측정이 끝난 순간부터 시간을 따라가 보겠습니다.

경과 시간일어나는 일q[1]의 위상 관계
0측정 결과 m이 QPU 옆 제어장치에 잡힘살아 있음
수 µs미리 배치된 조건 로직이 m을 확인하고 즉시 Z 펄스 발사 — 같은 코히어런스 창 안에서 보정 완료살아 있음 — 보정 반영됨
직후이어지는 H 게이트가 중첩을 확정값으로 접고, 측정이 그 값을 읽음 — r == mH로 접힌 뒤라 더 이상 필요 없음

모든 게이트가 창이 닫히기 전에 끝났으므로, 위 표에서 두 경우를 계산한 그대로 r == m이 확률 100%로 성립합니다.
이것이 기준선입니다.
측정 결과에 대한 결정이 같은 창 안에서 내려지는 한, 프로그램은 결정론적으로 동작합니다.

2단계 — 딱 하나만 바꿉니다: 같은 21를 호스트의 인터프리터가 실행하면

이제 프로그램은 한 글자도 고치지 않고, 실행 방식 하나만 바꿉니다.
if 한 줄을 회로로 컴파일해 내려보내는 대신, 호스트에 남겨 두고 인터프리터가 실행하게 합니다. 

    bit m = M(q[0]);
    if (m == 1) { Z(q[1]); }    // ← 이 한 줄만 호스트의 인터프리터가 실행하도록 바뀜
    H(q[1]);

이렇게 바꾸는 순간 m은 QPU 옆을 떠나 호스트까지 왕복 여행을 해야 합니다.
같은 시계로 다시 따라가 보겠습니다.

경과 시간일어나는 일q[1]의 위상 관계
0측정 결과 m이 QPU 옆 제어장치에 잡힘 — 1단계와 완전히 동일살아 있음
수 µsm을 호스트로 전송하는 중코히어런스 창과 함께 소모되는 중
수 ms인터프리터가 if (m == 1)을 평가하고 Z 명령을 회신 — 그러나 마이크로초 단위의 창은 이미 닫힌 지 오래무작위로 흐트러진 뒤

Z 명령이 도착했을 때는 고쳐야 할 위상 관계 자체가 이미 무작위로 흐트러진 뒤입니다.
1단계에서 r == m을 보장하던 것은 표의 두 행에 적힌 +와 − 부호가 살아 있다는 사실이었는데, 그 부호가 사라졌으므로 이어지는 H 게이트는 더 이상 상태를 |0⟩이나 |1⟩ 하나로 접지 못합니다.
그 결과 r는 m과 무관하게 0과 1이 각각 50%로 나오는 동전 던지기가 됩니다.
1단계와 2단계는 경과 시간 0의 순간까지 완전히 같았습니다.
달라진 것은 결정이 내려지는 장소 하나뿐인데, 그 장소의 시계가 밀리초 단위라서 마이크로초 단위의 창을 통째로 넘겨 버린 것입니다.

밀리초는 마이크로초의 천 배입니다

호스트 왕복 한 번이 밀리초 단위라면, 그 답이 돌아오는 동안 마이크로초 단위의 코히어런스 창은 천 번 넘게 열렸다 닫히고도 남습니다.
큐빗은 호스트의 답을 기다려 주지 않고 먼저 죽습니다.

정리 — 두 실행을 한 표로 겹쳐 보면

두 경우를 각각 따라왔으니, 이제 같은 시계 위에 겹쳐 놓으면 분기하는 지점이 정확히 보입니다.

경과 시간컴파일되어 QPU 옆 제어장치에 배치 (1단계)호스트의 인터프리터가 if를 실행 (2단계)
0측정 결과 m이 QPU 옆 제어장치에 잡힘동일
수 µs미리 배치된 조건 로직이 m을 확인하고 즉시 Z 펄스 발사 — 같은 창 안에서 보정 완료, r == mm을 호스트로 전송하는 중 — 그동안 q[1]이 들고 있는 위상 관계가 코히어런스 창과 함께 소모됨
수 ms이미 실행이 끝난 뒤인터프리터가 if (m == 1)을 평가하고 Z 명령을 회신 — 그러나 마이크로초 단위의 창은 이미 닫힌 지 오래이고, q[1]의 위상 관계는 사라진 뒤라 r는 m과 무관한 동전 던지기가 됨

이 대조를 두 런타임의 말로 요약하면, 장의 첫머리에서 본 표가 마지막 한 행을 얻어 완성됩니다.

인터프리터 입장가능물리적으로 불가 (호스트 왕복은 창보다 천 배 이상 느림)

그래서 왜 정적 컴파일인가 — 반응 로직은 미리 배치되어야 합니다

2단계의 실패에서 결론이 곧바로 나옵니다.
위 예제의 if처럼 측정 결과에 곧바로 반응하는 회로 내 반응 로직은, 컴파일되어 QPU 옆 제어장치에 미리 배치되어 있어야만 실행될 수 있습니다.
Qora가 이 if를 호스트 코드로 남겨 두지 않고 회로 명령의 일부로 컴파일해 내보내는 이유가 여기에 있습니다.
이 요구는 하드웨어가 커질수록 강해집니다.
오류정정 시대의 하드웨어는 모든 논리 연산에 이런 반응, 즉 실시간 신드롬 디코딩과 조건부 정정을 요구하므로, 컴파일은 선택이 아니라 물리적 강제가 됩니다.
14장이 정적 컴파일의 숙명이라 부른 동적 값 차단은 바로 이 강제의 대가이며, 회로를 컴파일 타임에 확정해야만 실기계에서 실행할 수 있으므로 Qora는 그 대가를 받아들입니다.

시뮬레이터에서는 왜 이 문제가 보이지 않을까요

시뮬레이터 위에서는 두 런타임이 같은 프로세스입니다.
측정 결과도 그냥 메모리에 있는 값이라서, 위 예제의 if를 호스트 언어의 if로 실행해도 아무런 시간 압박이 없습니다.
인터프리터 기반 언어의 우아함은 정확히 이 조건 위에서 성립하고, 실기계로 가는 순간 그 동일성이 깨집니다.
동적 인덱스나 런타임 경계처럼 "컴파일 타임에 모르는 값"의 문제 (8·14장)는 바로 이때 태어납니다.
큐빗 수를 컴파일 타임에 확정해야 하는 것도 같은 이유에서 나옵니다.
실기계의 큐빗은 칩에 고정된 자원이라서, 컴파일러가 모든 레지스터의 물리 배치를 미리 정해 두어야 하기 때문입니다.
