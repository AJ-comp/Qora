2. ModuleLoader — 여러 파일을 하나의 프로그램으로 병합

경로 해석 · 재귀 import · 중복 제거 · 순환 진단 · 후속 패스 계약

← 1. QoraLowering 패스 목차

ModuleLoader는 사용자가 작성한 import 경로를 따라가며 여러 .qor 파일을 읽고, 후속 패스가 파일 시스템을 다시 확인하지 않아도 되도록 하나의 QProgram으로 합칩니다.
아래에서는 컴파일러 내부 코드 대신 같은 Qora 프로젝트가 단계마다 어떻게 처리되는지를 바로 따라갑니다.

PASS SPECIFICATIONModuleLoader
이 패스가 하는 일

엔트리 파일의 import부터 시작해 도달 가능한 파일을 읽고, 중복 파일은 한 번만 포함하며, 아직 처리 중인 파일로 되돌아오는 순환은 오류로 보고합니다.
이어서 파일별 연산과 open 정보를 병합해 다음 패스가 읽을 프로그램 하나를 만듭니다.

1

import를 찾으려면 기준 폴더가 필요합니다

INPUT · C:\work\app\main.qor
import "lib/prepare.qor";
import "lib/readout.qor";

operation Main() {
  use q = Qubit[1];
  Demo.Prepare(q);
  Demo.Readout(q);
}
저장된 main.qor를 컴파일한 경우 컴파일러는 main.qor가 있는 폴더를 알고 있으므로 두 import를 그 폴더에서 찾기 시작합니다. 
import 기준 폴더
C:\work\app

다음 처리
lib/prepare.qor부터 확인
import 기준 폴더가 없는 입력으로 전달한 경우 상대 경로를 시작할 위치가 없으므로 파일을 하나도 열지 않고 import 두 곳에 각각 QSEM020을 기록합니다. 
추가로 읽은 파일  없음
Operations        [Main]
Imports           두 경로 그대로 유지
Errors            [QSEM020, QSEM020]
import가 전혀 없는 독립적인 입력 
operation Main() {
  use q = Qubit[1];
  H(q[0]);
}
이 코드에는 추가로 읽을 파일이 없으므로 ModuleLoader는 입력 프로그램을 그대로 반환하며 오류도 만들지 않습니다.

구현에서는 import를 찾기 시작할 기준 폴더를 baseDir에 담습니다.

2

상대 경로는 import를 작성한 파일에서 시작합니다

엔트리 파일 C:\work\app\main.qor 
import "lib/prepare.qor";
import "lib/readout.qor";
현재 import 파일 C:\work\app\lib\prepare.qor 
import "shared/gates.qor";

namespace Demo {
  open Shared;

  operation Prepare(Qubit[] q) {
    H(q[0]);
    Mark(q);
  }
}
main.qor의 폴더
C:\work\app
+ lib/prepare.qor
→
C:\work\app\lib\prepare.qor
prepare.qor의 폴더
C:\work\app\lib
+ shared/gates.qor
→
C:\work\app\lib\shared\gates.qor

여기서 중요한 점은 상대 경로의 기준이 항상 main.qor의 폴더로 고정되는 것은 아니라는 점입니다.
컴파일러는 import 문을 만날 때마다 그 import 문이 들어 있는 파일의 폴더에서 경로 계산을 시작합니다.
따라서 main.qor에 적힌 lib/prepare.qor는 C:\work\app에서 출발합니다.
반면에 prepare.qor에 적힌 shared/gates.qor는 prepare.qor가 들어 있는 C:\work\app\lib에서 출발하므로, 최종 경로는 C:\work\app\lib\shared\gates.qor가 됩니다.
만약 이 경로도 main.qor의 폴더에서 계산한다면 C:\work\app\shared\gates.qor라는 전혀 다른 파일을 찾게 됩니다.

계산을 마친 절대 경로는 이후 각 파일을 구별하는 이름표 역할을 합니다.
컴파일러는 같은 절대 경로가 현재 불러오는 중인 경로에 있으면 순환으로 판단하고, 이미 읽으려고 시도한 경로에 있으면 중복으로 판단합니다.

3

아직 처리 중인 파일로 돌아오면 순환입니다

sourcePath가 제공된 파일 컴파일에서 잠시 바꾼 C:\work\app\lib\shared\gates.qor
import "../../main.qor";

namespace Shared {
  operation Mark(Qubit[] q) {
    Z(q[0]);
  }
}
현재 따라가던 경로와 실제 진단 
main.qor
  → prepare.qor
    → gates.qor
      → main.qor  QSEM021

실제 진단
imports form a cycle: main.qor -> prepare.qor -> gates.qor -> main.qor — remove one of these imports

이 결과는 저장된 main.qor 자체를 컴파일하여 엔트리 경로가 sourcePath로 제공된 경우를 기준으로 합니다.
마지막 import가 가리키는 main.qor는 아직 prepare.qor와 gates.qor의 처리가 끝나기를 기다리는 상위 파일입니다.
따라서 단순한 재사용이 아니라 순환으로 판단하며, 구현에서는 이렇게 현재 처리가 끝나지 않은 파일 경로를 chain에 기록합니다.
순환 검사는 중복 검사보다 먼저 수행하므로 main.qor가 이미 알려진 경로라는 이유로 조용히 건너뛰지 않습니다.
sourcePath 없이 기준 폴더만 제공한 호출에서는 엔트리 경로를 미리 등록할 수 없으므로 순환을 발견하는 시점과 표시 경로가 달라질 수 있습니다.
다음 단계부터는 gates.qor에서 이 순환 import를 제거한 정상 코드로 돌아갑니다.

4

다른 경로에서 다시 만난 파일은 한 번만 읽습니다

C:\work\app\lib\prepare.qor
import "shared/gates.qor";

namespace Demo {
  open Shared;

  operation Prepare(Qubit[] q) {
    H(q[0]);
    Mark(q);
  }
}
C:\work\app\lib\readout.qor
import "shared/gates.qor";

namespace Demo {
  open Shared;

  operation Readout(Qubit[] q) {
    Mark(q);
  }
}
prepare.qor → gates.qor처음 만난 절대 경로이므로 파일을 읽고 병합합니다.
readout.qor → gates.qor이미 시도한 같은 절대 경로이므로 다시 읽지 않습니다.
관찰되는 결과gates.qor는 한 번만 병합되며 오류는 없습니다.

두 import는 서로 다른 파일에 적혀 있지만 모두 C:\work\app\lib\shared\gates.qor를 가리킵니다.
구현에서는 한 번이라도 처리 대상으로 등록한 절대 경로를 loaded에 남겨 두며, 이 기록은 직접 중복과 위와 같은 다이아몬드 공유에서 같은 파일을 다시 읽는 일을 막습니다.

같은 파일 안의 직접 중복도 같은 규칙 
import "shared/gates.qor";
import "shared/./gates.qor";
두 표기는 정규화하면 같은 절대 경로가 되므로 첫 번째 import만 파일을 읽고 두 번째 import는 건너뜁니다.
5

가져온 파일 안의 import도 먼저 끝까지 처리합니다

정상 상태의 C:\work\app\lib\shared\gates.qor
namespace Shared {
  operation Mark(Qubit[] q) {
    Z(q[0]);
  }
}
1. main.qor첫 번째 import인 prepare.qor를 읽습니다.
2. prepare.qorPrepare를 바로 합치지 않고, 그 안의 gates.qor를 먼저 처리합니다.
3. gates.qor파일을 Qora IR로 바꾸고 Mark를 병합할 준비를 마칩니다.
4. prepare.qor로 복귀자식 처리가 끝났으므로 이제 Prepare를 병합합니다.
5. readout.qorgates.qor는 이미 처리했으므로 생략하고 Readout을 병합합니다.
파일은 읽었지만 문법이 잘못된 경우 
// broken.qor
operation Broken(Qubit[] q) {
  H(q[0])  // 세미콜론 누락
}
import 파일을 파싱할 수 없으면 그 파일의 연산과 open, 자식 import는 병합하지 않습니다.
대신 원래 파서 오류 코드와 파일 이름을 오류 목록에 남기고 같은 단계의 다음 import를 계속 확인합니다.
6

자식 파일의 연산을 먼저 합치고 open 중복을 제거합니다

두 import 파일에 반복해서 적힌 외부 Qora 선언
// prepare.qor와 readout.qor의 Demo namespace
namespace Demo {
  open Shared;
  // 각 파일의 operation이 이어짐
}
병합된 연산 순서 
Operations
1. Main      Namespace: (global)
2. Mark      Namespace: Shared
3. Prepare   Namespace: Demo
4. Readout   Namespace: Demo
병합된 open 
prepare.qor   Demo → [Shared]
readout.qor   Demo → [Shared]
gates.qor     Shared → []
                 ↓ 같은 Target 제거
최종          Demo → [Shared]
              Shared → []

엔트리 파일의 Main은 처음부터 목록 앞에 보존됩니다.
이어서 prepare.qor의 자식인 gates.qor에서 Mark를 먼저 추가하고, 재귀 처리가 돌아온 뒤 Prepare를 추가합니다.
두 번째 gates.qor 방문은 생략되므로 Mark는 한 번만 들어가며, 마지막으로 Readout이 추가됩니다.
또한 두 import 파일의 open Shared;는 모두 Demo 네임스페이스에 속하므로 최종 목록에는 같은 Target이 한 번만 남습니다.
gates.qor의 Shared 네임스페이스에는 open이 없지만, 네임스페이스가 존재한다는 빈 목록 Shared → []도 함께 남습니다.

import와 open의 차이 import "lib/prepare.qor";는 파일을 실제로 읽어 프로그램에 포함합니다.
반면에 open Shared;는 이미 포함된 Shared 네임스페이스의 이름을 짧게 찾도록 도울 뿐이며, 파일을 읽지는 않습니다.
7

후속 패스와 QASM 생성 중단

누락 경로는 prepare.qor에서 처음 만났을 때 QSEM020을 만들며, readout.qor에서 같은 절대 경로를 다시 만나도 이미 시도한 경로이므로 같은 오류를 반복하지 않습니다.
다만 prepare.qor와 readout.qor 자체는 정상적으로 읽었으므로 두 연산은 부분 결과에 남습니다.
경로 기준이 있는 확장 과정에 들어간 뒤에는 성공 여부와 관계없이 루트 Imports를 null로 소비하지만, 오류가 하나라도 있으면 현재 호출자인 QoraParser.Parse가 그 부분 결과를 후속 패스에 전달하지 않습니다.
반면에 1단계처럼 import 기준 폴더 자체가 없어서 조기에 끝난 경우에는 원래 Imports가 그대로 남습니다.

import 파일의 위치를 엔트리 Span으로 사용하지 않는 이유

QoraError.Start와 End는 편집기에 열려 있는 엔트리 문서의 문자 위치를 뜻합니다.
import 파일에서 계산한 위치를 그대로 넣으면 편집기가 그 숫자를 엔트리 문서의 위치로 오해하여 다른 부분에 오류 표시를 그리게 됩니다.
따라서 import 파일의 IR 노드는 위치 없이 만들며, import 파일의 파싱 오류도 파일 이름은 메시지에 남기되 위치는 -1..-1로 반환합니다.

후속 패스에 넘기는 계약

항목ModuleLoader가 준비하는 값다음 패스가 이어서 하는 일
연산 목록 엔트리 연산을 먼저 보존하고, import 파일 연산을 자식 우선 순서로 한 번씩 추가합니다.
 전체 연산을 대상으로 측정 조건 변환, 이름 해소, 의미 검증을 수행합니다.
파일 경계 경로 기준이 있는 확장 결과의 루트 Imports를 null로 만듭니다.
 후속 패스는 파일 시스템에 접근하지 않고 QProgram 하나만 읽습니다.
네임스페이스 파일마다 나뉜 open을 선언 네임스페이스별로 합치고 같은 Target을 제거합니다.
 Resolver가 Namespace와 Opens를 사용해 짧은 호출 이름의 대상을 확정합니다.
오류 성공한 파일을 포함한 부분 결과와 지금까지 모은 오류를 함께 반환합니다.
 현재 호출자는 오류가 하나라도 있으면 후속 패스와 QASM 생성을 실행하지 않습니다.
연산 이름과 본문 파일을 병합할 뿐 연산 이름을 완전 수식 이름으로 바꾸거나 본문을 재작성하지 않습니다.
 Resolver와 각 전용 IR 패스가 필요한 이름과 노드만 이후에 변환합니다.

정리하면 ModuleLoader는 Qora 소스에 적힌 파일 관계를 따라가며 읽을 수 있는 내용을 하나의 프로그램에 모으고, 정상 공유와 순환을 구분한 뒤 다음 패스가 사용할 전체 프로그램 문맥을 준비합니다.
이름의 의미를 확정하거나 연산 본문을 바꾸는 작업은 이 단계에서 수행하지 않습니다.
