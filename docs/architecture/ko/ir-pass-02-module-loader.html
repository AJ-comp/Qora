2. ModuleLoader — 여러 파일을 하나의 프로그램으로 병합

이전 패스의 단일 파일 IR을 import가 반영된 통합 IR로 확장

← 1. QoraLowering 패스 목차 3. MeasureConditionLowering →

1장에서 QoraLowering은 엔트리 파일 하나를 QProgram으로 만들었습니다.
ModuleLoader는 이 QProgram과 baseDir·sourcePath를 입력받아, Imports가 가리키는 파일을 하나의 QProgram으로 병합합니다.

PASS SPECIFICATIONModuleLoader
입력

입력: QProgram, baseDir, sourcePath

ModuleLoader는 Imports[0..1]과 두 경로 값을 입력으로 받되, 각 값은 해당 조건에 도달했을 때만 사용합니다.
이미 만들어진 Operations[0]의 Main 본문은 변경하지 않습니다.

1

Imports를 확인하고, 필요한 경우에만 baseDir를 확인합니다

먼저 program.Imports가 비어 있는지 확인합니다.1.1
처리할 import가 없으면 입력 QProgram을 그대로 반환합니다.1.1.1
첫 검사를 통과하면 Imports가 있다는 뜻이므로, 그때 baseDir를 확인합니다.1.2
여기서 baseDir가 없으면 파일을 하나도 읽지 않고 각 QImport에 QSEM020을 기록한 뒤, 원래 Imports가 남은 입력 QProgram을 반환합니다.1.2.1
두 검사를 모두 통과하면 빈 loaded를 만들고, sourcePath를 절대 경로로 정규화해 엔트리 main.qor를 먼저 등록합니다.
이 준비가 끝난 뒤 첫 import의 경로 계산으로 넘어갑니다.

2

QImport.Target을 현재 파일 기준의 절대 경로로 계산합니다

Imports는 소스 순서대로 방문하며, 각 QImport.Target을 현재 파일의 기준 폴더와 결합한 뒤 Path.GetFullPath로 정규화합니다.2.1
이 계산은 경로 문자열을 절대 경로 형태로 정리할 뿐이며, 해당 위치에 파일이 실제로 존재하는지는 아직 확인하지 않습니다.
경로를 결합하거나 정규화할 수 없으면 해당 import에 QSEM020을 기록하고 같은 목록의 다음 import로 넘어갑니다.2.1.1
이 예제에서는 Target = "lib/prepare.qor"에서 계산한 full = C:\work\app\lib\prepare.qor이 3절의 입력이 됩니다.

3

계산한 full을 loaded에 등록해 첫 방문인지 확인합니다

3절은 파일 하나를 통째로 처리하는 단계가 아니라, 현재 파일의 QImport 하나마다 실행되는 검사입니다.
1절은 두 검사를 통과한 뒤 sourcePath를 정규화하여 엔트리 main.qor를 loaded에 먼저 등록합니다.
이어서 2절이 첫 import의 절대 경로 full을 계산하면, 3절은 1절의 loaded와 2절의 full을 전달받아 첫 방문 여부를 판정합니다.

loaded.Add(full)은 같은 절대 경로가 있는지 확인하고, 없으면 full을 집합에 넣은 뒤 true를 반환합니다.3.1
true이면 호출이 끝난 시점에 경로 등록까지 완료되어 있으며, continue를 실행하지 않고 4절로 진행합니다.3.1.1
반면에 false이면 이미 등록된 경로이므로 continue를 실행하여 현재 QImport를 끝냅니다.3.1.2
그러면 foreach는 같은 Imports 목록에 다음 QImport가 있는지 확인합니다.

YES 다음 QImport 있음

같은 목록의 다음 QImport를 선택합니다.

2절 · 경로 계산으로 이동

NO 다음 QImport 없음

현재 LoadAll 호출을 종료합니다.

재귀 호출6절 · 병합으로 이동
엔트리 호출7절 · 최종 반환으로 이동

두 분기 모두 1절의 초기 검사는 다시 실행하지 않습니다.

4

처음 만난 import 파일의 AST를 만들고, 그 AST를 자식 QProgram으로 옮깁니다

ModuleLoader가 처음 전달받는 program은 1장에서 엔트리 main.qor의 Janglim AST를 QoraLowering으로 옮겨 만든 QProgram입니다.
반면에 엔트리의 QImport에는 Target = "lib/prepare.qor"라는 경로 문자열만 있으므로, prepare.qor의 AST와 자식 QProgram은 4절에서 파일을 처음 읽은 뒤에 만들어집니다.

먼저 2절의 Path.GetFullPath는 경로 문자열만 정규화했으므로, 실제 디스크의 파일 존재 여부를 이 절의 File.Exists에서 처음 확인합니다.4.1
loaded.Add(full) = true인 새 경로에 파일이 있으면 File.ReadAllText로 내용을 읽어 src 문자열을 얻습니다.4.2
이 src는 아직 IR이 아니라 import 파일의 원본 소스 문자열입니다.

이어서 QoraParser.ParseToIr(src)가 먼저 Janglim lexer와 parser로 src를 읽어 해당 import 파일의 Janglim AST를 만듭니다.
파싱에 성공하면 QoraLowering.Lower(ast, withSpans: false)를 호출하여 그 AST를 import 파일 하나를 나타내는 자식 QProgram으로 옮깁니다.4.3
따라서 ParseToIr가 문자열을 곧바로 IR로 바꾸는 것이 아니라, 함수 내부에서 소스 → AST → Qora IR의 두 경계를 차례로 통과한다고 이해해야 합니다.

ParseToIr 내부에서 수행하는 범위

ParseToIr는 전체 컴파일 파이프라인을 끝까지 다시 실행하는 함수가 아니라, 새 import 파일을 자식 QProgram으로 만드는 단일 파일 변환 구간만 수행하는 경량 진입점입니다.
ParseToIr 내부에서 수행: 렉싱, Janglim 파싱, AST 생성, QoraLowering, 단일 파일 QProgram 생성.
ParseToIr 밖에서 이어서 수행: 자식 Imports의 재귀 확장은 현재 ModuleLoader가 담당하며, 이름 해소·검증·후속 패스·QASM 생성은 병합이 끝난 뒤에 실행합니다.
엔트리 main.qor는 이 과정에서 다시 변환하지 않습니다.

파일이 없으면 QSEM020을 기록하고 현재 import를 끝냅니다.4.1.1
파일을 읽을 수 없는 경우에도 QSEM020을 기록하고 현재 import를 끝냅니다.4.2.1
ParseToIr 안의 Janglim 파싱에 실패하면 QoraLowering은 호출하지 않으며, 원래 파서 오류 코드에 import 파일 이름을 붙여 기록합니다.4.3.1
이 세 경우에는 같은 목록의 다음 QImport가 있는지 확인하고, 다음 항목이 있으면 2절의 경로 계산부터 반복합니다.

빈 파일은 파싱할 AST 루트가 없으므로 ParseToIr가 sub = null을 반환하며, 현재 import에는 병합할 내용이 없는 것으로 처리합니다.4.3.2
반면에 AST가 만들어지면 QoraLowering이 Imports·Operations·Opens를 담은 sub, 즉 현재 import 파일 하나의 자식 QProgram을 반환합니다.4.3.3

ParseToIr는 여기서 작업을 마치며, sub.Imports의 대상 파일까지 내부에서 읽지는 않습니다.
자식 Imports가 있으면 바깥의 ModuleLoader가 prepare.qor의 폴더를 기준 경로로 전달하고, 각 자식 import를 2장 2절의 경로 계산부터 재귀적으로 처리합니다. 이때 1절의 초기 검사는 반복하지 않습니다.4.4
자식 import가 없으면 재귀 호출을 생략하며, 어느 경로든 다음에는 6절에서 현재 prepare.qor의 Operations와 Opens를 전체 결과에 병합합니다.

5

이미 등록된 경로를 다시 만나면 현재 import를 건너뜁니다

4절에서 만든 자식 QProgram의 Imports에도 2절의 경로 계산과 3절의 loaded 등록을 같은 순서로 적용합니다.
이때 계산한 full이 이미 loaded에 있으면 loaded.Add(full)이 false가 되며, 해당 파일을 다시 읽지 않고 현재 import만 건너뜁니다.

5.1 · 사례 A · 순환 재방문
main.qor → prepare.qor → gates.qor 순서로 경로를 등록하며 재귀적으로 들어간 상태에서, 현재 처리 중인 gates.qor가 ../../main.qor를 import하는 경우입니다.

5.2 · 사례 B · 공유 경로 재방문
순환이 없어도 서로 다른 부모가 같은 자식 파일을 가리키면 같은 검사가 작동합니다.
아래에서는 prepare.qor와 readout.qor가 모두 같은 gates.qor를 가리킵니다.

사례 A에서는 엔트리 main.qor로 돌아가는 import를 건너뜁니다.5.1.1
사례 B에서는 readout.qor가 다시 가리킨 gates.qor import를 건너뜁니다.5.2.4
두 경우 모두 판단 기준은 정규화된 절대 경로 full이며, loaded에는 처음 만난 경로만 추가됩니다.

6

자식 처리가 끝난 뒤 현재 파일의 Operations와 Opens를 병합합니다

이제 5절의 사례 B에서 각 경로의 방문 여부를 판정했으므로, 실제 필드가 어떤 순서로 병합되는지 확인합니다.
이 예제에서 main.qor는 Main, prepare.qor는 Prepare, gates.qor는 Mark, readout.qor는 Readout을 제공합니다.
또한 prepare.qor와 readout.qor에는 각각 open Shared;가 있습니다.

결과 연산 목록은 엔트리 Operations의 복사본으로 시작하므로 Main은 처음부터 맨 앞에 보존됩니다.
각 import 파일은 자식 그래프를 먼저 끝낸 뒤 자신의 연산을 추가하며, 같은 단계의 import는 원래 Imports 순서를 유지합니다.

prepare.qor의 연산은 자식 gates.qor 처리가 끝난 뒤 추가되므로 Mark가 Prepare보다 앞에 놓입니다.6.1
Opens는 namespace별로 합친 뒤 같은 QOpen.Target을 하나만 남기므로, 두 open Shared;는 Demo → [Shared]로 정리됩니다.6.2

7

전체 또는 부분 QProgram을 오류 목록과 한 쌍으로 반환합니다

Expand의 반환값은 처리 결과와 오류 목록을 묶은 (Program, Errors) 한 쌍입니다.
아래에서는 모든 import를 처리한 반환값과 gates.qor를 찾지 못한 반환값을 각각 완전한 튜플로 비교합니다.

모든 import를 처리하면 전체 연산과 Errors = []를 함께 반환합니다.7.1
gates.qor를 찾지 못하면 Mark를 제외한 부분 Program과 QSEM020을 함께 반환합니다.7.2
QoraParser는 Errors.Count = 0일 때만 Program을 다음 패스로 전달하며, 오류가 있으면 후속 패스를 실행하지 않습니다.7.3

baseDir 엔트리 파일에 작성된 import의 기준 폴더
예: C:\work\app		 엔트리의 각 QImport.Target과 결합해 절대 경로를 계산합니다. IR에는 저장하지 않습니다.
sourcePath 엔트리 파일 자체의 경로
예: C:\work\app\main.qor		 엔트리의 정규화된 절대 경로를 처음부터 loaded에 등록하여 자식이 엔트리를 다시 import해도 파일을 재시도하지 않게 합니다. IR에는 저장하지 않습니다.
Imports가 있을 때 null이어도 baseDir가 있으면 확장은 진행되지만 엔트리 사전 등록은 할 수 없습니다. 이 경우 엔트리로 돌아오는 첫 import는 새 경로처럼 처리되어, 엔트리 파일을 한 번 더 읽고 병합할 수 있습니다.
값을 전달한다면 Path.GetFullPath로 처리할 수 있는 경로여야 하며, 이 등록 단계의 예외는 QSEM020으로 변환되지 않습니다.
반환 (Program, Errors) 병합된 전체 또는 부분 QProgram과 오류 목록 파싱·병합한 파일의 IR과 import 처리 오류를 함께 보존합니다. Errors가 비어 있을 때만 호출자가 Program을 다음 단계로 전달합니다.

재귀 처리 상태와 검사 순서

dir은 현재 import를 작성한 파일의 폴더이며, 각 QImport에는 다음 순서를 적용합니다.

QImport 하나의 처리 순서 
Path.Combine(dir, QImport.Target) → Path.GetFullPath
├─ 실패 → QSEM020 → 다음 import
└─ 성공 → loaded.Add(full)
          ├─ false → 순환·중복 import 생략 · 진단 없음
          └─ true  → File.Exists
                     ├─ false → QSEM020 → 다음 import
                     └─ true  → ReadAllText
                                ├─ 실패 → QSEM020 → 다음 import
                                └─ 성공 → ParseToIr(src)
                                           ├─ Janglim 파싱 실패
                                           │  └─ 파일 이름과 함께 파서 오류 기록 → 다음 import
                                           └─ Janglim 파싱 성공
                                              ├─ AST 없음 · 빈 파일 → sub == null → 다음 import
                                              └─ AST 있음
                                                 └─ QoraLowering.Lower(ast, withSpans: false)
                                                    └─ sub · 현재 import 파일의 QProgram
                                                       ├─ 자식 Imports 있음 → ModuleLoader 재귀 처리
                                                       └─ 재귀가 없거나 끝남
                                                          → 현재 파일의 Operations·Opens 병합
상태담는 값역할
dir 현재 처리 중인 Imports를 작성한 파일의 디렉터리 중첩 import가 엔트리가 아니라 자신을 작성한 파일을 기준으로 해석됩니다.
full Path.GetFullPath(Path.Combine(dir, QImport.Target))으로 계산한 절대 경로 loaded가 같은 파일을 비교하는 식별자가 됩니다.
QImport.Target에는 다시 저장하지 않습니다.
loaded 정규화에 성공해 이미 한 번 처리 대상으로 등록한 절대 경로 loaded에 이미 등록된 상위 파일로 되돌아가는 import, 직접 중복, 여러 부모가 공유하는 파일, 같은 누락·읽기·파싱 실패를 다시 처리하지 않도록 막습니다.
경로 비교에서는 대소문자를 무시합니다.

생략과 오류 처리

상황진단현재 import 처리반환 QProgram
Imports = null 또는 Count = 0 진단 없음 baseDir·sourcePath를 확인하지 않고 파일 I/O 없이 종료 입력 program을 그대로 반환합니다.
import가 있으나 baseDir 없음 각 import에 QSEM020 파일을 하나도 읽지 않고 즉시 종료 입력 program과 원래 Imports를 그대로 반환합니다.
절대 경로 계산 실패 해당 import에 QSEM020 같은 목록의 다음 import로 이동 나머지 import의 처리를 계속하며, 이미 병합한 결과는 보존합니다.
순환 또는 중복 경로
loaded.Add(full) = false		 진단 없음 현재 import를 생략하고 같은 목록의 다음 import로 이동 지금까지 누적한 결과를 변경하지 않으며, 해당 경로의 파일 I/O를 다시 실행하지 않습니다.
파일 없음 또는 읽기 실패 QSEM020 현재 import를 생략 경로가 이미 loaded에 있으므로 같은 실패를 다시 보고하지 않습니다.
import 파일 파싱 실패 원래 파서 오류 코드
메시지에 파일 이름
위치 -1..-1		 그 파일의 자식 import·연산·open을 모두 생략 앞서 병합한 다른 파일의 값은 보존합니다.
ParseToIr 결과가 null 진단 없음 빈 파일로 보고 같은 목록의 다음 import로 이동 지금까지 누적한 결과를 그대로 보존합니다.
import된 파일 안에서 발생한 오류 위치가 -1..-1인 이유

QoraError.Start와 End는 편집기에 열려 있는 엔트리 문서의 문자 위치를 뜻합니다.
ParseToIr는 import 파일의 Janglim AST를 만든 뒤 QoraLowering.Lower(ast, withSpans: false)로 옮기므로, 반환된 IR 노드에는 엔트리 문서의 위치로 오해할 수 있는 span을 남기지 않습니다.
반면에 Janglim 파싱 오류 자체에는 해당 import 파일 안의 위치가 들어 있습니다.
이 숫자를 엔트리 문서의 위치로 오해하지 않도록 ModuleLoader가 원래 오류 코드와 메시지를 보존해 새 오류를 만들고, 파일 이름을 붙인 뒤 위치를 -1..-1로 바꿉니다.
같은 이유로 위치 정보가 없는 중첩 QImport에서 발생한 QSEM020도 -1..-1입니다.

반환 필드와 패스 경계

반환 필드ModuleLoader가 보장하는 값다음 단계에 남기는 판단
Operations 엔트리 연산을 먼저 보존하고, import 파일은 같은 단계의 소스 순서와 각 파일의 자식 우선 순서로 한 번씩 추가합니다. 연산 이름과 본문은 바꾸지 않으며, 그 해석은 후속 패스가 맡습니다.
Opens 파일별 값을 namespace별로 합친 뒤, 대소문자까지 같은 QOpen.Target의 중복을 제거해 처음 항목 하나만 남깁니다. ModuleLoader는 대상 namespace나 연산의 존재 여부를 판정하지 않습니다.
Imports Imports가 하나 이상이고 baseDir가 있어 확장 경로에 진입하면, 처리 오류 유무와 관계없이 루트 Imports = null입니다. 후속 패스는 파일을 다시 읽지 않고 병합된 QProgram만 사용합니다.
Errors 부분 병합 결과와 지금까지 수집한 import 처리 오류를 함께 반환합니다. 호출자는 오류가 없을 때만 후속 패스를 실행합니다.

ModuleLoader는 loaded에 이미 등록된 순환·중복 경로를 다시 열지 않으며, 처음 만난 경로 가운데 ParseToIr 결과가 있는 파일의 IR만 import 그래프의 순서에 따라 하나의 QProgram으로 병합합니다.
