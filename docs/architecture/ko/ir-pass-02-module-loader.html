2. ModuleLoader — 여러 파일을 하나의 프로그램으로 병합

경로 해석 · 재귀 import · 중복 제거 · 순환 진단 · 후속 패스 계약

← 1. QoraLowering 패스 목차

QoraLowering은 소스 파일 한 개를 QProgram으로 변환합니다.
그러나 엔트리 파일이 다른 파일을 import하면, 그 시점의 QProgram에는 엔트리 파일에 선언된 연산만 들어 있고 import 대상 파일의 연산은 아직 들어 있지 않습니다.
따라서 이름을 해소하거나 중복 정의를 검사하기 전에 모든 import 파일을 읽어 하나의 프로그램으로 합쳐야 합니다.
ModuleLoader가 바로 이 파일 병합을 담당합니다.

PASS SPECIFICATIONModuleLoader
이 패스가 하는 일

ModuleLoader는 엔트리 QProgram.Imports에 남아 있는 경로를 따라가며, 도달 가능한 파일을 하나의 QProgram에 편입합니다.
단순히 파일 내용을 이어 붙이는 것이 아니라, 각 경로의 기준 디렉터리를 계산하고 정상적인 중복과 실제 순환을 구분한 뒤, 파일별 연산과 open 정보를 정해진 순서로 병합합니다.

  1. 처리할 import와 상대 경로의 기준이 있는지 먼저 확인합니다.Imports가 비어 있으면 입력 프로그램을 그대로 반환합니다.
    import가 있지만 baseDir가 없으면 파일을 전혀 읽지 않고 각 import에 QSEM020을 기록하며, 이때 원래 Imports도 그대로 남습니다.
  2. 현재 파일의 import를 소스 순서대로 절대 경로로 바꿉니다.baseDir가 있으면 import를 작성한 파일의 디렉터리와 QImport.Target을 결합하고 GetFullPath로 정규화합니다.
    특정 경로를 만들 수 없으면 해당 import에 QSEM020을 기록하고 다음 import로 넘어갑니다.
  3. chain을 먼저 확인하여 순환을 찾습니다.대상 절대 경로가 현재 처리 중인 재귀 경로에 있으면 상위 파일로 되돌아간 것이므로 QSEM021을 생성합니다.
    이 검사를 중복 검사보다 먼저 수행하므로 순환을 단순 재사용으로 놓치지 않습니다.
  4. loaded로 이미 시도한 파일을 다시 읽지 않습니다.순환이 아닌 새 경로는 파일 존재 확인 전에 loaded에 등록합니다.
    이미 등록된 경로라면 직접 중복이나 다이아몬드 공유로 보고 건너뛰며, 같은 읽기 실패도 한 번만 보고합니다.
  5. 새 파일을 읽어 단일 파일 IR로 바꾸고, 그 파일의 import를 재귀적으로 처리합니다.파일을 읽은 뒤 QoraParser.ParseToIr로 QProgram을 만듭니다.
    자식 import가 있으면 현재 파일을 chain에 넣고, 현재 파일의 디렉터리를 새 기준으로 전달하여 같은 절차를 반복합니다.
  6. 자식 import가 끝난 뒤 현재 파일의 연산과 open을 병합합니다.엔트리 연산은 처음부터 목록 앞에 보존됩니다.
    같은 단계의 import는 소스 순서대로 방문하지만, 각 import 하위 그래프에서는 자식 파일의 연산을 먼저 추가한 뒤 그 자식을 import한 파일의 연산을 추가합니다.
    open은 namespace별 목록에 누적합니다.
  7. 병합 결과와 지금까지 수집한 오류를 함께 반환합니다.namespace별 open은 같은 Target의 중복을 제거하고, 경로 기준이 있었던 확장 경로에서는 처리한 import를 Imports = null로 소비합니다.
    오류가 하나라도 있으면 호출자는 후속 패스를 실행하지 않습니다.
패스를 통과하기 전과 후
BEFORE

엔트리 파일 하나의 QProgram

  • Operations와 Opens에는 엔트리 파일에서 lowering한 내용만 들어 있습니다.
  • Imports에는 아직 해석하지 않은 상대 경로가 소스 순서대로 남아 있습니다.
  • import 대상 파일의 연산과 namespace 정보는 현재 프로그램에 포함되지 않았습니다.
  • 같은 파일을 여러 경로에서 가리키는지, import 순환이 있는지도 확인되지 않았습니다.
→
AFTER

import 그래프를 반영한 병합 결과

  • Operations는 엔트리 연산으로 시작합니다.
    같은 단계의 import는 소스 순서를 따르며, 각 import 하위 그래프에서는 자식 파일의 연산이 그 자식을 import한 파일의 연산보다 먼저 한 번씩 추가됩니다.
  • Opens는 namespace별로 합쳐지며 같은 Target은 한 번만 남습니다.
  • 경로 문맥이 있는 확장 경로에서는 처리할 import를 모두 소비하고 Imports = null로 반환합니다.
  • 누락 파일, 읽기 실패, 파싱 오류, 순환은 오류 목록에 누적되며, 목록이 비어 있을 때만 후속 패스로 넘어갑니다.

1파일 병합이 필요한 이유

Qora 소스는 여러 파일로 나눌 수 있지만, 후속 IR 패스는 파일을 직접 읽지 않습니다.
예를 들어 엔트리 파일의 Main이 import 파일에 선언된 Prepare를 호출한다면, 이름 해소 패스는 Prepare의 선언을 같은 QProgram 안에서 찾을 수 있어야 합니다.
그러므로 ModuleLoader는 파일 경계를 먼저 제거하고, 엔트리 파일과 모든 import 파일의 연산을 Operations 한 목록에 모읍니다.
이때 파일 하나를 점으로, 파일 사이의 import 관계를 화살표로 나타낸 전체 연결 구조를 import 그래프라고 부릅니다.

import와 open의 차이 import "lib/shared.qor";는 파일을 찾아 읽고, 그 파일에 선언된 연산과 네임스페이스 정보를 현재 프로그램에 포함시킵니다.
반면에 open Shared;는 이미 프로그램에 포함된 Shared 네임스페이스의 이름을 짧게 사용할 수 있게 할 뿐이며, 파일을 읽지는 않습니다.
따라서 다른 파일에 있는 이름을 사용하려면 먼저 import로 파일을 포함해야 하며, 필요한 경우 그다음에 open을 적용해야 합니다.
// app/main.qor
import "lib/prepare.qor";

namespace App {
  open Library;

  operation Main() {
    use q = Qubit[1];
    Prepare(q);
  }
}

// app/lib/prepare.qor
namespace Library {
  operation Prepare(Qubit[1] q) {
    H(q[0]);
  }
}

위 예제에서 open Library;만 남기고 import를 제거하면 Library 자체가 프로그램에 포함되지 않습니다.
반대로 import가 정상적으로 처리되면 Main과 Prepare가 같은 Operations 목록에 들어갑니다.
그 결과 Resolver는 Prepare(q)가 Library.Prepare(q)를 뜻한다는 사실을 판단할 수 있습니다.

QoraLowering — 엔트리 파일 한 개를 QProgram으로 변환
▼
단일 파일 QProgram — Imports에 아직 처리하지 않은 파일 경로가 남아 있음
▼
ModuleLoader.Expand 각 import 경로를 그 문장을 작성한 파일의 위치를 기준으로 절대 경로로 정규화합니다.
이어서 chain으로 순환을 먼저 판별하고, loaded로 이미 시도한 파일의 중복 읽기를 막습니다.
새 파일만 단일 파일 IR로 변환한 뒤 그 파일의 자식 import부터 확장하며, 자식 연산에서 부모 연산으로 이어지는 순서로 Operations에 추가합니다.
마지막으로 namespace별 open 목록을 합치고 같은 Target을 제거합니다.
▼ 오류가 없을 때만
MeasureConditionLowering — 병합된 모든 연산 본문에 동일한 변환 적용
▼
Resolver — 전체 Operations와 Opens를 기준으로 호출 이름 해소

2입력값과 경로 문맥

public static (QProgram Program, List<QoraError> Errors) Expand(
    QProgram program,
    string? baseDir,
    string? sourcePath)

여기서 엔트리 파일은 사용자가 컴파일 대상으로 직접 지정한 첫 파일을 뜻합니다.
반면에 import 파일은 엔트리 파일 또는 다른 import 파일이 경로로 참조한 파일을 뜻합니다.
두 종류의 파일을 구분해야 하는 이유는 상대 경로의 기준이 항상 import 문장을 작성한 파일의 위치이기 때문입니다.

인자 / 반환값담고 있는 값필요한 이유
program 엔트리 파일 하나를 lowering한 QProgram입니다.
엔트리 파일의 Operations, 소스 순서대로 기록된 Imports, 네임스페이스별 Opens를 포함합니다.
 병합 결과의 기준이 되며, 엔트리 연산을 최종 목록의 앞부분에 보존하는 데 사용됩니다.
baseDir 엔트리 파일에 적힌 상대 import를 해석할 기준 디렉터리입니다.
예를 들어 엔트리 경로가 C:\work\app\main.qor라면 일반적으로 C:\work\app이 전달됩니다.
 첫 번째 import 경로를 절대 경로로 바꾸는 데 필요합니다.
null이면 파일을 찾을 기준이 없으므로 import마다 QSEM020을 생성합니다.
sourcePath 엔트리 파일 자체의 경로입니다.
예: C:\work\app\main.qor		 엔트리 파일로 되돌아오는 import를 순환으로 즉시 알아내는 데 사용됩니다.
선택값이므로 null일 수 있지만, 제공되면 정규화한 경로를 loaded와 chain에 미리 등록합니다.
Program 엔트리 연산과 읽어 온 파일의 연산·open 정보를 합친 QProgram입니다.
 오류가 없으면 다음 패스의 입력으로 사용됩니다.
다만 오류가 있더라도 진단을 위해 현재까지의 부분 병합 결과가 반환될 수 있습니다.
Errors 경로, 파일 읽기, 순환, import 파일 파싱 문제를 수집한 목록입니다.
 호출자는 목록이 비어 있을 때만 후속 패스를 실행합니다.
따라서 부분 병합 결과가 실수로 유효한 프로그램처럼 처리되지 않습니다.

상대 경로의 기준

엔트리 파일의 import는 baseDir를 기준으로 해석합니다.
그러나 import 파일 안에 또 다른 import가 있으면, 그 경로는 엔트리 디렉터리가 아니라 현재 import 파일의 디렉터리를 기준으로 해석합니다.
따라서 라이브러리 폴더 전체를 다른 프로젝트로 옮겨도 라이브러리 내부의 상대 경로 관계를 그대로 유지할 수 있습니다.

현재 읽는 문장기준 디렉터리계산되는 대상
main.qor의
import "lib/a.qor";		 C:\work\app C:\work\app\lib\a.qor
lib/a.qor의
import "shared/b.qor";		 C:\work\app\lib C:\work\app\lib\shared\b.qor

3실제 처리 순서

ModuleLoader는 import 문장을 소스에 적힌 순서대로 확인합니다.
다만 한 파일이 다시 다른 파일을 import하면 그 자식 파일을 먼저 끝까지 처리한 뒤 현재 파일의 연산을 추가합니다.
이러한 깊이 우선 방문 방식 때문에 import 관계가 여러 단계로 이어져도 같은 규칙으로 처리할 수 있습니다.
구현과 아래 표에서 사용하는 DFS는 이 깊이 우선 탐색을 뜻하는 약어입니다.

  1. 빠른 종료 조건을 확인합니다.
    program.Imports가 null이거나 비어 있으면 파일 작업이 필요하지 않으므로 입력 program 객체를 그대로 반환합니다.
    반면에 import는 있지만 baseDir가 없으면 각 import에 QSEM020을 기록하고, 이 경우에도 입력 프로그램을 그대로 반환합니다.
  2. 병합용 목록을 준비합니다.
    operations에는 엔트리 파일의 연산을 먼저 복사하고, opens에는 엔트리 파일의 네임스페이스별 open 목록을 복사합니다.
    따라서 최종 결과에서도 엔트리 연산이 항상 앞에 남습니다.
  3. 상대 경로를 절대 경로로 바꿉니다.
    현재 파일의 디렉터리와 QImport.Target을 Path.Combine으로 연결한 뒤, Path.GetFullPath로 정규화합니다.
    이 절대 경로가 이후의 순환 검사와 중복 검사에서 파일의 식별자로 사용됩니다.
  4. 현재 방문 경로로 되돌아가는지 먼저 확인합니다.
    대상 경로가 chain에 있으면 아직 처리가 끝나지 않은 상위 파일로 되돌아간 것이므로 순환입니다.
    이때 QSEM021을 추가하고 해당 import 간선만 중단합니다.
  5. 이미 시도한 파일인지 확인합니다.
    순환이 아니라면 대상 경로를 loaded에 추가합니다.
    같은 절대 경로가 이미 들어 있으면 다른 import 경로에서 처리했거나 처리 시도한 파일이므로 다시 읽지 않습니다.
  6. 파일을 읽고 단일 파일 IR로 변환합니다.
    파일 존재 여부를 확인한 뒤 File.ReadAllText로 내용을 읽고, QoraParser.ParseToIr로 해당 파일만 lowering합니다.
    파싱에 실패하면 오류를 기록하며, 그 파일의 자식 import와 연산은 병합하지 않습니다.
  7. 자식 import를 먼저 처리합니다.
    읽은 파일에 import가 있으면 현재 파일을 chain에 추가하고, 현재 파일의 디렉터리를 새 경로 기준으로 전달해 재귀 호출합니다.
    자식 처리가 끝나면 현재 파일을 chain에서 제거합니다.
  8. 현재 파일의 IR을 병합합니다.
    자식 처리가 끝난 뒤 현재 파일의 Operations를 추가하고, 네임스페이스별 Opens를 누적합니다.
    모든 import를 처리한 뒤에는 같은 네임스페이스에서 같은 대상을 여는 중복 open을 제거하고, 반환 프로그램의 Imports를 null로 설정합니다.
loaded 등록 시점 대상 경로는 파일 존재 확인과 읽기보다 먼저 loaded에 추가됩니다.
따라서 존재하지 않거나 읽을 수 없는 파일을 여러 곳에서 import하더라도 첫 시도에서만 파일 오류가 생성되며, 같은 절대 경로에 대한 후속 시도는 중복으로 건너뜁니다.

4하나의 예제로 보는 병합 결과

다음 예제는 main.qor가 두 파일을 import하고, 두 파일이 다시 같은 shared.qor를 import하는 구조입니다.
여러 경로가 하나의 파일로 모이므로 흔히 다이아몬드 import라고 부르지만, 먼저 파일 내용과 최종 IR이 어떻게 연결되는지 확인하는 편이 이해하기 쉽습니다.

입력 파일

// app/main.qor
import "lib/prepare.qor";
import "lib/readout.qor";
namespace Demo {
  open Shared;
  operation Main() {
    use q = Qubit[1];
    Prepare(q);
    Readout(q);
  }
}

// app/lib/prepare.qor
import "shared.qor";
namespace Demo {
  open Shared;
  operation Prepare(Qubit[1] q) {
    H(q[0]);
  }
}

// app/lib/readout.qor
import "shared.qor";
namespace Demo {
  open Shared;
  operation Readout(Qubit[1] q) {
    Mark(q);
  }
}

// app/lib/shared.qor
namespace Shared {

마지막으로 readout.qor를 처리할 때 shared.qor를 다시 만나지만, 이미 같은 절대 경로를 처리했으므로 두 번째 읽기와 병합은 생략하고 Readout만 추가합니다.
결과적으로 연산 순서는 Main → Mark → Prepare → Readout이 됩니다.

Opens가 네임스페이스별 사전인 이유

open은 프로그램 전체에 무조건 적용되는 선언이 아니라, 해당 namespace 블록 안에서 이름을 찾을 때 사용하는 문맥입니다.
따라서 QProgram.Opens는 단순한 문자열 목록이 아니라 선언 네임스페이스 이름을 키로 하고, 그 안에서 연 대상들을 값으로 갖는 사전입니다.
위 예제에서는 Demo 네임스페이스 안에서 Shared를 열었으므로 "Demo" → [QOpen("Shared")]로 저장됩니다.

처리 전 파일별 값병합 과정처리 후 값

 "Demo" → ["Shared"]

같은 Target은 DistinctBy로 제거되며, 처음 들어온 QOpen이 보존됩니다.
prepare.qor
"Demo" → ["Shared"]		 기존 Demo 목록 뒤에 연결됩니다.

이 단계에서는 연산 이름을 완전 수식 이름으로 바꾸지 않습니다.
따라서 Prepare와 Readout의 Name은 아직 짧은 이름이며, 원래 선언된 위치는 Namespace = "Demo"에 따로 남아 있습니다.
이후 Resolver가 병합된 Operations와 Opens를 함께 읽어 이름을 확정합니다.

5자료구조가 담는 값과 필요한 이유

파일 병합에는 “한 번이라도 만난 파일”과 “지금 처리 중인 경로에 있는 파일”이라는 서로 다른 정보가 필요합니다.
두 정보는 겉보기에는 비슷하지만, 전자는 중복 읽기를 막고 후자는 순환을 찾아내므로 하나의 집합으로 합칠 수 없습니다.
구현에서는 각각 loaded와 chain이라는 이름으로 구분합니다.

자료구조실제로 담는 값필요한 이유변경 시점
List<QoraError>
errors		 파일별 경로·읽기·순환·파싱 오류 한 파일의 문제만 보고 즉시 끝내지 않고, 처리 가능한 형제 import의 문제까지 함께 보고하기 위해 사용합니다.
 문제를 발견할 때마다 추가하며 Expand 반환까지 유지합니다.
List<QOperation>
operations		 엔트리 연산과 방문이 끝난 import 파일의 연산 여러 파일에 흩어진 연산을 후속 패스가 읽을 수 있는 하나의 순서 있는 목록으로 만들기 위해 사용합니다.
 엔트리 연산으로 시작하며, 각 파일의 자식 처리가 끝난 뒤 현재 파일의 연산을 추가합니다.
Dictionary<string,
List<QOpen>> opens		 네임스페이스 이름과 그 안에서 선언된 open 대상 목록 같은 네임스페이스 블록이 여러 파일에 나뉘어 있어도 Resolver가 하나의 이름 탐색 문맥으로 사용할 수 있게 병합합니다.
 엔트리 값으로 시작하며, 파일 병합 시 같은 키의 목록 뒤에 값을 추가합니다.
HashSet<string>
loaded		 GetFullPath로 정규화한, 이미 처리 대상으로 등록된 절대 경로 직접 중복 import와 다이아몬드 import에서 같은 파일을 다시 읽고 같은 연산을 두 번 추가하는 문제를 막습니다.
 순환 검사를 통과한 직후 추가하며, DFS 전체가 끝날 때까지 제거하지 않습니다.
List<(Path, Name)>
chain		 현재 재귀 호출에서 아직 처리가 끝나지 않은 파일의 절대 경로와 표시용 파일 이름 대상이 현재 경로의 상위 파일이면 순환으로 판정하고, 사람이 읽을 수 있는 파일 이름 연결을 오류 메시지에 표시합니다.
 자식 import로 들어가기 전에 추가하고, 자식 처리가 끝나면 즉시 제거합니다.
LoadAll(imports,
dir, importer)		 현재 파일의 import 목록, 상대 경로 기준 디렉터리, 진단에 표시할 현재 파일 이름 엔트리와 중첩 import에 같은 처리 절차를 재사용하면서도 각 파일의 경로 기준과 진단 문맥을 유지합니다.
 각 import 파일의 자식 목록을 처리할 때 재귀 호출합니다.
chain에 있으면 현재 경로로 되돌아온 순환이며, loaded에만 있으면 이미 다른 경로에서 만난 중복입니다.

예제 처리 중 두 구조의 상태

시점loadedchain판단
시작 { main } [ main ] sourcePath가 제공되었으므로 엔트리를 미리 등록합니다.
prepare의 자식 처리 { main, prepare } [ main, prepare ] prepare가 아직 자식 import 처리를 마치지 않았습니다.
첫 shared 등록 { main, prepare, shared } [ main, prepare ] shared는 자식 import가 없는 파일이므로 연산을 병합하고 복귀합니다.
두 번째 shared 발견 { main, prepare, shared, readout } [ main, readout ] shared는 chain에는 없고 loaded에만 있으므로 오류 없이 건너뜁니다.

6정상·중복·다이아몬드·순환 흐름

아래 순환 예제는 sourcePath가 제공되어 엔트리 파일이 처음부터 chain에 등록된 경우를 기준으로 합니다.
이 조건에서는 b가 main을 다시 가리키는 순간 엔트리 파일이 현재 방문 경로에 있음을 확인하고 곧바로 순환을 보고합니다.
반면에 sourcePath가 null이면 엔트리 파일의 실제 경로를 미리 등록할 수 없으므로, main을 다시 읽은 뒤 그 안의 import가 기존 chain으로 돌아갈 때 순환을 발견합니다.

정상적인 일렬 import

main → a → b처럼 한 방향으로만 이어지면 각 파일을 한 번씩 읽습니다.
먼저 b의 연산을 추가하고, 재귀 호출이 돌아오면서 a의 연산을 추가합니다.
엔트리 연산은 처음부터 목록에 있으므로 최종 순서는 main 연산 → b 연산 → a 연산입니다.

같은 파일의 직접 중복 import

한 파일이 shared.qor를 두 번 import하면 첫 번째 import에서 절대 경로를 loaded에 등록합니다.
따라서 두 번째 import는 같은 경로를 확인하는 즉시 건너뛰며, 파일 읽기와 연산 병합도 반복하지 않습니다.
이 경우는 순환이 아니므로 오류를 생성하지 않습니다.

다이아몬드 import

prepare와 readout이 모두 shared를 import하더라도 첫 경로에서만 shared를 병합합니다.
두 번째 경로가 shared에 도달할 때에는 첫 방문이 끝났으므로 chain에는 없고 loaded에만 남아 있습니다.
따라서 정상적인 공유로 판단하고 다시 읽지 않습니다.

순환 import

main → a → b → main처럼 아직 처리가 끝나지 않은 상위 파일로 되돌아가면 대상이 chain에서 발견됩니다.
이 경우 마지막 간선을 더 따라가지 않고 QSEM021을 생성합니다.
순환 검사를 loaded 검사보다 먼저 수행하므로 순환을 단순 중복으로 잘못 처리하지 않습니다.

두 구조 모두 같은 파일을 다시 가리키지만, 대상이 현재 chain에 있는지에 따라 정상 공유와 순환이 구분됩니다.

7오류 처리와 부분 결과

로더는 하나의 import에서 문제가 발생하더라도 같은 단계의 다른 import를 계속 확인합니다.
따라서 사용자는 여러 개의 누락 파일이나 읽기 문제를 한 번의 컴파일 결과에서 함께 확인할 수 있습니다.
다만 오류가 하나라도 있으면 QoraParser.Parse가 후속 패스를 실행하지 않으므로, 반환된 부분 병합 결과는 진단용으로만 사용됩니다.

import 파일의 Span을 제거하는 이유

QoraError.Start와 End는 편집기에 열려 있는 엔트리 문서의 문자 위치를 뜻합니다.
만약 import 파일의 문자 위치를 그대로 넣으면, 편집기는 그 숫자를 엔트리 문서의 위치로 해석해 전혀 다른 부분에 오류 표시를 그리게 됩니다.
따라서 ParseToIr는 import 파일을 QoraLowering.Lower(ast, withSpans: false)로 변환하여 IR 노드의 Span을 제거합니다.
또한 import 파일의 파싱 오류는 메시지에 파일 이름을 남기는 대신 위치를 -1..-1로 바꿉니다.

부분 결과의 의미 baseDir가 있는 확장 경로에서는 일부 파일을 읽지 못하더라도, 현재까지 병합된 Operations와 Opens를 담은 프로그램을 반환합니다.
그러나 Errors.Count > 0이면 호출자가 MeasureConditionLowering과 Resolver를 실행하지 않습니다.
그러므로 이 프로그램은 디버깅과 진단을 위한 중간 상태이며, 방출 가능한 유효 IR로 간주하면 안 됩니다.

8후속 패스에 넘기는 계약

import가 있고 경로 문맥도 제공된 확장 경로에서는, 성공 여부와 관계없이 반환용 레코드를 만들면서 Imports = null로 설정합니다.
다만 후속 패스로 실제 전달되는 경우는 오류 목록이 비어 있는 성공 경로뿐입니다.
성공한 프로그램에서는 모든 파일의 연산과 네임스페이스별 open 정보가 하나의 QProgram에 모여 있으므로, 이후 단계가 파일 시스템에 접근할 필요가 없습니다.

항목ModuleLoader가 보장하는 내용후속 패스가 담당하는 내용
연산 목록 엔트리 연산을 먼저 보존하고, import 파일 연산을 깊이 우선 순서로 한 번씩 추가합니다.
 호출 이름 해소, 의미 검증, 연산 본문 변환
파일 경계 성공한 확장 결과에서 Imports = null이며, 후속 단계가 읽어야 할 import 경로가 남지 않습니다.
 파일 I/O 없음
네임스페이스 같은 네임스페이스의 open 목록을 합치고, 같은 Target의 중복을 제거합니다.
 Resolver가 Namespace와 Opens를 사용해 짧은 호출 이름을 완전 수식 이름으로 바꿉니다.
중복 연산 이름 파일이 같은 경로인지 여부만 판단하며, 서로 다른 파일에 같은 연산 이름이 있는지는 허용 여부를 판단하지 않습니다.
 Resolver가 연산 이름을 완전 수식 이름으로 바꾼 뒤, QoraValidator가 전체 이름을 비교하여 전역 중복 QSEM008 또는 네임스페이스 내부 중복 QSEM022를 생성합니다.
노드 내용과 Id 연산 본문과 이름을 재작성하지 않으며, program with { ... }로 반환하므로 기존 QProgram.Id를 유지합니다.
 각 전용 IR 패스가 필요한 노드만 이후에 변환합니다.
심볼릭 링크 GetFullPath 결과를 대소문자 구분 없이 비교합니다.
 서로 다른 경로가 같은 물리 파일을 가리키는지까지 판별하지는 않습니다.
ModuleLoader 정상 출력 — 하나의 QProgram, 처리할 Imports 없음, 파일별 Operations와 Opens 병합 완료
▼
MeasureConditionLowering — 어느 파일에서 온 연산인지와 관계없이 측정 조건을 동일한 IR 형태로 변환
▼
Resolver — 전체 Operations와 네임스페이스별 Opens를 사용해 호출 대상을 확정
▼
QoraValidator — 병합 뒤에 드러난 중복 정의와 호출 규칙을 전체 프로그램 단위로 검사

정리하면 ModuleLoader의 책임은 경로로 연결된 여러 파일을 하나의 IR 컨테이너에 모으는 데까지입니다.
이름의 의미를 결정하거나 연산 본문을 바꾸지 않으며, 그 작업에 필요한 전체 프로그램 문맥만 준비합니다.
이러한 책임 분리 덕분에 후속 패스는 파일이 몇 개였는지 알 필요 없이 QProgram 하나만 처리할 수 있습니다.
