1컴파일 파이프라인

소스 한 줄이 OpenQASM으로 나가기까지 거치는 단계입니다. 검증은 게이트예요 — 의미 에러가 하나라도 있으면 방출은 실행되지 않고 모든 위반을 보고합니다. SemanticModel은 마지막 검증(모노모피제이션 후)이 한 번 만들어 파이프라인 끝까지 재사용됩니다.

▼ QoraLowering

▼ ModuleLoader → MeasureConditionLowering → Resolver
해소된 IR — import 병합, 측정-조건 desugar, 이름 FQN화
▼ 검증 → Monomorphizer → 재검증

두-번-검증의 이유 모노모피제이션이 제네릭 레지스터 크기를 확정하므로, 크기 의존 검사(인덱스 범위·레지스터 크기 일치)는 크기가 정해진 뒤에야 돌 수 있습니다. 그래서 검증이 기저 1회 + 특수화 후 1회이고, 마지막에 돈 검증이 SemanticModel을 소유합니다.

3IR 노드 레퍼런스

IR은 봉인된 record 계층입니다. 모든 문장·연산·파라미터가 생성 시 안정 Id를 갖고, with 재작성은 Id를 상속하며(같은 노드, 편집), new는 새 Id를 발급합니다(다른 노드). 복제 패스는 ReId로 새 Id를 찍고 DerivedFrom 계보를 남깁니다 — 이게 Id-키 사이드테이블이 복사본에서도 원본을 되짚게 하는 근거예요.

노드의미주요 필드
QProgram프로그램 전체Operations
QOperation연산 정의 (Q#의 operation)Name, Params, Body, Id
QParam파라미터 슬롯Name, Type, RegisterSize/SizeParam, Id
QUse큐빗 레지스터 할당 use q = Qubit[n]Name, Size, Id
QGate게이트 적용 또는 op 호출 (같은 노드)Functors, Name, Args, Id
QDecl / QAssign고전 선언 / 대입 (측정 포함 가능)Name, Type, Value, Id
QIf / QFor / QWhile / QRepeat제어흐름 (컨테이너)Cond/Var·From·To·Step, Body, Id
QConjugatewithin/apply 켤레 — 내부 IR 전용 (표면 문법 없음)Within, Apply, Id
QArg게이트 인자: QQubitArg(reg[idx]) / QTextArg(통짜·각도식)—
QExpr / QCond값 표현식 (QText·QMeasure) / 조건HasCall 플래그
within/apply는 의도적으로 표면 문법이 없습니다 QConjugate는 자동 주입기(rung ④)만 생성하는 내부 노드예요. 사용자는 그냥 ancilla를 쓰고, 켤레(compute–act–uncompute) 구조는 컴파일러가 자동으로 만듭니다 (Silq식 완전 자동).

4SemanticModel — 지속 사이드테이블

검증이 증명한 것을 버리지 않고 안정 Id를 키로 한 사이드테이블에 담아 파이프라인 끝까지 나릅니다 (Roslyn SemanticModel / rust-analyzer AstId 방식). 설계 원칙은 SSOT입니다 — 각 사실은 생산자 패스가 정확히 하나, 사실은 추가만 하고 덮어쓰지 않음.

테이블키 → 값생산자
_rootScopeByOpop.Id → 루트 Scope (심벌테이블)Validator
_symbolByDeclId선언 노드 Id → SymbolValidator
_programScope최상위 스코프 — op들이 심벌Validator
_derivedFrom복사본 Id → 원본 Id (계보)복제 패스 (ReId)
_qubitEventsByOpop.Id → 큐빗이벤트 스트림EffectAnalysis
_effectSummaryByOpIdop.Id → 파라미터 효과 요약EffectAnalysis
_emittedNameByDeclId선언 노드 Id → 방출(맹글링 후) 이름NameMangler
SSOT — "하나 깜빡하면 틀어진다"를 구조로 막기 조회가 복사본 노드에서 미스나면 Resolve가 DerivedFrom 계보를 타고 원본까지 올라갑니다. 같은 사실을 두 군데 넣지 않으므로(예: 방출이름은 별도 테이블, 소스이름은 Symbol) 동기화 코드가 0줄 — "동기화를 깜빡한다"는 실패 모드 자체가 없습니다.

5심벌테이블 & 이름의 두 도메인

스코프 하나 = 심벌테이블 하나(이름 → Symbol 맵 + 부모 링크). 이것들이 부모–자식으로 엮여 스코프 트리를 이룹니다. 연산(op)도 심벌이에요 — 프로그램 최상위 스코프에 들어가고, 파라미터·레지스터·변수와 같은 단일 삽입 문(Scope.TryAdd)으로 등록됩니다.

SymbolKind = Parameter · Register · MeasureBit · Var · Const · LoopVar · Operation.

소스 이름 (Symbol.SourceName)

사용자가 쓴 표기. 검증 시점에 얼어붙어 영구 불변. 진단·--stages 심벌 뷰가 이걸 씀.

방출 이름 (FindEmittedName)

맹글링 후 QASM에 나가는 이름. NameMangler가 기록. null이면 "맹글러가 아직 안 봄"(≠ 무변경).

같은 선언에 대해 두 이름이 각자 한 곳에 살고 노드 Id로 조인됩니다. 그래서 var x = Foo처럼 연산을 값으로 쓰면 정확히 QSEM028로 거부돼요 — op이 스코프에 심벌로 있으니 "Foo는 연산이라 값이 아님"을 정확히 말할 수 있습니다.

6효과분석 & 큐빗이벤트 (rung ①)

효과분석은 리프 문장(게이트·측정·use)마다 큐빗당 이벤트 하나를 방출해, 각 연산의 프로그램 순서 use/def 스트림을 만듭니다. 컨테이너(for/if)는 자기 이벤트가 없고 자식 리프가 정밀 detail을 가집니다.

이벤트 종류의미예
Read참조하되 계산기저 값 보존 (컨트롤, 또는 대각 게이트 타깃)CNOT(q0,q1)의 q0
Write값이 바뀔 수 있음 (타깃·reset·레지스터 탄생)X(q0)의 q0
Measure측정으로 붕괴 (비가역)M(q0)의 q0

QubitEvent(Qubit, Kind, Order, StmtId) — 하나의 스트림에서 두 뷰가 유도됩니다:

큐빗별 타임라인 → liveness

큐빗 q로 필터(Overlaps) → Order 최소 = 탄생, 최대 = 마지막 사용(사망점). LiveRange는 이 min/max 조회일 뿐 — 저장 없음.

StmtId로 그룹핑 → 얽힘

같은 StmtId를 공유하는 이벤트 = 그 문장에서 상호작용한 큐빗들. 여기서 얽힘 엣지가 읽힙니다.

얽힘 두 종류 — 값 vs 위상

값얽힘: 0/1 값이 바뀜 (CNOT·CCX). "쓰기(Write)"에 잡힘. 위상얽힘: 값은 그대로, 부호(위상)만 붙음 (CZ·CS). 값을 안 바꿔서 "쓰기"에 안 잡힘 — 대각 게이트 접촉으로 따로 챙겨야 합니다.

7자동 uncomputation 사다리

목표는 사용자가 안 쓴 정리(uncompute) 코드를 컴파일러가 자동으로 끼워주는 것입니다. 아래→위로 의존하는 4단이고, rung ①·②는 이미 이 데이터시트의 6절 자료구조 위에서 조회로 끝납니다.

①
use/def — 효과분석구현됨
리프마다 큐빗이벤트(Read/Write/Measure). SemanticModel에 op별 스트림으로 적재.
②
liveness구현됨 (조회)
LiveRange = 이벤트 Order의 min/max = 탄생·사망점. 무거운 패스가 아니라 min/max 조회.
③
qfree — 되돌림 안전 판정다음
"이 값이 다른 큐빗들로부터 유도되나(되만들 수 있나) + 되돌려도 측정된 것을 안 건드리나."
④
자동 주입
ancilla가 죽는 지점에 역연산(QConjugate) 주입. 역연산 커널은 이미 존재 → 역합성 파이프라인.
사다리의 알맹이: 측정으로 안 붙잡힌 유도값은 전부 되돌려 |0⟩로 청소한다.

8ancilla와 uncompute — 왜, 무엇을

ancilla는 계산 중간에 잠깐 쓰는 보조 큐빗(연습장)입니다. |0⟩로 태어나 데이터와 얽혔다가, 다 쓰면 다시 |0⟩로 되돌려 반납해야 해요 — 안 되돌리면 얽힌 쓰레기가 최종 결과를 오염시킵니다. 큰 게이트를 작은 게이트로 쪼갤 때 중간값을 담을 곳이 필요해서 생깁니다. 

// a AND b AND c 일 때만 d 뒤집기 — 3중 컨트롤 게이트가 없어서 t로 쪼갬
CCX(a, b, t);   // ① compute: t = a AND b        (t = 다리, 정체는 a&b)
CCX(t, c, d);   // ② action:  d 뒤집음 (t AND c)  ← t의 마지막 사용 = 사망점
CCX(a, b, t);   // ③ uncompute: t → |0⟩ (rung ④가 자동 주입할 줄)

무엇이 uncompute 대상인가 — 큐빗마다 질문 2개

A. 프로그램이 만든 값인가?

게이트의 타깃(Write)으로 값이 바뀐 적 있으면 = 만든 값(되돌릴 레시피 있음). 읽기만 하면 = 입력(root, 되돌릴 것 없음).

B. 결과로 필요한가?

측정됨 → 출력, 유지. 측정된 것과 얽혀 되돌리면 그게 틀어짐 → 유지. 둘 다 아니면 → 청소.

uncompute 대상 = A "만들었다" + B "안 필요하다"
필터 (구조)제외 대상근거
1. 파라미터제외호출자 입력 — 내 것 아님
2. def 없음(읽기만)제외순수 입력(root) — 되돌릴 레시피 없음
3. 측정됨제외출력 — 측정이 값을 확정(닻)
4. 남은 것후보측정 안 된 유도값(다리) → rung ③이 안전 최종 판정
t는 되고 a,b,c는 안 되는 이유

t 되돌리기 CCX(a,b,t)는 a,b,t만 건드리고 측정된 d를 안 건드림 → 안전. 준비된 입력 a를 되돌리려면 측정된 d와 얽힌 a를 건드려 결과가 틀어짐 → 불가. 즉 판정은 "얽혔냐"가 아니라 "되돌려도 측정된 것이 보존되냐"(rung ③)입니다.
