1컴파일 파이프라인

이 장은 소스 한 줄이 OpenQASM으로 출력되기까지 거치는 단계를 보여줍니다.
검증은 방출을 통제하는 게이트입니다.
의미 에러가 하나라도 있으면 방출은 실행되지 않고, 컴파일러는 발견한 모든 위반을 보고합니다.
SemanticModel은 마지막 검증(모노모피제이션 후)에서 한 번 만들어지며 파이프라인 끝까지 재사용됩니다.

소스 — Qora 텍스트
▼ Janglim 파싱
AST — 엔진 문법 트리 (이후 읽기만)
▼ QoraLowering
QoraIR — 강타입 노드 QProgram (안정 Id 발급)
▼ ModuleLoader → MeasureConditionLowering → Resolver
해소된 IR — import 병합, 측정-조건 desugar, 이름 FQN화
▼ 검증 → Monomorphizer → 재검증
검증된 · 특수화된 IR — ★ 여기서 SemanticModel 1회 구축
▼ EffectAnalysis (rung ①)
+ 큐빗이벤트 / 요약 — 모델에 적재 (IR 무변경)
▼ ConjugationLowering → AdjointMaterializer → NameMangler → ReferentialCheck → OpenQasmLowering
방출용 IR — within/apply 평탄화, 역연산 실체화, 이름 맹글링, Id 무결성
▼ QasmEmitter
OpenQASM 3 — 순수 프린터 출력
두 번 검증하는 이유 모노모피제이션이 제네릭 레지스터 크기를 확정하므로, 크기 의존 검사(인덱스 범위·레지스터 크기 일치)는 크기가 정해진 뒤에야 정확하게 실행될 수 있습니다. 그래서 검증은 기저 IR에서 1회, 특수화 후 1회 수행되며, 마지막 검증 결과가 SemanticModel을 소유합니다.

3IR 노드 레퍼런스

IR은 봉인된 record 계층입니다.
모든 문장·연산·파라미터는 생성 시점에 안정 Id를 가집니다.
with 재작성은 같은 노드를 편집하는 방식이므로 Id를 상속하고, new는 다른 노드를 만드는 방식이므로 새 Id를 발급합니다.
복제 패스는 ReId로 새 Id를 찍고 DerivedFrom 계보를 남깁니다.
이것이 Id-키 사이드테이블이 복사본에서도 원본을 되짚게 하는 근거입니다.

노드의미주요 필드
QProgram프로그램 전체Operations
QOperation연산 정의 (Q#의 operation)Name, Params, Body, Id
QParam스칼라 또는 배열 파라미터 슬롯Name, Type, IsArray, RegisterSize, Id
QUse큐빗 레지스터 할당 use q = Qubit[n]Name, Size, Id
QGate게이트 적용 또는 op 호출 (같은 노드)Functors, Name, Args, Id
QDecl / QAssign고전 스칼라·배열 선언 / 전체 변수 또는 배열 원소 대입Name, Type, IsArray / Index, Value, Id
QIf / QFor / QWhile / QRepeat제어흐름 (컨테이너)Cond/Var·From·To·Step, Body, Id
QConjugatewithin/apply 켤레 — 내부 IR 전용 (표면 문법 없음)Within, Apply, Id
QArg게이트 인자: QQubitArg(reg[idx]) / QTextArg(통짜·각도식)—
QExpr / QCond값 표현식 (QText·QMeasure·QArrayLiteral·QArrayNew) / 조건HasCall 플래그 또는 배열 원소·타입·길이
within/apply는 의도적으로 표면 문법이 없습니다 QConjugate는 자동 주입기(rung ④)만 생성하는 내부 노드입니다. 사용자는 앤실라만 작성하고, 켤레(compute–act–uncompute) 구조는 컴파일러가 자동으로 만듭니다 (Silq식 완전 자동).

4SemanticModel — 지속 사이드테이블

검증이 증명한 사실은 버리지 않고 안정 Id를 키로 한 사이드테이블에 담아 파이프라인 끝까지 전달합니다 (Roslyn SemanticModel / rust-analyzer AstId 방식).
설계 원칙은 SSOT입니다.
각 사실은 정확히 하나의 생산자 패스가 만들며, 한번 기록된 사실은 추가만 하고 덮어쓰지 않습니다.

테이블키 → 값생산자
_rootScopeByOpop.Id → 루트 Scope (심벌테이블)Validator
_symbolByDeclId선언 노드 Id → SymbolValidator
_programScope최상위 스코프 — op들이 심벌Validator
_derivedFrom복사본 Id → 원본 Id (계보)복제 패스 (ReId)
_qubitEventsByOpop.Id → 큐빗이벤트 스트림 (시간·역할·플래그)EffectAnalysis
_qubitGraphByOpop.Id → 큐빗그래프 (값 버전 노드 + "만들어짐" 부모 엣지) — 이벤트와 같은 손·같은 순간에 기록, 상시 일치성 스윕이 어긋남을 QINTERNAL로 차단EffectAnalysis
_effectSummaryByOpIdop.Id → 파라미터 효과 요약EffectAnalysis
_emittedNameByDeclId선언 노드 Id → 방출(맹글링 후) 이름NameMangler
SSOT — 동기화 누락을 구조로 막습니다 조회가 복사본 노드에서 실패하면 Resolve가 DerivedFrom 계보를 따라 원본까지 올라갑니다. 같은 사실을 두 군데 넣지 않으므로(예: 방출이름은 별도 테이블, 소스이름은 Symbol) 동기화 코드가 필요하지 않습니다. 따라서 동기화 누락이라는 실패 모드 자체가 사라집니다.

5심벌테이블 & 이름의 두 도메인

스코프 하나 = 심벌테이블 하나(이름 → Symbol 맵 + 부모 링크)이며, 이것들이 부모–자식으로 엮여 스코프 트리를 이룹니다.
연산(op)도 심벌입니다.
연산은 프로그램 최상위 스코프에 들어가며, 파라미터·레지스터·변수와 같은 단일 삽입 문(Scope.TryAdd)으로 등록됩니다.

SymbolKind = Parameter · Register · MeasureBit · Var · Const · LoopVar · Operation.

소스 이름 (Symbol.SourceName)

사용자가 쓴 표기입니다.
검증 시점에 고정되어 영구 불변으로 남습니다.
진단·--stages 심벌 뷰는 이 이름을 사용합니다.

방출 이름 (FindEmittedName)

맹글링 후 QASM에 나가는 이름입니다.
NameMangler가 기록합니다.
null이면 "맹글러가 아직 처리하지 않음"을 뜻하며, "무변경"을 뜻하지 않습니다.

같은 선언에 대해 두 이름은 각자 한 곳에 저장되고 노드 Id로 조인됩니다.
그래서 var x = Foo처럼 연산을 값으로 쓰면 정확히 QSEM028로 거부됩니다.
연산이 스코프에 심벌로 있으므로 "Foo는 연산이라 값이 아니다"라고 정확히 진단할 수 있습니다.

6효과분석 & 큐빗이벤트 (rung ①)

효과분석은 리프 문장(게이트·측정·use)마다 큐빗당 이벤트 하나를 방출해, 각 연산의 프로그램 순서 use/def 스트림을 만듭니다.
컨테이너(for/if)는 자기 이벤트를 만들지 않고 자식 리프가 정밀한 세부를 가집니다.
레지스터의 탄생(use)은 방출기의 선언 호이스팅과 동일하게 스트림 맨 앞에 기록되므로, 게이트가 텍스트상 자기 레지스터의 use보다 앞에 와도 올바르게 분석됩니다.

이벤트 종류의미예
Read참조하되 계산기저 값 보존 (컨트롤, 또는 대각 게이트 타깃)CNOT(q0,q1)의 q0
Write값이 바뀔 수 있음 (타깃·reset·레지스터 탄생)X(q0)의 q0
Measure측정으로 붕괴 (비가역)M(q0)의 q0

QubitEvent(Qubit, Kind, Order, StmtId, Irreversible, NonQfree, NodeId) — Irreversible은 역연산이 없는 접촉(reset·비가역 호출의 쓰기에만) 표시이고, NonQfree는 자동 청소를 불가능하게 만드는 쓰기(H·Rx·Ry의 중첩 생성 또는 Y·CY의 위상 치환 — qfree와 Y·CY 차단에서 설명, 호출을 통해 전이 전파) 표시이며, NodeId는 큐빗그래프(값 버전의 족보 DAG — 쓰기·측정은 그 이벤트가 태어나게 한 노드와 1:1, 읽기는 읽힌 당시의 현재 버전)로 가는 열쇠입니다.
역할 분담은 다음과 같습니다: 이벤트는 시간·역할·플래그를, 그래프는 관계(버전과 부모)를, IR은 구조(컨테이너)를 담으며, 한 사실은 정확히 한 곳에만 저장됩니다.
하나의 스트림에서 두 뷰가 유도됩니다.

큐빗별 타임라인 → liveness

큐빗 q로 필터(Overlaps)하면 Order의 최솟값이 탄생, 최댓값이 마지막 사용(사망점)입니다.
LiveRange는 이 min/max 조회 결과일 뿐이며 별도로 저장하지 않습니다.

StmtId로 그룹핑 → 얽힘

같은 StmtId를 공유하는 이벤트들이 그 문장에서 상호작용한 큐빗들입니다.
여기서 얽힘 엣지가 읽힙니다.

얽힘 두 종류 — 값 vs 위상

값얽힘은 0/1 값이 바뀌는 상호작용(CNOT·CCX)으로, 그 문장에 쓰기(Write) 이벤트가 있습니다.
위상얽힘은 값은 그대로 두고 부호만 새기는 상호작용으로, 두 큐빗 이상이 한 문장(StmtId)을 공유하며 이벤트가 전부 읽기(all-Read)일 때입니다 (CZ, Controlled S/T/Z/Rz).
단독 S·T·Z·Rz는 국소 위상일 뿐이어서 1-큐빗 그룹은 얽힘 엣지를 만들지 않습니다.
두 종류 모두 StmtId 그룹핑에서 유도되므로 별도 추적이 필요 없습니다.
어떤 상호작용이 되돌림을 막는지는 상태벡터에서 시작하는 장들이 자세히 설명합니다.

7자동 uncomputation 사다리

목표는 사용자가 작성하지 않은 정리(uncompute) 코드를 컴파일러가 자동으로 끼워 넣는 것입니다.
사다리는 아래→위로 의존하는 4단 구조이며, rung ①·②는 8장의 자료구조 위에서 조회로 끝납니다.

①
use/def — 효과분석구현됨
리프마다 큐빗이벤트(Read/Write/Measure). SemanticModel에 op별 스트림으로 적재.
②
liveness구현됨 (조회)
LiveRange = 이벤트 Order의 min/max = 탄생·사망점. 무거운 패스가 아니라 min/max 조회.
③
되돌림 안전 판정 — IsSafelyUncomputable구현됨 (조회)
UncomputeSafety(op, q) = 전제 + 4-clause 판정 + 사유·범인 문장. 전제: 안전 질문은 청소 후보에게만 던집니다 — 측정 여부는 앤실라와 청소 후보의 판별(청소 대상인가)의 판결이므로 여기서 재판정하지 않고 그대로 전달합니다. 조항: 비가역 접촉 없음(Irreversible) · 비-qfree 쓰기 없음(NonQfreeWrite — qfree·Y·CY 차단) · 컨테이너 안 쓰기 없음(ContainedWrite — 루프와 호출) · 재료 생존(CoWrittenPartner/SourceDied — 그래프에 기록된 부모(재료)가 청소 지점까지 그 값 그대로여야 하며, 컨테이너 안에서 죽으면 검사 구간이 최외곽 컨테이너 끝까지 연장됩니다).
④
자동 주입다음
언어 규칙: 측정하지 않은 use 레지스터는 스코프 끝에 |0⟩로 반납됩니다 — 자동으로 치울 수 없으면 조용히 넘어가지 않고 컴파일 에러가 됩니다. 주입 지점은 사망점, 컨테이너 안에서 죽으면 최외곽 컨테이너 직후입니다. 역연산 커널은 이미 존재합니다 → 역합성 파이프라인.
사다리의 핵심: 측정으로 고정되지 않은 유도값은 전부 되돌려 |0⟩로 정리합니다.

8자동 uncomputation

Qora는 다 쓴 보조 큐빗(앤실라)을 |0⟩로 되돌리는 정리 코드를 컴파일러가 자동으로 삽입합니다.
어떤 큐빗이 정리 대상이 되는지, 어떤 계산은 자동으로 되돌릴 수 있고 어떤 계산은 왜 컴파일 에러가 되는지 — 아래 장들이 기초부터 차례로 답합니다.

장제목다루는 것
1앤실라와 uncomputation — 작업 큐빗은 왜 되돌려야 하는가use로 만든 작업 큐빗과 자동 되돌리기
2컴파일러는 어떤 큐빗을 정리하는가 — 판별 사다리세 질문으로 정리 대상을 가려낸다 — 소유·측정·복원 가능성
3상태를 읽는 법 — 상태벡터와 진폭기저 상태와 진폭 — 이후 모든 판정을 설명하는 틀
4상태벡터를 큐빗별로 분해할 수 있는가 — 곱 상태와 얽힌 상태네 진폭의 대각선 곱 비교와 텐서 곱 분해 — 곱 상태와 얽힌 상태의 판정
5곱에서 얽힘으로 — 프로그램 단계별 확인H는 곱 상태, CNOT 한 줄은 얽힌 상태 — 단계별로 확인
6언제 위상이 측정에 드러나는가 — 전역 위상과 상대 위상복소수 진폭의 위상 — 공통 인자는 전역 위상, 항 사이의 위상 차이는 상대 위상
7전역 위상과 상대 위상 — 측정으로 확인하기같은 Z 게이트가 항 1개엔 전역 위상(결과 유지), 항 2개엔 상대 위상(결과 뒤집힘)
8게이트는 상태벡터를 어떻게 바꾸는가기저 값 · 진폭 · 중첩 생성 — 편집의 세 종류
9청소의 계약과 qfree되돌릴 수 있는 쓰기의 조건
10Y·CY가 차단되는 이유 — 위상값에 따라 달라지는 위상과 얽힘
11주입기 — 문장 붙여넣기 기계자동 청소가 실제로 하는 일
12루프와 호출 — 왜 판정이 달라지는가문장 구조가 판정을 가르는 이유
13담요 문제 — 질의 폭과 문장 폭레지스터 통짜 쓰기와 원소 질문의 충돌
14차단의 두 종류에러의 지도 — 물리의 벽과 구현의 여지
15왜 정적 컴파일인가 — 두 개의 런타임호스트와 QPU — 인터프리터가 들어갈 수 없는 곳
