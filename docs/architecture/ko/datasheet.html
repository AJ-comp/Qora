1컴파일 파이프라인

이 절은 소스 한 줄이 OpenQASM으로 출력되기까지 거치는 단계를 보여준다. 검증은 방출을 통제하는 게이트다. 의미 에러가 하나라도 있으면 방출은 실행되지 않고, 컴파일러는 발견한 모든 위반을 보고한다. SemanticModel은 마지막 검증(모노모피제이션 후)에서 한 번 만들어지며 파이프라인 끝까지 재사용된다.

두 번 검증하는 이유 모노모피제이션이 제네릭 레지스터 크기를 확정하므로, 크기 의존 검사(인덱스 범위·레지스터 크기 일치)는 크기가 정해진 뒤에야 정확하게 실행될 수 있다. 그래서 검증은 기저 IR에서 1회, 특수화 후 1회 수행되며, 마지막 검증 결과가 SemanticModel을 소유한다.

3IR 노드 레퍼런스

IR은 봉인된 record 계층이다. 모든 문장·연산·파라미터는 생성 시점에 안정 Id를 갖는다. with 재작성은 같은 노드를 편집하는 방식이므로 Id를 상속하고, new는 다른 노드를 만드는 방식이므로 새 Id를 발급한다. 복제 패스는 ReId로 새 Id를 찍고 DerivedFrom 계보를 남긴다. 이것이 Id-키 사이드테이블이 복사본에서도 원본을 되짚게 하는 근거다.

노드의미주요 필드
QProgram프로그램 전체Operations
QOperation연산 정의 (Q#의 operation)Name, Params, Body, Id
QParam파라미터 슬롯Name, Type, RegisterSize/SizeParam, Id
QUse큐빗 레지스터 할당 use q = Qubit[n]Name, Size, Id
QGate게이트 적용 또는 op 호출 (같은 노드)Functors, Name, Args, Id
QDecl / QAssign고전 선언 / 대입 (측정 포함 가능)Name, Type, Value, Id
QIf / QFor / QWhile / QRepeat제어흐름 (컨테이너)Cond/Var·From·To·Step, Body, Id

QArg게이트 인자: QQubitArg(reg[idx]) / QTextArg(통짜·각도식)—
QExpr / QCond값 표현식 (QText·QMeasure) / 조건HasCall 플래그
within/apply는 의도적으로 표면 문법이 없다 QConjugate는 자동 주입기(rung ④)만 생성하는 내부 노드다. 사용자는 ancilla만 작성하고, 켤레(compute–act–uncompute) 구조는 컴파일러가 자동으로 만든다 (Silq식 완전 자동).

4SemanticModel — 지속 사이드테이블

검증이 증명한 사실은 버리지 않고 안정 Id를 키로 한 사이드테이블에 담아 파이프라인 끝까지 전달한다 (Roslyn SemanticModel / rust-analyzer AstId 방식). 설계 원칙은 SSOT다. 각 사실은 정확히 하나의 생산자 패스가 만들며, 한번 기록된 사실은 추가만 하고 덮어쓰지 않는다.

테이블키 → 값생산자
_rootScopeByOpop.Id → 루트 Scope (심벌테이블)Validator
_symbolByDeclId선언 노드 Id → SymbolValidator
_programScope최상위 스코프 — op들이 심벌Validator
_derivedFrom복사본 Id → 원본 Id (계보)복제 패스 (ReId)
_qubitEventsByOpop.Id → 큐빗이벤트 스트림 (시간·역할·플래그)EffectAnalysis
_qubitGraphByOpop.Id → 큐빗그래프 (값 버전 노드 + "만들어짐" 부모 엣지) — 이벤트와 같은 손·같은 순간에 기록, 상시 일치성 스윕이 어긋남을 QINTERNAL로 차단EffectAnalysis
_effectSummaryByOpIdop.Id → 파라미터 효과 요약EffectAnalysis
_emittedNameByDeclId선언 노드 Id → 방출(맹글링 후) 이름NameMangler
SSOT — 동기화 누락을 구조로 막는다 조회가 복사본 노드에서 실패하면 Resolve가 DerivedFrom 계보를 따라 원본까지 올라간다. 같은 사실을 두 군데 넣지 않으므로(예: 방출이름은 별도 테이블, 소스이름은 Symbol) 동기화 코드가 필요하지 않다. 따라서 동기화 누락이라는 실패 모드 자체가 사라진다.

5심벌테이블 & 이름의 두 도메인

스코프 하나 = 심벌테이블 하나(이름 → Symbol 맵 + 부모 링크). 이것들이 부모–자식으로 엮여 스코프 트리를 이룬다. 연산(op)도 심벌이다. 연산은 프로그램 최상위 스코프에 들어가며, 파라미터·레지스터·변수와 같은 단일 삽입 문(Scope.TryAdd)으로 등록된다.

SymbolKind = Parameter · Register · MeasureBit · Var · Const · LoopVar · Operation.

소스 이름 (Symbol.SourceName)

사용자가 쓴 표기다. 검증 시점에 고정되어 영구 불변으로 남는다. 진단·--stages 심벌 뷰는 이 이름을 사용한다.

방출 이름 (FindEmittedName)

맹글링 후 QASM에 나가는 이름이다. NameMangler가 기록한다. null이면 "맹글러가 아직 처리하지 않음"을 뜻하며, "무변경"을 뜻하지 않는다.

같은 선언에 대해 두 이름은 각자 한 곳에 저장되고 노드 Id로 조인된다. 그래서 var x = Foo처럼 연산을 값으로 쓰면 정확히 QSEM028로 거부된다. op이 스코프에 심벌로 있으므로 "Foo는 연산이라 값이 아니다"라고 정확히 진단할 수 있다.

6효과분석 & 큐빗이벤트 (rung ①)

효과분석은 리프 문장(게이트·측정·use)마다 큐빗당 이벤트 하나를 방출해, 각 연산의 프로그램 순서 use/def 스트림을 만든다. 컨테이너(for/if)는 자기 이벤트를 만들지 않고 자식 리프가 정밀 detail을 가진다.

이벤트 종류의미예
Read참조하되 계산기저 값 보존 (컨트롤, 또는 대각 게이트 타깃)CNOT(q0,q1)의 q0
Write값이 바뀔 수 있음 (타깃·reset·레지스터 탄생)X(q0)의 q0
Measure측정으로 붕괴 (비가역)M(q0)의 q0

QubitEvent(Qubit, Kind, Order, StmtId, Irreversible, CreatesSuperposition, NodeId) — Irreversible은 역연산이 없는 접촉(reset·비가역 호출의 쓰기에만), CreatesSuperposition은 중첩을 만드는 쓰기(H/Rx/Ry, 호출로 전이 전파), NodeId는 큐빗그래프(값 버전의 족보 DAG — 쓰기·측정은 그 이벤트가 태어나게 한 노드와 1:1, 읽기는 읽힌 당시의 현재 버전)로 가는 열쇠다. 역할 분담: 이벤트 = 시간·역할·플래그 / 그래프 = 관계(버전·부모) / IR = 구조(컨테이너) — 한 사실 한 집. 하나의 스트림에서 두 뷰가 유도된다.

큐빗별 타임라인 → liveness

큐빗 q로 필터(Overlaps) → Order 최소 = 탄생, 최대 = 마지막 사용(사망점). LiveRange는 이 min/max 조회 결과일 뿐이며 별도로 저장하지 않는다.

StmtId로 그룹핑 → 얽힘

같은 StmtId를 공유하는 이벤트 = 그 문장에서 상호작용한 큐빗들. 여기서 얽힘 엣지가 읽힌다.

얽힘 두 종류 — 값 vs 위상 (검증으로 확정된 사실)

값얽힘: 0/1 값이 바뀜 (CNOT·CCX) — 그 문장에 "쓰기(Write)"가 있다. 위상얽힘: 값은 그대로, 부호만 — 두 큐빗 이상이 한 문장(StmtId)을 공유하며 이벤트가 전부 읽기(all-Read)일 때 (CZ, Controlled S/T/Z/Rz). 단독 S·T·Z·Rz는 국소 위상일 뿐 — 1-큐빗 그룹은 얽힘 엣지를 만들지 않는다. 즉 둘 다 StmtId 그룹핑에서 유도되며 별도 추적이 필요 없다. 그리고 상태벡터 검증(적대검증 5라운드 퍼저)으로 확정된 되돌림 안전성 — 두 종류의 "쓰기"가 되돌림을 막는다: 중첩을 만든 쓰기(H·Rx·Ry), 그리고 기저의존 위상을 실은 치환 쓰기(Y·CY) — Y는 0→1에 +i, 1→0에 −i를 실어, 타깃이 얽혀 있으면 그 위상이 생존자 분기 사이의 상대 위상이 되고 주입된 역연산이 이를 벗겨내 하류 측정을 뒤집는다(Silq가 Y를 qfree에서 뺀 이유; "Y는 안전, Silq보다 넓다"던 우리의 이전 주장은 검증으로 반증됨). 반면 대각 위상 게이트(Z·S·T·CZ)는 안전 — 값을 쓰지 않아(all-Read) qfree 쓰기 자체가 아니므로 uncompute를 방해하지 않는다.

목표는 사용자가 작성하지 않은 정리(uncompute) 코드를 컴파일러가 자동으로 끼워 넣는 것이다. 사다리는 아래→위로 의존하는 4단 구조이며, rung ①·②는 이미 이 데이터시트의 6절 자료구조 위에서 조회로 끝난다.

되돌림 안전 판정 — IsSafelyUncomputable구현됨 (조회)
UncomputeSafety(op, q) = 전제 + 4-clause 판정 + 사유·범인 문장. 전제(1단 위임): 안전 질문은 청소 후보에게만 — 측정 여부는 1단(청소 대상인가)의 판결이라 여기서 재판정하지 않고 그대로 전달(Measured = 출력으로 승격 / NotACleanupCandidate = 앤실라 아님). 조항: 비가역 접촉 없음(Irreversible) · 비-qfree 쓰기 없음(NonQfreeWrite — 중첩 생성 H·Rx·Ry 또는 기저의존 위상치환 Y·CY) · 컨테이너 안 쓰기 없음(ContainedWrite — if는 조건부, 루프는 반복이라 직선 역연산이 실행을 못 미러링) · 재료 생존(CoWrittenPartner — SWAP류 동시쓰기, 그리고 원소 쿼리에서 q보다 넓은 담요 쓰기는 차단하되 use 탄생만 면제(유일한 무부모 쓰기 노드로 생성 시 QINTERNAL로 강제) / SourceDied — 그래프에 기록된 부모(재료)가 청소 지점까지 그 값 그대로여야 하며, 컨테이너 안에서 죽으면 검사 구간이 최외곽 컨테이너 끝까지 연장). 적대검증 5라운드(표적 4 + 열거 퍼저)가 직전 수정마다 결함을 찾아내며 다듬은 조건 — 최신 수정분은 차기 라운드 검증 대기.
④
자동 주입다음
확정된 언어 규칙(B): 측정 안 한 use 레지스터는 스코프 끝에 |0⟩로 반납된다 — 선포·버전 경계, 못 치우면 조용한 스킵이 아니라 컴파일 에러. 주입 지점 = 사망점, 컨테이너 안에서 죽으면 최외곽 컨테이너 직후. 역연산 커널은 이미 존재 → 역합성 파이프라인.
사다리의 핵심: 측정으로 고정되지 않은 유도값은 전부 되돌려 |0⟩로 정리한다.

8ancilla와 uncompute — 왜, 무엇을

ancilla는 계산 중간에 잠깐 쓰는 보조 큐빗(연습장)이다. 정의는 탄생 기준 하나: use로 만든(= 이 연산 소유, |0⟩로 시작하는) 지역 작업공간 큐빗 — 문헌 표준(Q# use, Quipper with_ancilla, Qiskit AncillaRegister)과 동일하다. |0⟩로 태어나 데이터와 얽혔다가, 다 쓰면 다시 |0⟩로 되돌려 반납해야 한다. 되돌리지 않으면 얽힌 쓰레기가 최종 결과를 오염시킨다. 여기서 용어 하나 더: 앤실라 중 측정된 것은 출력으로 승격되어(값이 배달됨 — 지연측정 원리상 출력 와이어) 청소 대상에서 빠지고, 측정되지 않은 앤실라만 "청소 후보"다. 한 개념 = 한 용어: 앤실라(탄생) / 청소 후보(측정 안 된 앤실라) / safe(rung ③ 통과한 후보). 

// a AND b AND c 일 때만 d 뒤집기 — 3중 컨트롤 게이트가 없어서 t를 중간값으로 사용
CCX(a, b, t);   // ① compute: t = a AND b        (t = 다리, 정체는 a&b)
CCX(t, c, d);   // ② action:  d 뒤집음 (t AND c)  ← t의 마지막 사용 = 사망점
CCX(a, b, t);   // ③ uncompute: t → |0⟩ (rung ④가 자동 주입할 줄)

무엇이 uncompute 대상인가 — 큐빗마다 질문 2개

A. 프로그램이 만든 값인가?

게이트의 타깃(Write)으로 값이 바뀐 적이 있으면 프로그램이 만든 값이다. 이 경우에는 되돌릴 레시피가 있다. 반대로 읽기만 했다면 입력(root)이므로 되돌릴 대상이 아니다.

B. 결과로 필요한가?

측정된 값은 출력이므로 유지한다. 또한 측정된 값과 얽혀 있어서 되돌리면 출력이 틀어지는 값도 유지한다. 둘 다 아니라면 정리 대상이다.

uncompute 대상 = A "프로그램이 만든 값" + B "결과에 필요하지 않은 값"
필터 (구조)제외 대상근거
1. 파라미터제외호출자 입력 — 현재 연산이 만든 값이 아니다
2. def 없음(읽기만)제외순수 입력(root) — 되돌릴 레시피 없음
3. 측정됨제외출력으로 승격 — 값이 배달됨(앤실라로 태어난 건 변함없으나 청소 대상 아님)
4. 남은 것청소 후보측정되지 않은 앤실라 → rung ③이 안전성 최종 판정
t는 되돌릴 수 있고 a,b,c는 되돌릴 수 없는 이유

t를 되돌리는 CCX(a,b,t)는 a,b,t만 변경하고 측정된 d를 변경하지 않는다. 따라서 안전하다. 반면 준비된 입력 a를 되돌리려면 측정된 d와 얽힌 a를 변경하게 되어 결과가 틀어진다. 따라서 불가능하다. 즉 판정 기준은 "얽혔는가"가 아니라 "되돌려도 측정된 것이 보존되는가"(rung ③)다.

여기까지가 "무엇을 치우는가"다. 이어지는 네 절이 나머지를 기초부터 답한다: 9절 상태를 읽는 장부 모델 → 10절 어떤 쓰기가 되돌려지는가(qfree)와 Y 사건 → 11절 주입기가 실제로 하는 일과 루프·담요의 한계 → 12절 이 설계가 서 있는 자리(두 런타임과 언어 지형).

9상태를 읽는 법 — 장부 모델

rung ③의 판정 조건들("이 게이트는 왜 위험한가")을 이해하려면 양자 상태를 읽는 틀이 하나 필요하다. 이 절의 장부 모델 하나로 뒤 절의 모든 판정이 설명된다.

기계 전체의 상태는 장부 한 권이다. 장부의 한 줄(=가지)은 "모든 큐빗의 값 조합 하나 + 그 조합의 계수 한 칸"이다. 예를 들어 상태 (|0,0⟩ + |1,1⟩)/√2 (표기: |b,a⟩)를 장부로 쓰면:

줄(가지)b 값a 값계수
1행001/√2
2행111/√2
계수 칸은 줄마다 하나 — "a의 계수"라는 것은 없다

수식으로 i·(|0⟩_b ⊗ |1⟩_a) = (i|0⟩_b) ⊗ |1⟩_a = |0⟩_b ⊗ (i|1⟩_a) — 곱셈이라 i를 어디에 붙여 읽어도 같다. 즉 위상(계수)은 특정 큐빗의 소유물이 아니라 줄(조합 전체)의 속성이다. 이 사실이 뒤 절의 Y 사건을 만든다.

게이트는 전부 장부 편집 연산으로 읽을 수 있다:

게이트장부에 하는 일예
X각 줄에서 해당 값 칸만 뒤집기줄 (b=0,a=0) → (b=0,a=1), 계수 불변
CNOT컨트롤 칸이 1인 줄에서만 타깃 값 칸 뒤집기값 칸만, 계수 불변
Z·S·T·CZ값 칸은 안 건드리고 계수에만 부호/위상 곱하기a=1인 줄의 계수에 −1 (Z)
H줄을 두 줄로 쪼개기 (한 값 → 두 값의 중첩)1줄 장부 → 2줄 장부
Y값 칸 뒤집기 + 뒤집은 값에 따라 다른 도장을 그 줄 계수에0→1이면 +i, 1→0이면 −i
M (측정)한 값으로 줄들을 삭제하고 결과를 bit로 배달비가역
전역 위상 vs 상대 위상 — 위험의 기준

계수에 붙는 부호가 모든 줄에 똑같이 붙으면(전역 위상) 물리적으로 아무 의미가 없다 — 측정으로 절대 구분 불가. 줄마다 다르게 붙으면(상대 위상) 나중에 H 같은 간섭 게이트가 그 부호 차이를 측정 확률 차이로 바꿔 읽는다. 위험한 것은 "위상이 있다"가 아니라 "줄마다 다르게 붙었다"이다.

10qfree — 어떤 쓰기를 되돌릴 수 있나 (그리고 Y 사건)

자동 청소의 계약: "앤실라만 |0⟩로 빼내고, 생존자들의 장부는 부호 하나까지 그대로 둔다." 이 계약을 지킬 수 있는 쓰기가 어떤 것인지가 rung ③의 2번 조항(qfree)이고, 장부 모델로 완전히 분류된다:

쓰기가 장부에 한 짓게이트판정이유
값 칸만 고침X, CNOT, CCX, SWAP✅ 청소 가능역연산도 값 칸만 도로 고침 — 계수 무접촉
값 안 씀 + 계수만Z, S, T, CZ✅ 애초에 쓰기 아님(읽기)청소할 것 자체가 없음 — Grover 오라클(위상 킥백)이 사는 이유
값 칸 + 값의존 도장Y, CY❌ 차단 (NonQfreeWrite)얽힌 타깃이면 도장이 상대 위상이 됨 — 아래 사건
줄 쪼개기 (중첩 생성)H, Rx, Ry❌ 차단 (NonQfreeWrite)생존자가 기록한 뒤엔 쪼개진 줄을 도로 못 접음

Y 사건 — 상태벡터가 반증한 판정 버그 (적대검증 5차)

초기 설계는 "Y는 치환이고 정확한 역연산을 주입하니 안전 — Silq보다 넓다"고 봤다. 열거 퍼저가 상태벡터로 반증했다. 반례와 전체 트레이스 (표기 |b,a⟩): 

use a = Qubit[1];  use b = Qubit[1];
H(b[0]);  CNOT(b[0], a[0]);  Y(a[0]);  H(b[0]);  bit m = M(b[0]);
단계장부설명
H(b)(|0,0⟩ + |1,0⟩)/√2b 중첩 — 줄 2개
CNOT(b,a)(|0,0⟩ + |1,1⟩)/√2a가 b와 얽힘 — 줄마다 a 값이 다름
Y(a)(+i|0,1⟩ −i|1,0⟩)/√2줄1은 a=0을 뒤집어 +i, 줄2는 a=1을 뒤집어 −i — 줄마다 다른 도장 = 상대 위상

a의 사망점(여기)에서 세 갈래를 비교하면:

갈래이후 H(b); M(b)의 결과
① 청소 안 함m = 0/1 각 50%
② 계약이 약속한 상태 (a만 |0⟩로, 부호 유지)b가 |−⟩ → m=1 확률 100%
③ 실제 역연산 주입 (Y†; CNOT†)도장을 도로 걷어가 b가 |+⟩ → m=0 확률 100%
②≠③ — 계약 위반. 그래서 Y·CY는 차단이다

a는 세 갈래 모두 문제없이 |0⟩로 돌아온다. 깨진 것은 생존자 b의 부호다. Y의 도장은 a 개인 소유가 아니라 줄(조합)의 공동 소유라서, 역연산이 도장을 회수하는 순간 b의 재산을 건드린 것이 된다. 정해진 기저값 위의 Y는 도장이 전역 위상이라 진짜 안전하지만 (그 정밀화는 TODO #15), 일괄 차단이 건전한 최소 수정이다. Silq도 같은 이유로 Y를 qfree에서 제외하며, Classiq은 반대 거래를 택했다 — Y를 허용하는 대신 살아있는 후보와 접촉한 생존자에게 비치환 연산을 금지(의존성 수갑). 같은 위험을 다른 자리에서 막는 것.

11주입기의 실체 — 붙여넣기 기계, 루프, 담요

rung ④(주입기)의 모델은 단순하다: "앤실라를 쓴 문장들의 목록을 가져와, 각 문장의 역을, 역순으로, 사망점에 한 번씩 붙여넣는다." 카탈로그: X→X, CNOT→CNOT, 호출→Adjoint 호출. 판정(rung ③)의 여러 조항은 이 모델이 성립하는 경우만 safe로 통과시키는 울타리다.

같은 계산, 갈리는 판정 — 루프의 붙여넣기 문제

벌거벗은 루프 — ❌ 차단 (ContainedWrite)

use a = Qubit[3];
for i in 0..2 { X(a[i]); }
// 붙여넣기 시도:
// X(a[i]);  ← i가 루프 밖엔 없음 (성립 불가)
// 게다가 문장 1개 ≠ 실행 3번

for는 "문장"이 아니라 문장을 감싼 비계다 — 카탈로그에 "for의 역" 항목이 없고, 알맹이 X(a[i])는 비계 밖으로 꺼내는 순간 의미를 잃는다.

연산으로 감싼 같은 루프 — ✅ safe

operation Flip(Qubit[3] p){
  for i in 0..2 { X(p[i]); }
}
use a = Qubit[3];
Flip(a);
Adjoint Flip(a);  // 붙여넣기 성립 — 한 문장

Flip(a)는 어엿한 문장 하나 = 실행도 한 번. 역도 문장 하나로 존재하고, 내부 루프는 Adjoint Flip 안에서 알아서 거꾸로 돈다.

호출 경계의 복합 역변환 — callee 안의 루프가 문제없는 이유

호출을 원자 단위로 통째 뒤집으면 내부의 반복이 뒤집힌 단위 안에서 그대로 미러링된다. 이것이 Classiq이 "몸통이 전부 치환인 복합문은 그 자체로 하나의 치환"이라는 규칙으로 일반화한 복합 역변환이고, Qora는 호출 경계에서 이미 암묵적으로 수행한다. 벌거벗은 루프/if로의 일반화(#11 — 루프 역방향 재건축 또는 컴파일러가 임시 연산으로 감싸는 outlining)는 실증된 설계도가 있는 공사 항목이다. 단 측정 bit 조건의 if는 감싸도 치환이 아니라서 "역변환 시점에 조건이 같다"는 증명(고전 bit 흐름, 갭 #11)이 별도로 필요하다.

담요 문제 — 질의 폭 vs 문장 폭

문장이 레지스터 전체를 한 덩어리로 쓰는 경우(브로드캐스트 X(a), 불투명 호출, 동적 인덱스 루프)를 담요 쓰기라 부른다. 문제의 본질은 mismatch 하나다: 역연산의 폭은 문장 폭인데, 질문의 폭이 원소면 문장을 원소로 썰 수 없다.

담요 + 전원 스크래치 — ✅ 문제없음

operation Bcast(Qubit[2] p){ for i in 0..1 { X(p[i]); } }
use a = Qubit[2];
Bcast(a);            // 아무도 측정 안 함
// → a: cleanup candidate — safe

질의 폭(레지스터 전체) = 역연산 폭(문장 전체). 썰 필요가 없으니 썰기 문제도 없다.

담요 + 운명 혼합 — ❌ 원소가 막힘

use a = Qubit[2];
Bcast(a);
bit c = M(a[0]);     // a[0]만 측정
// → a[1] blocked: statement-wide write of `a`

a[1]만 치워야 하는데 Adjoint Bcast(a)는 a[0]까지 되쓴다(측정값 훼손). 원소 질의에 담요 쓰기 = 차단.

혼합 케이스에서 에러를 받은 사용자의 수정 경로 4가지:

수정코드의미
① 구조 변경 (추천)Flip(a[0]); Flip(a[1]); 원소별 호출, 또는 운명별 레지스터 분리정밀한 역사 → 컴파일러가 자동으로 치움

12왜 정적 컴파일인가 — 두 개의 런타임과 언어 지형

양자 프로그램의 "런타임"은 하나가 아니라 둘이며, 이 분리가 언어 설계 전체를 결정한다.

호스트 런타임QPU 런타임
장소노트북/서버 CPU — Python이 사는 곳코히어런스 창 안 — 회로가 실행되는 곳
시간 규모밀리초마이크로초
인터프리터 입장가능물리적으로 불가 (백만 배 느림)

"측정 결과가 1이면 같은 코히어런스 창 안에서 X" 같은 회로 내 반응 로직은 컴파일돼서 QPU 옆 제어장치에 미리 가 있어야만 존재할 수 있다. 오류정정 시대(=하드웨어가 커지는 방향)는 모든 논리 연산에 이 반응(실시간 신드롬 디코딩+fixup)을 요구하므로, 하드웨어가 클수록 컴파일은 선택이 아니라 물리적 강제가 된다. 시뮬레이터에서는 두 런타임이 한 프로세스라 인터프리터가 실행 중에도 전지(全知)였다 — Silq의 우아함이 성립하는 조건이 정확히 이것이고, 실기계로 가는 순간 그 동일성이 깨지며 담요 문제 같은 "컴파일 타임에 모름" 문제가 태어난다.

지형 (2026-07 팩트체크 확정)

Qiskit/Cirq: Python이 회로를 조립(언어 아님, 보증 없음 — 현재 최대 점유). Silq: 진짜 언어 + 자동 uncompute, 그러나 인터프리터 — 실기계 경로 없음. CUDA-Q(NVIDIA): 컴파일러 인프라(MLIR→QIR) + GPU 시뮬 + 하드웨어 연결 — 언어들이 올라타는 기반. Guppy(Quantinuum, 2025): 정적 컴파일+선형 타입(누수·복제를 컴파일 에러로), 실시간 제어 축의 최전선 — 자동 uncompute는 명시적 future work. Classiq Qmod(2026): 추론 자동 uncompute를 상용 출하한 유일 시스템 — 자동화 축의 최전선. 신규 스택 전부가 정적 컴파일 방향이며, 보증을 약속한 시스템은 전부 "자동으로 못 하면 컴파일 에러 + 명시적 탈출구" 구조로 수렴했다.
