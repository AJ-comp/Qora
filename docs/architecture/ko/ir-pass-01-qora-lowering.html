1. QoraLowering — AST를 Qora IR로 옮기는 경계

Janglim의 파싱 결과를 Qora 전용 중간 표현으로 바꾸는 단계

패스 목차 2. ModuleLoader →

소스 코드를 파싱했다고 해서 이후 컴파일러 패스가 곧바로 다루기 쉬운 형태가 만들어지는 것은 아닙니다.
파서가 생성한 트리는 Qora 문법을 인식한 결과이지만, 그 구조와 타입은 파서 라이브러리인 Janglim에 맞추어져 있습니다.
따라서 QoraLowering은 이 트리를 한 번만 읽고, 이후 모든 패스가 공통으로 사용할 Qora 전용 중간 표현으로 옮깁니다.
이처럼 같은 의미를 컴파일러가 다루기 쉬운 형태로 옮기는 작업을 lowering이라고 부릅니다.
다만 이 단계에서는 프로그램의 동작을 바꾸거나 최적화하지 않으며, 같은 의미를 Qora가 직접 소유하는 자료구조로 다시 표현합니다.

PASS SPECIFICATIONQoraLowering
이 패스가 하는 일

QoraLowering은 Janglim이 만든 의미 AST를 한 번 읽어, 이후 패스가 Janglim 없이 처리할 수 있는 Qora 전용 QProgram 트리로 옮깁니다.
이 과정에서는 프로그램의 의미를 바꾸거나 최적화하지 않고, 같은 정보를 Qora가 직접 소유하는 타입과 필드로 다시 표현합니다.

  1. 최상위 항목을 소스 순서대로 분류합니다.Operation, Import, Namespace를 구분하고, namespace 안의 open과 연산도 함께 수집합니다.
  2. 연산의 서명을 QOperation과 QParam으로 만듭니다.연산 이름과 파라미터 타입을 읽으며, 고정 크기 Qubit[2]와 기호 크기 Qubit[n]도 서로 다른 필드에 기록합니다.
  3. 본문 문장을 타입이 있는 QStmt 노드로 바꿉니다.게이트, 선언, 대입, 분기, 반복문을 각각 QGate, QDecl, QAssign, QIf, QFor 등으로 변환하고 중첩 본문은 재귀적으로 처리합니다.
  4. 후속 패스가 구분해야 할 식만 구조화합니다.q[i]는 레지스터와 인덱스로 나누고 단독 측정은 QMeasure로 만듭니다.
    반면에 일반 식은 QText, QTextArg, QCond에서 사용할 문자열로 평탄화하며, 호출이 섞인 식에는 HasCall을 표시합니다.
  5. 노드 식별자와 진단 위치를 붙여 다음 단계로 넘깁니다.주요 IR 노드에는 안정 Id를 발급하고 진입 문서에는 QSpan을 기록합니다.
    다만 import 파일 읽기와 이름 해소, 의미 검증은 수행하지 않습니다.

1AST와 IR, 패스 계약

AST는 소스 코드의 문법 구조를 트리로 정리한 자료입니다.
예를 들어 H(q[0]);라는 문장을 파싱하면, Janglim은 이 문장을 게이트 호출이라는 의미를 가진 노드로 만들고 그 아래에 이름 H와 인자 q[0]을 배치합니다.
Qora는 이후 변환에 필요한 요소만 남긴 별도의 의미 AST를 사용하므로, 세미콜론이나 괄호처럼 문법을 확인하는 데 필요했던 일부 토큰은 이 트리에서 제외될 수 있습니다.

반면에 IR은 파싱이 끝난 뒤 컴파일러 내부에서 계속 사용하는 중간 자료 형식입니다.
Qora IR은 프로그램 전체를 나타내는 QProgram에서 시작하고, 그 아래에 연산 정의, 파라미터, 문장을 실제 C# record 타입으로 배치합니다.
따라서 이후 패스는 문자열 태그를 해석하거나 Janglim의 노드 구조를 다시 추측할 필요 없이, QGate나 QIf처럼 목적이 분명한 타입을 직접 처리할 수 있습니다.
여기서 말하는 Qora IR은 Qora 컴파일러의 비공개 메모리 표현이며, LLVM 기반 표준인 Microsoft QIR과는 다른 자료구조입니다. 

public static QProgram? Lower(AstSymbol? ast, bool withSpans = true)

첫 번째 인자 ast에는 Janglim이 만든 의미 AST의 루트가 전달됩니다.
루트가 자식을 가질 수 있는 AstNonTerminal이면 변환을 진행하여 새 QProgram을 반환하지만, AST가 없거나 예상한 루트 타입이 아니면 null을 반환합니다.
두 번째 인자 withSpans는 생성한 IR에 진입 문서의 문자 위치를 기록할지 결정하며, 일반적인 진입 파일은 기본값 true를 사용합니다.
다만 파싱 성공 여부와 문법 오류는 호출자인 QoraParser가 이미 처리하므로, 이 메서드는 별도의 오류 목록을 만들지 않습니다.

2입력에서 읽는 정보

QoraLowering은 원본 소스 문자열을 다시 읽지 않고 AST에 남아 있는 정보만 사용합니다.
먼저 루트의 자식 노드를 소스 순서대로 살펴보고, 각 노드의 Name이 Operation, Import, Namespace 가운데 무엇인지 확인합니다.
그런 다음 Items에 들어 있는 자식 노드와 토큰을 읽어 이름, 파라미터, 본문, 경로를 꺼냅니다.
다만 오류 표시 범위를 계산할 때에는 의미 AST에서 빠진 세미콜론과 괄호까지 포함해야 하므로, 필요한 경우 ConnectedParseTree가 가리키는 전체 구문 트리(parse tree)의 토큰도 함께 읽습니다.

입력 정보쉬운 의미실제 사용처
AstNonTerminal.Name현재 노드가 나타내는 문법 단위의 이름최상위 항목과 각 문장을 알맞은 IR 타입으로 분기
AstNonTerminal.Items현재 문법 단위 안에 들어 있는 이름, 식, 하위 문장연산 이름, 파라미터, 인자, 조건, 본문 추출
AstTerminal식별자, 숫자, 연산자처럼 더 나눌 필요가 없는 토큰이름과 리터럴을 읽고 토큰의 시작·끝 위치 계산
ConnectedParseTree의미 AST가 만들어지기 전의 완전한 문법 트리세미콜론과 괄호를 포함하는 문장 전체 범위 계산
withSpans소스 위치를 IR에 남길지 결정하는 스위치진입 문서는 true, import 문서는 false

이처럼 의미 AST와 구문 트리를 함께 사용하는 이유는 두 트리의 목적이 다르기 때문입니다.
의미 AST는 변환에 필요한 내용만 간결하게 제공하는 반면, 구문 트리는 사용자가 실제로 입력한 문장의 전체 범위를 보존합니다.
따라서 구조 변환에는 의미 AST가 적합하고, 편집기 진단 위치에는 구문 트리가 적합합니다.

3자료구조와 역할

Qora IR은 일반 명령을 일렬로 나열하는 저수준 명령 목록이 아니라, 소스의 중첩 구조를 유지하는 record 트리입니다.
프로그램의 최상위 루트는 QProgram이며, 그 아래의 QOperation은 하나의 연산 정의를 나타냅니다.
또한 연산 본문은 별도의 블록 객체를 사용하지 않고 IReadOnlyList<QStmt>에 문장을 순서대로 보관합니다.
그 결과 if나 반복문의 내부 문장도 같은 QStmt 목록을 재귀적으로 가지므로, 소스에서 보았던 제어 흐름을 그대로 따라갈 수 있습니다. 

QProgram
├─ Imports: IReadOnlyList<QImport>?
├─ Opens: namespace → IReadOnlyList<QOpen>
└─ Operations: IReadOnlyList<QOperation>
   ├─ Name / Namespace
   ├─ Params: IReadOnlyList<QParam>
   └─ Body: IReadOnlyList<QStmt>
      ├─ QUse / QGate / QDecl / QAssign
      └─ QIf / QFor / QWhile / QRepeat
         └─ 다시 IReadOnlyList<QStmt>
자료구조담는 값이 값이 필요한 이유변환 직후 상태
List<QOperation>현재 파일에서 찾은 모든 연산 정의최상위 연산과 namespace 내부 연산을 하나의 실행 단위로 모으기 위해 사용연산끼리의 소스 순서 유지
List<QImport>따옴표를 제거한 상대 경로와 선언 위치현재 패스가 파일을 열지 않고도 다음 패스에 로드 대상을 전달하기 위해 사용import가 없으면 null
Dictionary<string, IReadOnlyList<QOpen>>namespace 이름별 open 대상 목록같은 namespace 블록이 여러 번 나와도 가시성 지시문을 한 키 아래 모으기 위해 사용namespace 블록이 없으면 null입니다.
반면에 namespace는 있지만 open 선언이 없으면 해당 이름과 빈 목록을 보존합니다.
QProgram연산 목록, import 목록, namespace별 open 목록이후 패스가 받는 단일 루트 객체를 만들기 위해 사용현재 파일 한 개를 나타내는 IR
QOperation연산 이름, 파라미터, 본문, 원래 namespace연산 정의에 필요한 정보를 한 노드에서 읽기 위해 사용이름은 아직 완전 수식 이름으로 해소되지 않은 상태
QParam파라미터 이름, 종류, 고정 또는 기호 레지스터 크기이후 호출 검사와 제네릭 크기 특수화에 필요한 서명을 만들기 위해 사용Qubit[2]는 RegisterSize=2, Qubit[n]은 SizeParam="n"
QStmt할당, 게이트, 선언, 분기, 반복문 등 한 문장의 구조문장 종류를 문자열이 아닌 C# 파생 타입으로 구분하기 위해 사용중첩 본문까지 재귀적으로 변환된 상태
구조화하는 값과 텍스트로 남기는 값

큐빗 한 칸을 가리키는 q[0]은 레지스터 이름과 인덱스를 따로 읽어 QQubitArg("q", "0")로 만듭니다.
반면에 현재 Qora 파이프라인은 theta / 2를 나눗셈 노드로 다시 분석하거나 계산 순서를 바꾸지 않습니다.
따라서 lowering은 이 식을 하나의 텍스트 값인 "theta / 2"로 보존합니다.
다만 텍스트 안에 함수 호출이 섞여 있으면 OpenQASM으로 그대로 내릴 수 없으므로, HasCall을 true로 기록하여 검증 패스가 해당 경우를 놓치지 않도록 합니다.

4실제 변환 순서

변환은 루트에서 본문으로 내려가며 진행됩니다.
먼저 프로그램 수준의 임시 수집 자료구조를 준비하고, 다음으로 각 연산의 이름과 파라미터 목록을 만든 뒤, 마지막으로 연산 본문의 문장을 재귀적으로 변환합니다.
이 순서를 따르면 상위 구조가 어느 하위 값을 소유하는지 분명해지고, 소스의 문장 순서도 자연스럽게 유지됩니다.

4.1 프로그램 루트와 모듈 선언 수집

루트가 올바른 AstNonTerminal인지 확인한 뒤, 빈 연산 목록, import 목록, namespace별 open 사전을 준비합니다.
그리고 루트의 자식을 한 번 순회하면서 Operation, Import, Namespace 태그에 따라 처리 경로를 나눕니다.
최상위 Operation은 즉시 QOperation으로 변환하여 연산 목록에 추가합니다.
반면에 Import는 따옴표만 제거한 상대 경로를 QImport.Target에 저장하며, 실제 파일 접근은 다음 ModuleLoader까지 미룹니다.

Namespace를 만나면 점으로 연결된 namespace 이름을 먼저 읽습니다.
그 안의 연산은 같은 방식으로 변환하되 QOperation.Namespace에 원래 namespace 이름을 추가하며, open 선언은 해당 namespace를 키로 하는 목록에 저장합니다.
동일한 namespace 블록이 소스에 여러 번 나타날 수 있으므로, 기존 목록을 지우지 않고 뒤에 이어 붙입니다.

4.2 연산 이름과 파라미터 목록 변환

각 연산에서는 먼저 이름 토큰을 읽고, 이어서 Param 태그가 붙은 자식을 모두 QParam으로 변환합니다.
고전 파라미터인 int, bit, float, angle은 각각 대응하는 QType 값을 받습니다.
큐빗 파라미터는 선언 형태에 따라 세 가지로 나뉩니다.
Qubit q는 단일 큐빗, Qubit[2] q는 크기가 확정된 레지스터, Qubit[n] q는 나중에 호출 지점에서 크기가 결정될 레지스터입니다.
따라서 마지막 형태에서는 RegisterSize를 비워 두고 SizeParam에 n을 기록합니다.

여기서 의미 AST는 Qubit 키워드 자체를 남기지 않는다는 점에 주의해야 합니다.
따라서 파라미터 토큰에 고전 타입 키워드가 없으면 큐빗으로 판단하고, 남은 토큰의 모양으로 세 형태를 구분합니다.
이름 하나만 남으면 Qubit q, 숫자와 이름이 남으면 Qubit[2] q, 크기 기호와 이름이 차례로 남으면 Qubit[n] q에 해당합니다.

4.3 본문 문장을 재귀적으로 변환

파라미터를 제외한 연산 자식은 소스 순서대로 본문 문장으로 간주하며, 각 문장은 의미 태그에 대응하는 QStmt 파생 타입으로 바뀝니다.
예를 들어 큐빗 레지스터 준비 선언은 QUse, 게이트 또는 사용자 연산 호출은 QGate, 변수 선언은 QDecl로 변환됩니다.
또한 QIf, QFor, QWhile, QRepeat는 내부에 다시 문장 목록을 가지므로, 같은 변환 함수를 재귀적으로 적용합니다.

AST 의미 태그생성하는 IR변환 시 보존하는 정보

GateQGateAdjoint/Controlled 접두사, 점으로 연결된 호출 이름, 인자 목록
ConstDeclQDecl(IsConst: true)선택적 타입, 이름, 초기값
VarDeclQDecl(IsConst: false)선택적 타입, 이름, 초기값

ForQFor반복 변수, 시작 식, 종료 식, 본문
While / RepeatQWhile / QRepeat조건과 반복 본문

4.4 호출, 식, 조건의 예외 규칙

소스 문법에서는 내장 게이트 호출과 사용자 연산 호출이 모두 이름(인자) 형태이므로, lowering 단계에서는 둘을 모두 QGate로 만듭니다.
어느 이름이 내장 게이트이고 어느 이름이 사용자 연산인지는 import와 이름 해소가 끝난 뒤에야 정확히 판단할 수 있기 때문입니다.
따라서 이 단계는 호출 이름과 인자를 잃지 않고 보존하는 데 집중합니다.

선언이나 대입의 오른쪽 전체가 정확히 M(q[i]) 한 번으로 구성되어 있으면 측정 의미를 분명히 나타내는 QMeasure를 생성합니다.
반면에 다른 함수 호출이거나 호출과 산술식이 섞인 경우에는 내용을 삭제하지 않고 전체 식을 QText로 남긴 뒤 HasCall=true를 설정합니다.
그러면 이후 검증 패스가 OpenQASM으로 표현할 수 없는 호출식을 명확한 의미 오류로 보고할 수 있습니다.

조건도 기본적으로 공백으로 연결한 문자열로 보존하지만, !expr의 부정 범위는 별도로 보호합니다.
Qora 문법에서 !는 뒤따르는 식 전체를 부정하므로, 해당 식을 괄호로 감싸서 나중에 OpenQASM이 첫 토큰만 부정하는 것으로 해석하지 않도록 합니다.
또한 괄호 안팎에도 공백을 두어, 이후 이름 변경 패스가 식별자를 독립된 토큰으로 찾을 수 있도록 합니다.

오류 유예와 방어 처리

레지스터 크기 숫자가 32비트 정수 범위를 벗어나면 int.Parse 예외로 컴파일을 중단하지 않고 크기 -1을 기록합니다.
이후 QoraValidator는 이 값을 잘못된 크기로 판정하여 QSEM016을 보고합니다.
또한 현재 문법이 생성하지 않는 알 수 없는 문장 태그가 들어오면 방어 경로에서 빈 인자의 QGate를 만들며, 정상적인 파서 출력에서는 이 경로를 사용하지 않습니다.

5입력부터 출력까지 이어지는 예제

다음 예제는 import 경로, namespace, open 선언, 두 종류의 파라미터, 일반 식 인자, 큐빗 인자, 측정 선언을 한 흐름에서 보여줍니다.
아래 세 단계는 서로 다른 예제를 나열하는 것이 아니라, 같은 Bell 연산이 소스에서 AST를 거쳐 Qora IR로 바뀌는 과정을 이어서 나타냅니다. 

import "lib/helpers.qor";

namespace Prep {
    open Shared;

    operation Bell(angle theta, Qubit[2] q) {
        Rx(theta / 2, q[0]);
        CNOT(q[0], q[1]);
        bit result = M(q[1]);
    }
}

5.1 파서가 넘기는 의미 AST

파서는 먼저 구문을 검사한 뒤, 의미가 있는 부분을 다음과 같은 트리로 정리합니다.
아래 표기는 설명을 위해 핵심 노드만 남긴 형태이며, 실제 디버그 출력 형식을 그대로 복사한 것은 아닙니다.
그래도 lowering이 읽는 Import, Namespace, Open, Operation, Param, 문장 태그의 부모·자식 관계는 그대로 드러납니다. 

Program
├─ Import
│  └─ "lib/helpers.qor"
└─ Namespace: Prep
   ├─ Open: Shared
   └─ Operation: Bell
      ├─ Param: angle theta
      ├─ Param: 2 q
      ├─ Gate: Rx → Expr(theta / 2), Qubit(q, 0)
      ├─ Gate: CNOT → Qubit(q, 0), Qubit(q, 1)
      └─ VarDecl: bit result → Expr(Call M(Qubit(q, 1)))

5.2 최상위 수집 과정

순회가 시작되면 세 임시 자료구조는 모두 비어 있습니다.
먼저 Import를 처리하면서 경로의 따옴표를 제거하여 imports에 추가합니다.
이어서 Namespace를 처리하면서 Shared를 opens["Prep"]에 넣고, 그 안의 Bell을 변환하여 operations에 추가합니다.
결과적으로 새 QProgram을 만들기 직전의 수집 상태는 다음과 같습니다. 

operations = [ Bell 연산의 QOperation ]
imports   = [ QImport("lib/helpers.qor") ]
opens     = { "Prep": [ QOpen("Shared") ] }

5.3 생성되는 Qora IR

마지막으로 세 임시 자료구조를 QProgram에 넣습니다.
이때 Bell이라는 짧은 이름과 Prep이라는 원래 namespace를 분리해서 보존하며, Prep.Bell이라는 완전 수식 이름은 아직 만들지 않습니다.
또한 theta / 2는 일반 식이므로 QTextArg에 텍스트로 남지만, q[0]과 q[1]은 레지스터와 인덱스가 분리된 QQubitArg가 됩니다. 

// 설명을 위해 생성자 인자를 펼친 개념적 결과
new QProgram(
  Operations: [
    new QOperation(
      Name: "Bell",
      Params: [
        new QParam("theta", QType.Angle, RegisterSize: null),
        new QParam("q", QType.Qubit, RegisterSize: 2)
      ],
      Body: [
        new QGate([], "Rx", [
          new QTextArg("theta / 2"),
          new QQubitArg("q", "0")
        ]),
        new QGate([], "CNOT",  [new QQubitArg("q", "0"),
                                 new QQubitArg("q", "1")]),
        new QDecl(
          IsConst: false,
          Type: QType.Bit,
          Name: "result",
          Value: new QMeasure(new QQubitArg("q", "1")))
      ],
      Namespace: "Prep")
  ],
  Imports: [new QImport("lib/helpers.qor")],
  Opens: { "Prep": [new QOpen("Shared")] })
소스의 값변환 결과아직 결정하지 않는 내용

theta / 2QTextArg("theta / 2")식의 타입과 값

예제에서 확인할 수 있는 책임 경계

이 결과에는 다음 패스가 작업을 시작하는 데 필요한 원본 정보가 모두 남아 있습니다.
그러나 import 파일은 아직 열리지 않았고, Shared가 실제 namespace인지도 확인되지 않았으며, Bell도 아직 Prep.Bell로 바뀌지 않았습니다.
따라서 import 병합은 ModuleLoader가 맡고, 완전 수식 이름 결정과 open 검사는 이후 Resolver가 맡습니다.

6Id와 소스 위치 규칙

lowering은 프로그램 구조만 만드는 것이 아니라, 이후 분석 결과를 같은 노드에 계속 연결할 수 있도록 식별자도 부여합니다.
QProgram, QOperation, QParam, 모든 QStmt는 생성될 때 QNodeIds.Next()에서 고유한 Id를 받습니다.
이후 패스가 C#의 with 식으로 노드의 이름이나 내용을 바꾸더라도 이 Id는 복제되어 유지됩니다.
따라서 별도 분석표인 SemanticModel은 IR 내용이 바뀐 뒤에도 같은 선언과 문장을 계속 식별할 수 있습니다.
반면에 QImport, QOpen, QArg, QExpr, QCond에는 안정 Id를 부여하지 않습니다.

생성 노드위치가 가리키는 범위사용 목적
QImport / QOpen선언문 전체경로 또는 namespace 관련 오류 표시
QOperation연산 이름 토큰연산 수준 오류를 이름에 표시
QParam파라미터 이름 토큰서명 오류를 해당 이름에 표시
QStmt세미콜론과 괄호를 포함한 문장 전체문장 의미 오류 범위 표시

소스 위치는 QSpan(Start, End)에 저장하며, Start는 첫 문자의 위치이고 End는 마지막 문자 바로 다음 위치입니다.
즉 [Start, End) 반열림 구간이므로, 편집기 API가 길이를 End - Start로 바로 계산할 수 있습니다.
특히 문장 범위는 연결된 parse tree의 첫 토큰부터 마지막 토큰까지 계산합니다.
따라서 의미 AST에서 제외된 세미콜론이나 괄호도 진단 표시 범위에 포함됩니다.

입력 파일withSpans생성되는 위치설계 이유
진입 문서true지원 노드에 QSpan 기록사용자가 연 문서의 정확한 오류 범위 제공
import 문서falseSpan = null다른 파일의 문자 위치를 진입 문서 위치로 잘못 보고하는 문제 방지
동시성 메모

소스 위치를 생략하는 상태는 모든 내부 함수에 반복해서 인자로 넘기지 않고 [ThreadStatic] _noSpans에 저장합니다.
하나의 Lower 호출은 같은 스레드에서 동기적으로 끝나므로 호출 도중 값이 바뀌지 않으며, 다른 스레드의 lowering과도 상태를 공유하지 않습니다.
IR을 조립하는 임시 가변 상태는 이 호출 안에서 생성한 목록과 사전으로 제한됩니다.
다만 노드를 생성할 때에는 고유한 Id를 발급하기 위해 프로세스 전역 QNodeIds 카운터가 원자적으로 증가합니다.
이 패스는 파일 I/O를 수행하지 않으며, SemanticModel도 변경하지 않습니다.

7다음 패스에 넘기는 계약

성공 결과인 QProgram은 문장 구조가 타입으로 구분되어 있지만, 아직 현재 파일 한 개만 나타냅니다.
Operations에는 이 파일에서 발견한 연산이 상대적인 소스 순서대로 들어 있고, Imports에는 아직 열지 않은 상대 경로가 들어 있습니다.
또한 Opens에는 namespace별 가시성 지시문이 저장되어 있지만, 대상 namespace의 존재 여부는 아직 확인되지 않았습니다.
이 상태를 받는 즉시 다음 패스인 ModuleLoader가 import 경로를 실제 파일과 연결하고, 가져온 파일의 연산과 open 정보를 현재 프로그램에 병합합니다.

전달 항목QoraLowering이 보장하는 상태후속 단계의 작업
Operations현재 파일의 연산과 본문이 모두 Qora IR 타입으로 변환된 상태import 파일의 연산을 뒤에 병합
Imports따옴표가 제거된 상대 경로와 진입 문서 선언 위치ModuleLoader가 경로 해석, 재귀 로드, 순환 검사를 수행하고, 확장 성공 후 null로 소비
Opens현재 파일에서 선언한 namespace별 open 목록import 파일의 목록과 병합한 뒤 Resolver가 이름 해소에 사용
호출 이름소스에 적힌 단순 이름 또는 점으로 연결된 이름Resolver가 완전 수식 이름으로 해소
소스 위치진입 문서 노드는 실제 범위, import 문서 노드는 null후속 진단에서 진입 문서 위치로 안전하게 변환

결과적으로 이 패스의 완료 조건은 “프로그램 전체의 의미가 유효함”이 아니라 “Janglim 없이도 다음 패스가 같은 소스 정보를 읽을 수 있음”입니다.
따라서 이름 중복, 알 수 없는 호출, 잘못된 인자 타입처럼 여러 파일의 이름과 선언 정보가 필요한 판정은 의도적으로 뒤 단계에 남겨 둡니다.
이러한 책임 분리를 통해 lowering은 표현 변환에 집중하고, 각 후속 패스는 자신에게 필요한 정보가 준비된 시점에 오류를 판정할 수 있습니다.
