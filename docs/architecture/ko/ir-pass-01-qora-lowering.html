1. QoraLowering — AST를 Qora IR로 옮기는 경계

파서가 알아본 구조를 Qora가 직접 처리할 수 있는 자료로 바꾸는 단계

패스 목차 2. ModuleLoader →

QoraLowering이 호출되기 전에, Janglim 파서는 Qora 코드가 문법에 맞는지 확인하고 코드의 구성 요소를 의미 AST로 정리합니다.
그러나 이 트리는 파서 라이브러리인 Janglim의 자료구조에 직접 의존하므로, 이후 컴파일러 패스까지 그대로 사용하면 파서와 후속 처리 단계 사이의 결합도가 지나치게 높아집니다.
따라서 파서가 알아낸 의미는 그대로 유지하면서도, 같은 내용을 Qora 전용 중간 표현으로 옮길 필요가 있습니다.
이 역할을 QoraLowering이 맡으며, 변환된 프로그램은 QProgram에 담깁니다.
아래에서는 main.qor를 파싱한 결과인 의미 AST를 입력으로 두고, AST의 각 부분이 어떤 IR 값으로 바뀌는지 바로 확인합니다.

PASS SPECIFICATIONQoraLowering
이 패스가 하는 일

QoraLowering은 파일 하나의 의미 AST를 읽어 연산, 파라미터, 문장, import와 open을 Qora IR 노드로 바꿉니다.
이때 프로그램의 동작을 최적화하거나 이름의 유효성을 판정하지 않으며, 다음 패스가 Janglim 없이 같은 정보를 읽을 수 있는 형태를 만드는 데 집중합니다.

입력

QoraLowering의 입력은 파서가 미리 만든 Janglim AST입니다

왼쪽 Qora 소스를 오른쪽 의미 AST로 바꾸는 작업은 QoraLowering이 아니라 앞선 파싱 단계에서 수행됩니다.
파서가 구문 분석과 AST 생성을 마친 뒤 QoraLowering.Lower(ast)를 호출하므로, 이 패스의 실제 처리는 오른쪽 Program 노드를 읽는 지점부터 시작합니다.

오른쪽 의미 AST에는 lowering에 필요한 구조와 값만 남으며, 괄호와 세미콜론처럼 문법 확인에만 사용한 토큰은 제외됩니다.
반면에 Qubit을 비롯한 타입 키워드는 파라미터의 타입을 결정하는 의미 정보이므로 AST에 그대로 남습니다.

1

최상위 항목을 QProgram의 세 영역으로 나눕니다

QoraLowering은 입력 AST의 최상위 항목을 원래 소스 순서대로 방문합니다.
아래 도식에서는 입력 AST의 각 노드가 원본 main.qor에서 나타내는 범위를 실제 코드 위에 표시했습니다.
번호가 붙은 색상 범위에서 화살표를 따라가면, 해당 코드가 QProgram의 어느 위치에 저장되는지 바로 확인할 수 있습니다.

이때 import "lib/gates.qor";에서는 경로 문자열만 QImport로 옮기며, 대상 파일은 읽지 않습니다.
실제 파일을 찾아 읽는 작업은 다음 패스인 ModuleLoader가 맡습니다.

namespace 블록 자체는 별도의 QNamespace 노드로 남지 않습니다.
대신 그 안의 연산에는 Namespace = "Demo"를 붙이고, open Shared는 Demo를 키로 하는 목록에 넣습니다.
따라서 같은 namespace 블록이 여러 번 등장하더라도 연산과 open을 같은 이름 아래 계속 모을 수 있습니다.

2

연산 선언부를 읽어 QOperation의 이름·namespace와 QParam을 만듭니다

QoraLowering은 연산 AST에서 연산 이름과 파라미터를 읽고, 그 연산을 감싼 namespace가 있으면 선언 위치도 함께 기록합니다.
아래 도식에는 두 연산의 실제 본문을 모두 남겨 두고, 이번 절에서 읽는 선언부만 밝게 표시했습니다.

파서는 Qubit[] q를 Param(ArrayType(Qubit), q) 형태의 의미 AST로 전달합니다.
따라서 QoraLowering은 q에서 파라미터 이름을, Qubit에서 원소 타입을 읽고, ArrayType 노드가 있다는 사실로 IsArray=true를 정합니다.
또한 소스의 Qubit[]에는 크기 숫자가 없으므로, 구체적인 큐빗 배열 크기를 나타내는 RegisterSize는 이 시점에 null입니다.
int[] 같은 고전 배열은 IsArray로 배열 여부만 기록하며, RegisterSize를 배열 길이로 사용하지 않습니다.

원래 선언AST에서 읽는 값변환 결과
Qubit qQubit, qType=Qubit, IsArray=false
Qubit[] qArrayType(Qubit), qType=Qubit, IsArray=true
angle thetaangle, thetaType=Angle, IsArray=false
int[] valuesArrayType(int), valuesType=Int, IsArray=true

이 단계에서는 Prepare와 Demo를 Demo.Prepare라는 하나의 문자열로 합치지 않습니다.
연산 이름은 Name = "Prepare", 선언된 namespace는 Namespace = "Demo"로 각각 보존합니다.

3

본문의 각 문장을 역할이 분명한 QStmt로 바꿉니다

연산 본문은 위에서 아래로 읽으며, 각 문장을 그 역할에 맞는 IR 타입으로 바꿉니다.
중첩된 if 본문도 같은 절차를 다시 적용하므로 소스의 문장 순서와 포함 관계가 그대로 유지됩니다.

위치Qora 소스생성되는 IR과 핵심 값
Main 1use q = Qubit[2];QUse · Name="q" · Size=2
Main 2Demo.Prepare(q);QGate · Name="Demo.Prepare" · 인자 텍스트 "q"
Main 3bit[] results = new bit[2];QDecl · Type=Bit · IsArray=true · 값은 QArrayNew(Bit, 2)
Main 4results[0] = M(q[0]);QAssign · Name="results" · Index="0" · 값은 QMeasure(q[0])
Main 5if (results[0] == 1) { ... }QIf · 조건 "results[0] == 1" · Else=[]
if 내부X(q[1]);QGate · Name="X" · 인자 q[1]
Prepare 1for i in 0..q.Count-1 { ... }QFor · Var="i" · From="0" · To="q.Count - 1"
for 내부H(q[i]);QGate · Name="H" · 인자 q[i]
Prepare 2Flip(q);QGate · Name="Flip" · 인자 텍스트 "q"

이 시점에는 내장 게이트 H와 사용자 연산 호출 Flip이 모두 QGate입니다.
두 호출의 문법 모양이 같고 import 파일도 아직 읽지 않았으므로, 어느 쪽이 사용자 연산인지는 뒤의 이름 해소 단계에서 판정합니다.

4

큐빗 위치와 단독 측정은 구조화하고, 일반 식은 텍스트로 보존합니다

소스의 값lowering 결과이렇게 나누는 이유
q[0]QQubitArg(Reg="q", Index="0")어느 레지스터의 몇 번째 큐빗인지 후속 패스가 바로 읽어야 합니다.
qQTextArg(Text="q")단일 큐빗인지 전체 레지스터인지는 선언과 호출 대상을 확인한 뒤 판정합니다.
M(q[0])QMeasure(Target=q[0])오른쪽 식 전체가 정확히 단독 측정일 때만 측정 노드로 구분합니다.
results[0] == 1QCond(Text="results[0] == 1")QoraLowering은 일반 조건식의 연산 트리를 만들지 않고 현재 토큰 순서를 텍스트로 보존합니다.
다만 조건 속 측정은 이후 전용 패스가 별도 문장으로 바꿀 수 있습니다.
단독 측정 
results[0] = M(q[0]);

Value = QMeasure(q[0])
측정 대상이 분명하므로 전용 노드가 만들어집니다.
호출이 다른 식과 섞인 경우 
var value = M(q[0]) + 1;

Value = QText("M(q[0]) + 1")
HasCall = true
내용을 버리지 않고 그대로 보존한 뒤, 후속 검증이 오류를 판정할 수 있도록 표시합니다.

따라서 lowering은 복잡한 호출식을 발견해도 이 자리에서 오류 목록을 만들지 않습니다.
대신 HasCall=true를 남겨 QoraValidator가 OpenQASM으로 표현할 수 없는 경우를 QSEM005로 보고하도록 합니다.

다음 패스가 그 파일을 읽어 별도의 단일 파일 QProgram으로 만든 뒤에야 두 프로그램이 병합됩니다.

6

노드를 만들 때 함께 기록한 Id와 소스 위치를 확인합니다

앞 단계에서 각 IR 노드를 생성할 때, 계속 추적해야 하는 주요 노드에는 고유한 Id도 함께 발급했습니다.
또한 진입 문서에서 오류가 발생했을 때 정확한 위치를 표시할 수 있도록 지원 노드에는 QSpan(Start, End)를 기록했습니다.

생성 노드고유 Id진입 문서에서 Span이 가리키는 범위
QProgram있음Span 필드 없음
QOperation있음Main, Prepare 같은 연산 이름 토큰
QParam있음파라미터 이름 토큰
모든 QStmt있음키워드·괄호·중괄호·세미콜론을 포함한 문장 전체
QImport / QOpen없음선언문 전체
QArg / QExpr / QCond없음자체 Span 없음
진입 파일 · main.qor 기본값 withSpans=true를 사용하므로 지원 노드에 실제 문자 범위를 기록합니다.
import 파일 · gates.qor withSpans=false로 다시 lowering하므로 모든 지원 Span이 null입니다.
다른 파일의 위치가 진입 문서 위치로 오해되는 일을 막기 위한 규칙입니다.

Id 번호는 프로세스 전체에서 계속 증가하므로 예제처럼 특정 숫자로 고정할 수 없지만, Id가 있는 각 노드는 서로 다른 값을 받습니다.
이로써 QoraLowering은 Janglim AST와의 연결을 끝내고, 구조·원본 값·추적 정보를 갖춘 단일 파일 QProgram을 다음 패스에 반환합니다.

CLI 단계 출력과 원시 lowering 결과의 차이

CLI의 --stages에 표시되는 ir은 상황에 따라 이미 import 병합, 조건 속 측정 변환과 이름 해소를 거친 결과일 수 있습니다.
따라서 그 출력에서 Shared.Flip이나 import 파일의 연산이 보이더라도, 해당 값이 QoraLowering 직후부터 존재했다고 해석하면 안 됩니다.
위의 QoraLowering 출력 트리는 ModuleLoader와 Resolver가 실행되기 전의 원시 QProgram만 나타냅니다.

Apply(q) 안의 qQTextArg("q")전체 문장은 QGate이며, 이 인자가 전체 레지스터인지 단일 큐빗인지는 후속 선언 검사에서 정해집니다.

이는 의미상 안전한 no-op이 아니며, 정상적인 파서 출력에서는 사용되지 않습니다.

전달 항목QoraLowering이 보장하는 상태후속 단계의 작업
Operations현재 파일의 연산, 파라미터, 중첩 본문이 모두 Qora IR 타입으로 변환된 상태ModuleLoader가 import 파일의 연산을 병합합니다.
Imports따옴표만 제거한 원문 경로 문자열이며, 진입 문서는 보통 Span이 있고 import 문서는 Span이 없습니다.경로 기준이 있으면 ModuleLoader가 정규화·재귀 로드·순환 검사를 수행한 뒤 루트 Imports를 null로 소비합니다.
반면에 기준이 없으면 QSEM020을 기록하고 원래 Imports를 유지합니다.
Opensnamespace별 open 목록이며 대상의 존재 여부는 확인하지 않은 상태파일 병합 뒤 Resolver가 이름 해소에 사용합니다.
호출 이름소스에 적힌 단순 이름 또는 점으로 연결된 이름Resolver가 내장 게이트와 사용자 연산 후보를 구분합니다.
사용자 연산 참조는 완전 수식 이름으로 바꾸고 CalleeOpId를 연결하지만, 내장 게이트는 표준 짧은 이름을 유지합니다.
식과 조건필요한 큐빗·측정 구조와 나머지 텍스트, HasCall 표시import 오류가 없으면 MeasureConditionLowering이 조건 속 측정을 먼저 재작성하고, 이어서 Resolver가 이름을 해소합니다.
Resolver 오류도 없을 때 QoraValidator가 호출식의 표현 가능성과 타입을 검사합니다.
