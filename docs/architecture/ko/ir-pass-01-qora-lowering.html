1. QoraLowering — AST를 Qora IR로 옮기는 경계

파서가 알아본 구조를 Qora가 직접 처리할 수 있는 자료로 바꾸는 단계

패스 목차 2. ModuleLoader →

QoraLowering이 호출되기 전에, Janglim 파서는 Qora 코드가 문법에 맞는지 확인하고 코드의 구성 요소를 의미 AST로 정리합니다.
그러나 이 트리는 파서 라이브러리인 Janglim의 자료구조에 직접 의존하므로, 이후 컴파일러 패스까지 그대로 사용하면 파서와 후속 처리 단계 사이의 결합도가 지나치게 높아집니다.
따라서 파서가 알아낸 의미는 그대로 유지하면서도, 같은 내용을 Qora 전용 중간 표현으로 옮길 필요가 있습니다.
이 역할을 QoraLowering이 맡으며, 변환된 프로그램은 QProgram에 담깁니다.
아래에서는 main.qor를 파싱한 결과인 의미 AST를 입력으로 두고, AST의 각 부분이 어떤 IR 값으로 바뀌는지 바로 확인합니다.

PASS SPECIFICATIONQoraLowering
이 패스가 하는 일

QoraLowering은 파일 하나의 의미 AST를 읽어 연산, 파라미터, 문장, import와 open을 Qora IR 노드로 바꿉니다.
이때 프로그램의 동작을 최적화하거나 이름의 유효성을 판정하지 않으며, 다음 패스가 Janglim 없이 같은 정보를 읽을 수 있는 형태를 만드는 데 집중합니다.

입력

QoraLowering의 입력은 파서가 미리 만든 Janglim AST입니다

왼쪽 Qora 소스를 오른쪽 의미 AST로 바꾸는 작업은 QoraLowering이 아니라 앞선 파싱 단계에서 수행됩니다.
파서가 구문 분석과 AST 생성을 마친 뒤 QoraLowering.Lower(ast)를 호출하므로, 이 패스의 실제 처리는 오른쪽 Program 노드를 읽는 지점부터 시작합니다.

Janglim AST에서 Use, Gate, VarDecl은 서로 다른 전용 클래스가 아니라 AstNonTerminal.Name에 들어가는 구분값입니다.
해당 노드의 실제 자식은 Items에 소스 순서대로 저장되고, 식별자·타입 키워드·숫자·연산자는 더 나뉘지 않는 AstTerminal로 남습니다.
다만 괄호와 세미콜론처럼 문법 확인에만 사용한 토큰은 의미 AST에서 제외됩니다.

1

최상위 항목을 QProgram의 세 영역으로 나눕니다

QoraLowering은 입력 AST의 루트인 Program에서 Items[0..2]를 소스 순서대로 방문합니다.
아래 도식은 원본 코드를 다시 보여주지 않고, 이 패스가 실제로 전달받는 AstNonTerminal과 AstTerminal 계층에서 시작합니다.
번호가 붙은 AST 노드에서 화살표를 따라가면, 해당 노드가 QProgram의 어느 필드에 기록되는지 확인할 수 있습니다.

이때 import "lib/gates.qor";에서는 경로 문자열만 QImport로 옮기며, 대상 파일은 읽지 않습니다.1
실제 파일을 찾아 읽는 작업은 다음 패스인 ModuleLoader가 맡습니다.1 보충

namespace 블록 자체는 별도의 QNamespace 노드로 남지 않습니다.34
대신 그 안의 연산에는 Namespace = "Demo"를 붙이고, open Shared는 Demo를 키로 하는 목록에 넣습니다.34
따라서 같은 namespace 블록이 여러 번 등장하더라도 연산과 open을 같은 이름 아래 계속 모을 수 있습니다.3 보충4 보충

도식 표기 기준

1번 절 이후의 도식에서도 QoraLowering이 실제로 전달받는 입력은 변함없이 Janglim AST입니다.
다만 AST 객체 계층을 매번 펼치면 변환 규칙보다 노드 구조를 해석하는 데 시선이 분산되므로, 아래 도식부터는 이해를 돕기 위해 각 AST 노드에 대응하는 원본 Qora 코드 범위를 왼쪽에 표시합니다.
따라서 화살표는 소스 코드를 직접 입력받는다는 뜻이 아니라, 강조한 코드 범위에 대응하는 AST 노드가 오른쪽 Qora IR로 변환됨을 나타냅니다.

2

연산 선언부를 읽어 QOperation의 이름·namespace와 QParam을 만듭니다

QoraLowering은 연산 AST에서 연산 이름과 파라미터를 읽고, 그 연산을 감싼 namespace가 있으면 선언 위치도 함께 기록합니다.
아래 도식에는 두 연산의 실제 본문을 모두 남겨 두고, 이번 절에서 읽는 선언부만 밝게 표시했습니다.

파서는 q: Qubit[]를 Param(q, ArrayType(Qubit)) 형태의 의미 AST로 전달합니다.6
따라서 QoraLowering은 q에서 파라미터 이름을, Qubit에서 원소 타입을 읽고, ArrayType 노드가 있다는 사실로 IsArray=true를 정합니다.6
또한 소스의 Qubit[]에는 크기 숫자가 없으므로, 구체적인 큐빗 배열 크기를 나타내는 RegisterSize는 이 시점에 null입니다.6
int[] 같은 고전 배열은 IsArray로 배열 여부만 기록하며, RegisterSize를 배열 길이로 사용하지 않습니다.6 보충

3

연산 본문의 각 문장을 QStmt로 바꾸어 QOperation.Body에 저장합니다

QDecl.Value와 QAssign.Value에 들어가는 QArrayNew와 QMeasure는 독립된 본문 문장이 아니라, 선언이나 대입의 오른쪽 값을 나타내는 QExpr입니다.3.14.1
따라서 이 값들은 QOperation.Body에 별도의 QStmt로 추가되지 않고 해당 QDecl 또는 QAssign 안에 포함됩니다.3.14.1

이 단계에서는 게이트 적용과 사용자 연산 호출을 포함한 모든 호출형 문장을 QGate로 보존합니다.25.11.12
QoraLowering은 호출 이름과 인자만 기록하고, 이후 Resolver가 이름을 해소하여 실제 호출 대상을 결정합니다.25.11.12

4

호출 인자의 단독 인덱스 참조와 단독 측정은 구조화하고, 일반 식·조건은 텍스트로 보존합니다

3절에서 만든 QGate, QAssign, QIf 안에는 각각 호출 인자, 오른쪽 값, 조건이 들어갑니다.
QoraLowering은 이 하위 값을 모두 같은 형태로 뭉개지 않고, 후속 패스가 구조를 직접 읽어야 하는 값만 전용 노드로 만들며 나머지는 AST의 토큰 순서를 공백으로 정규화한 텍스트로 보존합니다.

호출 인자 전체가 q[i]처럼 하나의 인덱스 참조이면 레지스터 이름과 인덱스를 나눈 QQubitArg가 되지만, 인덱스가 없는 q는 이 단계에서 QTextArg로 남습니다.12
따라서 q가 단일 큐빗인지, 큐빗 배열 전체인지, 고전 값인지는 이후 선언 정보와 호출 대상의 파라미터를 함께 확인하여 판정합니다.12

선언이나 대입의 오른쪽 식 전체가 정확히 대문자 M(q[i]) 한 호출이면 QMeasure가 만들어집니다.3
반면에 호출이 산술식이나 다른 값과 섞이면 내용을 버리지 않고 QText와 HasCall=true로 보존합니다.5
이때 HasCall은 오류 자체가 아니라 원래 AST에 호출 노드가 있었다는 표식이며, 이후 패스가 호출이 놓인 위치에 맞는 처리를 선택할 때 사용합니다.5
일반 식에 남은 호출은 QoraValidator가 QSEM005로 보고합니다.5
조건에 포함된 M(q[i])는 먼저 QCond의 텍스트와 HasCall=true로 남고, 다음 전용 패스인 MeasureConditionLowering이 측정 문장과 측정 결과를 사용하는 조건으로 분리합니다.4 보충

6

노드를 만들 때 함께 기록한 Id와 소스 위치를 확인합니다

앞 단계에서 각 IR 노드를 생성할 때, 계속 추적해야 하는 주요 노드에는 고유한 Id도 함께 발급했습니다.
또한 진입 문서에서 오류가 발생했을 때 정확한 위치를 표시할 수 있도록 지원 노드에는 QSpan(Start, End)를 기록했습니다.

생성 노드고유 Id진입 문서에서 Span이 가리키는 범위
QProgram있음Span 필드 없음
QOperation있음Main, Prepare 같은 연산 이름 토큰
QParam있음파라미터 이름 토큰
모든 QStmt있음키워드·괄호·중괄호·세미콜론을 포함한 문장 전체
QImport / QOpen없음선언문 전체
QArg / QExpr / QCond없음자체 Span 없음
진입 파일 · main.qor 기본값 withSpans=true를 사용하므로 지원 노드에 실제 문자 범위를 기록합니다.
import 파일 · gates.qor withSpans=false로 다시 lowering하므로 모든 지원 Span이 null입니다.
다른 파일의 위치가 진입 문서 위치로 오해되는 일을 막기 위한 규칙입니다.

Id 번호는 프로세스 전체에서 계속 증가하므로 예제처럼 특정 숫자로 고정할 수 없지만, Id가 있는 각 노드는 서로 다른 값을 받습니다.
이로써 QoraLowering은 Janglim AST와의 연결을 끝내고, 구조·원본 값·추적 정보를 갖춘 단일 파일 QProgram을 다음 패스에 반환합니다.

입력 정보쉬운 의미이 패스에서 사용하는 곳

CLI 단계 출력과 원시 lowering 결과의 차이

CLI의 --stages에 표시되는 ir은 상황에 따라 이미 import 병합, 조건 속 측정 변환과 이름 해소를 거친 결과일 수 있습니다.
따라서 그 출력에서 Shared.Flip이나 import 파일의 연산이 보이더라도, 해당 값이 QoraLowering 직후부터 존재했다고 해석하면 안 됩니다.
위의 QoraLowering 출력 트리는 ModuleLoader와 Resolver가 실행되기 전의 원시 QProgram만 나타냅니다.

전달 항목QoraLowering이 보장하는 상태후속 단계의 작업
Operations현재 파일의 연산, 파라미터, 중첩 본문이 모두 Qora IR 타입으로 변환된 상태ModuleLoader가 import 파일의 연산을 병합합니다.
Imports따옴표만 제거한 원문 경로 문자열이며, 진입 문서는 보통 Span이 있고 import 문서는 Span이 없습니다.경로 기준이 있으면 ModuleLoader가 경로 정규화와 재귀 로드를 수행하고, loaded.Add(full) = false인 순환 back-edge와 중복 경로를 오류 없이 생략한 뒤 루트 Imports를 null로 소비합니다.
반면에 기준이 없으면 QSEM020을 기록하고 원래 Imports를 유지합니다.
Opensnamespace별 open 목록이며 대상의 존재 여부는 확인하지 않은 상태파일 병합 뒤 Resolver가 이름 해소에 사용합니다.
호출 이름소스에 적힌 단순 이름 또는 점으로 연결된 이름Resolver가 내장 게이트와 사용자 연산 후보를 구분합니다.
사용자 연산 참조는 완전 수식 이름으로 바꾸고 CalleeOpId를 연결하지만, 내장 게이트는 표준 짧은 이름을 유지합니다.
식과 조건필요한 큐빗·측정 구조와 나머지 텍스트, HasCall 표시import 오류가 없으면 MeasureConditionLowering이 조건 속 측정을 먼저 재작성하고, 이어서 Resolver가 이름을 해소합니다.
Resolver 오류도 없을 때 QoraValidator가 호출식의 표현 가능성과 타입을 검사합니다.
