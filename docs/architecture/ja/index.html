Qora コンパイルパイプライン設計

ソースが AST、Qora IR、検証パス、OpenQASM 出力まで進む基準フロー

Qora は、パーサエンジン（Janglim）の上に載る小さなフロントエンドと、その上にQora が完全に所有する マルチパスのミドルエンドから成ります。コンパイルは、小さく単一目的なパスの一直線です。各パスは プログラムをある形で受け取り、ちょうど 1 つの仕事だけをこなし、少しだけ仕上がったプログラムを次のパスへ渡します — 最後のパスが OpenQASM 3 を出力するまで。本ドキュメントはその一直線を最初から最後まで 1 ステージずつ辿り、 以下のすべての AST・IR・QASM のリスティングに実際のコンパイラ出力を用います。

0例 — すべてのパスに触れる 1 ファイル

すべてのステージが実際に働く様子を見るため、名前空間と量子ビット配列パラメータ （Qubit[]）、呼び出し、そして Adjoint を 1 つのプログラムに詰め込みます: 

namespace Prep {
    operation Fanout(Qubit[] q) {        // 配列長は呼び出しごとに決まる
        for i in 0..q.Count-1 { H(q[i]); }
    }
}

operation Main() {
    use reg = Qubit[3];
    Prep.Fanout(reg);                    // 名前空間をまたぐ呼び出し
    Adjoint Prep.Fanout(reg);            // 元に戻す — コンパイラが逆演算を合成
    bit r = M(reg[0]);
}

3 つの名前がパスを通り抜けて変わっていく様子を追ってください。Fanout は名前空間を得 （Prep.Fanout）、具体的なサイズを得（Prep.Fanout__sz3）、逆演算の双子を得 （Prep.Fanout__sz3__adj）、最後に OpenQASM 用に平坦化されます（Prep_Fanout__sz3）。 これらの変化のそれぞれが、下記の 1 パスずつです。

12 つの世界 — エンジンが止まり Qora が始まる場所

コンパイラ全体で境界はちょうど 1 つだけです。その左には Janglim パーサエンジンがあります — 文法・レクサ、そしてテキストを構文木に変える仕事を所有します。右には Qora があります — プログラムに意味を与え、検査し、変換し、出力するすべてのパスがここにあります。2 つの世界は 1 つの関数で出会います — ローワリング（lowering） — この関数はエンジンの木をちょうど一度だけ読み、二度と振り返りません。

ソース → AST  │ エンジン (Janglim)
Qora │  IR → ⋯ → OpenQASM
なぜ境界をこれほど鋭く保つのか? Qora が扱うべきすべて — 型、逆演算、ジェネリック、名前の安全性 — は、エンジンの木ではなく Qora 自身の IR 上で処理されます。だからエンジンは量子について何も知らない汎用パーサのまま でいられ、Qora は文法に触れずに機能（新しいパス）を追加できます。エンジンに依存する唯一の地点はローワリング であり、残りのコンパイラはパーサを丸ごと差し替えても生き延びます。

2Parse — ソースから AST へ（エンジンの仕事）

in: ソーステキスト  ·  out: 構文木  ·  担当: Janglim エンジン

エンジンは Qora の文法からレクサと LALR パーサを作り、ソースをパース木に変えます。この段階 には意味がありません — Adjoint は呼び出しの隣にあるトークンにすぎず、 Prep.Fanout は 3 つのトークン（Prep、.、Fanout）であり、 呼び出しとゲートは同じ見た目です。例の実際の木: 

Nonterminal : Program
  Nonterminal : Namespace
    Terminal : Prep
    Nonterminal : Operation
      Terminal : Fanout
      Nonterminal : Param
        Nonterminal : ArrayType
          Terminal : Qubit
        Terminal : q
      Nonterminal : For
        Terminal : i
        Nonterminal : Expr ( 0 )
        Nonterminal : Expr ( q . Count - 1 )
        Nonterminal : Gate
          Terminal : H
          Nonterminal : IndexAccess ( q , i )
  Nonterminal : Operation
    Terminal : Main
    Nonterminal : Use ( reg , Qubit , 3 )
    Nonterminal : Gate                  ← 呼び出しも他のすべてと同じく Gate としてタグ付け
      Terminal : Prep
      Terminal : .
      Terminal : Fanout
      Nonterminal : Expr ( reg )
    Nonterminal : Gate
      Terminal : Adjoint              ← ファンクタは先頭のトークンにすぎない
      Terminal : Prep
      Terminal : .
      Terminal : Fanout
      Nonterminal : Expr ( reg )
    Nonterminal : VarDecl
      Terminal : bit
      Terminal : r
      Nonterminal : Expr → Call ( M , IndexAccess ( reg , 0 ) )
この木は一度読まれた後、二度と触れられません 以降のすべてのステージはこの木ではなく IR 上で作動します。Prep.Fanout__sz3 や __adj のような名前はまだ存在しません — すべてこの下流で生まれます。

3Lower — AST から Qora IR へ（境界を越える）

in: エンジン AST  ·  out: QProgram (Qora IR)  ·  パス: QoraLowering

ローワリングはエンジンの木を一度走査し、Qora 自身の IR として組み直します — 強く型付けされた レコードの階層です（QOperation、QGate、QFor、QIf、 QDecl、QUse、QMeasure …）。最も重要な変化は 2 つ: 文字列トークンが 型のあるノードになり、ファンクタがノード上のデータになります （functors=[Adjoint]）。例の IR: 

QOperation Prep.Fanout(Qubit q)
  QFor(i in 0..n - 1)
    QGate(functors=[], name=H, args=[q[i]])

QOperation Main()
  QUse(name=reg, size=3)
  QGate(functors=[],        name=Prep.Fanout, args=[reg])
  QGate(functors=[Adjoint], name=Prep.Fanout, args=[reg])
  QDecl(const=False, type=Bit, name=r, value=QMeasure(reg[0]))
なぜファンクタをデータに変えるのか? 「この呼び出しは逆演算か?」が、いまや再パースすべきトークンではなく値（functors=[Adjoint]） なので、後続のパスはその値を反転するだけで逆演算を作れます — 文字列の手術は不要です。ミドルエンド全体が IR ベースであるのも同じ理由です: パスが構造化されたノードを読み書きするので、各パスは短く保たれ、間違えにくいのです。

4Expand & Resolve — インポートと呼び出し先 FQN

パス: ModuleLoader.Expand → Resolver.Resolve

Expand はモジュールローダです。ファイルに import "lib/prep.qor" があったなら、 このパスがそのファイルをパース・ローワリングし、その演算群を 1 つの QProgram に統合して、以降の すべてが単一の平坦なプログラムを見るようにします。（例は何もインポートしないのでここでは何もしません。 ファイルが見つからない場合は QSEM020 になります。一方、循環する back-edge は loaded に登録済みのパスとしてエラーなしでスキップされます。QSEM021 は 廃止済みの予約コードとしてのみ残り、現在は出力されません。）

Expand と Resolve のあいだで、小さな脱糖（desugaring）が 1 つ走ります — MeasureConditionLowering です。OpenQASM には条件の中での測定がないので、if (M(q[0]) == 1) は OpenQASM が必要とする 2 段階の形へ書き換えられます: 測定は巻き上げられた bit へ引き出され、条件はそれを検査します。 while は各反復の末尾で、repeat … until は本体の末尾で再測定します。以降のすべては、 こうして素朴なビット条件だけを見ることになります: 

operation Main() {
    use q = Qubit[2];
    if (M(q[0]) == 1) { X(q[1]); }
}
                       ↓ MeasureConditionLowering
qubit[2] q;
bit __m0;
__m0 = measure q[0];          ← 条件の外へ巻き上げられた測定
if (__m0 == true) { x q[1]; }

続いて Resolve が演算定義と演算呼び出し先を完全修飾名（FQN）に変えます。 namespace Prep 内の演算 Fanout は Prep.Fanout になり、呼び出し側 Prep.Fanout(reg) がそれを指しているか確認されます。だから上の IR リスティングがすでに Fanout ではなく Prep.Fanout と読めるのです。ただし、このパスがすべての識別子を 解決するわけではありません。変数、ループ変数、キュービットのインデックス、条件式内の名前は、 次の検証段階でシンボルテーブルがスコープに基づいて確認します。呼び出し名が曖昧だった場合（open した 2 つの名前空間から見えた場合）は QSEM018、未知の名前空間やメンバは QSEM019 に なります。

5Validate — 意味のゲート

in: 解決済み IR  ·  out: エラーの一覧（またはなし）  ·  パス: QoraValidator + SymbolTableBuilder

検証は 1 つの問いに答えるパスです: このプログラムは OpenQASM が表現できる意味か? IR 全体を 一度走査し、すべての違反を集めます（最初のエラーで止まらない）。その結果は固いゲートです: エラーが 1 つでもあれば OpenQASM はまったく生成されません — だから壊れたプログラムが静かに 「成功したふり」をして誤った回路を返すことは決してありません。この段階で検証器は統合シンボルテーブルも作ります。 パラメータ、レジスタ、測定ビット、変数、定数、ループ変数がどのスコープに属するかを記録し、 各名前がどこで使われたかも集めます。すべての規則には安定したコード （QSEM001〜QSEM025）があり、エディタが波線を付けられます。そのいくつか:

コード拒否するもの
QSEM001逆演算を持たない Foo への Adjoint Foo（測定・リセット・while ループなど） — 理由はインバータから連鎖
QSEM002ユーザ op への Controlled Foo（OpenQASM に def 用の ctrl @ はない）
QSEM006シグネチャに合わない引数の呼び出し — 個数、種類（量子ビット vs 角度）、またはレジスタサイズ
QSEM007未知のゲート/演算名（「もしかして …?」ヒント付き）
QSEM011再帰呼び出し（OpenQASM の def は再帰できない）
QSEM016レジスタ範囲を超えるリテラルの量子ビットインデックス
QSEM025スコープ内で解決できない識別子 — 未知の名前、または宣言前に使われた古典名
なぜ変換パスより先に検証するのか? 以降のすべてのパスが、プログラムはすでに健全だと仮定できるようになるからです。単相化器はスコープで解決できない名前に決して 出会わず、adjoint 実体化器は逆演算不可能な Adjoint に決して出会わず（QSEM001 がすでに拒否済み）、 エミッタは不正な呼び出しに決して出会いません。ゲートが全員の前に立っているからこそ、各変換パスは単純でいられます。

6Monomorphize — 量子ビット配列呼び出しの特殊化

in: 検証済み IR  ·  out: 具体サイズの op を持つ IR  ·  パス: Monomorphizer

表面型 Fanout(Qubit[] q) には配列長を書きません。単相化器（monomorphizer）は呼び出しごとに 実際のレジスタサイズを読み取り、具体的な内部コピーを打ち出し、そのコピーの q.Count を実際のサイズに置き換えてから、呼び出し先を書き換えます。例の呼び出しは reg（量子ビット 3 個）を渡すので:

Prep.Fanout(Qubit[])  +  サイズ 3 のレジスタでの呼び出し  ⟶  Prep.Fanout__sz3(Qubit[3])

サイズが具体的になったので、プログラムは二度目の検証を受けます: q.Count が数になって 初めて意味を持つ検査 — インデックス範囲、レジスタサイズの一致 — が、ようやく照合する対象を得ます。このパスの 後には、ジェネリックな演算は残りません。

7Materialize Adjoint — 逆演算を本物の op に

in: 特殊化された IR  ·  out: 合成された逆演算 op を持つ IR  ·  パス: AdjointMaterializer

直前に走る — within/apply を平坦化（ConjugationLowering）

Qora の IR には、表層構文を持たない合成ノードが 1 つあります: QConjugate — within/apply （共役; conjugation）形式で、「within を実行し、次に apply、最後に within の逆演算を 実行する」ことを意味します — compute · use · uncompute のサンドイッチです。反転されるのは within だけ なので、その効果（一時的な補助ビット（アンシラ）の準備）は取り消され、一方で apply の効果は残ります。 ConjugationLowering はそのノードを普通のゲートへ平坦化し、末尾の逆演算を合成するために同じ反転 カーネルを再利用します: 

QConjugate(within=[X(a[0])], apply=[CNOT(a[0], d[0])])
                       ↓ ConjugationLowering  ( within ++ apply ++ inverse-of-within )
x a[0];                ← within (compute)
cx a[0], d[0];         ← apply   (use)
inv @ x a[0];          ← 合成された within の逆演算 (uncompute)

このパスがここ、実体化器の直前で走る理由は順序です: 合成された逆演算がユーザ演算 （Adjoint Foo）を呼ぶ場合、それはやはり本物の Foo__adj op にならなければならず — それは実体化器がこのパスの後に走るからこそ起こります。今日、ソースから QConjugate を 生み出すものは何もないので、普通のプログラムではこのパスは何もしません（no-op）; これは、来たる 自動アンコンピュテーションが供給する出力経路です（末尾のノートを参照）。

例の Main には Adjoint Prep.Fanout(reg) があります。このパスはその演算全体の adjoint を 普通の演算に変えます。反転カーネル（Inverter）を使って逆演算の本体を計算し — 文を逆順にし、各ゲートに dagger を取り — それを Prep.Fanout__sz3__adj という名の 本物の op としてプログラムに追加し、呼び出しを普通の Prep.Fanout__sz3__adj(reg) に書き換えます。 H ゲートのファンアウトの逆演算は、各 H に dagger を取った逆向きのループです:

(U₁ · U₂ · ⋯ · Uₙ)†  =  Uₙ† · ⋯ · U₂† · U₁†

この要求は推移的です: 逆演算の本体が Adjoint Bar を呼ぶなら Bar の 逆演算も実体化され、その集合が閉じるまで続きます。組み込みゲートへの adjoint（Adjoint H）は そのままにされます — それは出力時に inv @ h になります。ここで実体化されるのはユーザ演算だけです。

なぜ名前付けの前に実体化するのか（出力時ではなく）? 合成された逆演算 def は、最終 QASM が他のすべてのグローバル名（たとえば Prep.Fanout__sz3__adj というユーザレジスタ）と衝突しないよう守るべき名前です。名前マングラが走る前に — 本物の op として — ここで作れば、その逆演算は他のすべての演算と同じ衝突解決を通るので、生成された名前が名前付けの 後に生まれることは決してありません。単相化器が使うのと同じ手口です: 生成された op はマングリングの 前に実体化し、エミッタは名前をひねり出す必要が一切ありません。

8Mangle & Check — 衝突しない名前、そして検証

パス: NameMangler → ReferentialCheck

Mangle はすべての Qora 名を、妥当で衝突しない OpenQASM 識別子に対応づけます。2 つのことが 起きます: 名前空間付きの名前はドットを平坦化し（Prep.Fanout__sz3 → Prep_Fanout__sz3）、衝突しうる名前 — OpenQASM のキーワード、stdgates のゲート名、または既に 出力済みの他の名前と — は一意になるまで _ を付け足します。したがって衝突は決して コンパイルエラーではありません: 自動的に解決され、名前が調整されたことを読者が分かるよう // Qora: ノートが QASM ヘッダに書き込まれます。

名前は OpenQASM をそのまま映したスコープに住む 演算名とエントリ op の最上位宣言は 1 つのグローバルスコープを共有し（stdgates のように）、各 def は パラメータ・変数のための自分のローカルスコープを持ちます。だから（ゲート s とぶつかる） s という変数は s_ として出力されますが、同じ名前が別の def の中では自由に再び 現れられます。

Check（参照整合性のゲート）は、マングリングの直後に走る安全網です: プログラムが使う すべての識別子が、宣言された何かに解決されることを確認します — パラメータ・レジスタ・変数・演算・ 組み込み。ここでの宙ぶらりんな参照は、あるパスが宣言は改名したのにその使用箇所の 1 つを改名しなかったという 意味 — コンパイラのバグ — なので、静かに壊れた QASM として出力する代わりに QINTERNAL として声高に報告されます。正しいコンパイラでは決して発火しません; その種のバグを出荷不可能にするために存在します。

9Emit — OpenQASM 3 へ出す単純なプリンタ

in: マングル済み IR  ·  out: OpenQASM 3 テキスト  ·  パス: OpenQasmLowering → QasmEmitter

直前に走る — OpenQASM 向けターゲットローワリング（OpenQasmLowering）

プリンタの前に、最後の IR→IR ステップが 1 つ走ります — OpenQasmLowering、OpenQASM 固有の ターゲットローワリングです。Qora の const は任意の値を保持しうる不変の束縛（Q# の let 流）ですが、OpenQASM の const はコンパイル時定数の初期化子を要求します。そのため 実行時に束縛される const c = n; は、出力のために普通の int c = n; へ格下げされます — それでも再代入されることはないので、その不変性は何も失われません: 

int n = 2;
const c = n;          ← Qora の const: 不変の束縛、任意の値
                   ↓ OpenQasmLowering
int n = 2;
int c = n;            ← 普通の int（OpenQASM の const はコンパイル時定数でなければならない）

これは最初のターゲット別ローワリングパスです — Qora にとって正しい IR が、特定のバックエンドの 規則に合わせて調整される場所です。将来の QIR ターゲットは、上位のどのパスにも触れることなく、ここに独自の ローワリングを追加することになります。

最後のパスは意図的に最も賢くありません: 名前をひねり出しもせず、何も反転しません — あらゆる 変換はすでに上流で起きています。ただ各演算を def として、エントリ op を最上位の本体として出力する だけです。唯一の実質的な雑務は順序付けです: OpenQASM に前方宣言はないので、def 群は呼び出し 依存（トポロジカル）順で出力されます — だから Prep_Fanout__sz3__adj はそれが必要とするものより 先に来ず、すべての def は使われる前に宣言されます。例の完全な出力: 

// generated by Qora v0.18
OPENQASM 3;
include "stdgates.inc";

def Prep_Fanout__sz3(qubit[3] q) {      ← 名前空間 + 特殊化 + 平坦化
  for int i in [0:3 - 1] {
    h q[i];
  }
}

def Prep_Fanout__sz3__adj(qubit[3] q) {  ← 合成された逆演算（ソースになかったもの）
  for int i in [3 - 1:-1:0] {            ← 逆向きのループ
    inv @ h q[i];                         ← dagger されたゲート
  }
}

qubit[3] reg;
bit r;
Prep_Fanout__sz3(reg);
Prep_Fanout__sz3__adj(reg);              ← Adjoint Prep.Fanout(reg) がこの呼び出しになった
r = measure reg[0];
なぜエミッタをこれほど単純に保つのか? 出力するだけのプリンタは信頼しやすいからです。反転・特殊化・名前付けがすべて各自のパスに住めば、最後の テキスト生成ステップには下すべき決定が 1 つも残りません — だからそれが書く OpenQASM は、渡された IR そのまま、 1 文字まで同じです。

この形は意図されたものです: エンジンとの境界 1 つ、続いて各自が IR に 1 つのことだけ をする小さなパスの一直線、そしてその前に立つ検証ゲートのおかげで、それ以降のすべてが単純でいられます。 言語機能を追加するのは、たいてい 1 つのパスを追加または拡張することであり — コンパイラを書き直すことではなく — 末尾の出力専用エミッタは、あなたが得る OpenQASM がパスの作った IR そのままであることを意味します。

この先が向かう場所 — 自動アンコンピュテーション Silq 流の自動アンコンピュテーション — 一時的な量子ビットがもう不要になる地点を見つけ、その逆演算を 自動的に注入する — こそ、これらのパスが目指して作られているものです。最初のピースはすでに揃っています: ConjugationLowering が within/apply ノードを compute · apply · uncompute へ平坦化して 出力するので、下流の経路全体（逆演算の合成 → __adj の閉包 → マングリング → QASM）が端から端まで 実証されています。残るのは、そのノードを自ら生み出す解析です — ライブネス（一時変数がいつ死ぬか）と 効果解析（安全に反転できるか） — その後、小さなインジェクタが各死亡地点で QConjugate を出力します。

本ドキュメントのすべての AST・IR・OpenQASM リスティングは、上記の例を実際の Qora コンパイラでコンパイルして 得たものです（qora --json --stages）。
