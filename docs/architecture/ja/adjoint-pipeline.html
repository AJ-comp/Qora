Adjoint 逆合成パイプライン

Adjoint 呼び出しが逆 IR、合成 def、OpenQASM 呼び出しへ変わる動作フロー

Qora で Adjoint Foo(q) と書くと、コンパイラは Foo の逆演算を行う サブルーチン(Foo__adj)を自動的に合成し、その呼び出しに置き換えます。 この文書では、それが正確にどこで、どの順序で起こるのかを、 実際のコンパイラ出力で一段階ずつたどっていきます。 以下に登場する AST・IR・OpenQASM は、すべてこの例を実際にコンパイルして得られた結果です。

1サンプルプログラム — 5 つの構造をすべて含むコード

逆演算の合成がサポートする 5 つの静的構造 — ゲート、古典宣言、for ループ、if 分岐、ネスト呼び出し — を 1 つのプログラムにすべて詰め込みました。 

operation Inner(q: Qubit[]) {           // ネスト呼び出しの対象
    H(q[0]);
    T(q[1]);
}

operation Outer(q: Qubit[], b: bit) {
    var k: int = 2;                    // ① 古典宣言
    Inner(q);                          // ② ネスト呼び出し
    Rx(pi/k, q[0]);                    // ③ ゲート(宣言値を角度として使用)
    for i in 0..q.Count-1 {            // ④ 受け取った配列を回る for ループ
        X(q[i]);
    }
    if (b == 1) {                      // ⑤ if 分岐
        Z(q[0]);
        Adjoint S(q[0]);               //    ファンクタが既に付いたゲート
    }
}

operation Main() {
    use q = Qubit[2];
    var r: bit = M(q[0]);
    Outer(q, r);
    Adjoint Outer(q, r);               // ★ この 1 行が逆合成の全体を引き起こす
}

2ステップ 1. AST — パーサーが作る構文木

パーサー(Janglim エンジン)はソースを構文木に変換します。この段階には「逆」の痕跡がまったくありません。 Adjoint Outer(q, r) も、単なる「Adjoint トークンが付いた呼び出し文」の 1 つにすぎません。 Outer の部分木だけを見ると: 

Operation
├─ "Outer"
├─ Param(ArrayType(Qubit), q)   Param(bit, b)
├─ VarDecl(int, k, Expr: 2)              ← int k = 2;
├─ Gate("Inner", Expr: q)                ← 呼び出しもゲートも同じ Gate タグ
├─ Gate("Rx", Expr: pi / k, IndexAccess(q,0))
├─ For(i, 0, q.Count - 1)
│   └─ Gate("X", IndexAccess(q,i))
└─ If
    ├─ Condition(b == 1)
    ├─ Gate("Z", IndexAccess(q,0))
    └─ Gate("Adjoint", "S", IndexAccess(q,0))  ← ファンクタは子トークン
この木は以後、決して修正されません 後続の段階はこの木を読むだけです。Outer__adj のようなものは この時点では存在すらせず、ステップ 4 で新しく作られます。

3ステップ 2. QoraIR — コンパイラが所有する中間表現

Lowering(QoraLowering)が、エンジンの AST を Qora 自身の IR(QProgram)へ一度変換します。 ここからは文字列タグの代わりに強い型付けのノード(QGate、QFor、QIf、…)として扱います。 Outer の IR: 

QOperation Outer(Qubit[] q, Bit b)
  QDecl(const=False, type=Int, name=k, value=2)
  QGate(functors=[],        name=Inner, args=[q])
  QGate(functors=[],        name=Rx,    args=[pi / k, q[0]])
  QFor(i in 0..q.Count-1)
    QGate(functors=[],      name=X,     args=[q[i]])
  QIf(cond="b == 1")
    then:
      QGate(functors=[],        name=Z, args=[q[0]])
      QGate(functors=[Adjoint], name=S, args=[q[0]])

注目すべき点:ファンクタが functors=[Adjoint] のようにノードのデータになりました。 「このゲートは逆か」がテキストではなく値として表現されるため、次の段階ではこの値を反転するだけで逆を作れます。

4ステップ 3. AdjointMaterializer — Inverter カーネルで逆演算 op を合成する

Main に Adjoint Outer(...) があるので、コンパイラには Outer の逆が必要です。 実際の最上位パスは AdjointMaterializer です。このパスが必要な逆演算を追跡し、 Inverter カーネルで逆向きの本体を作ってから、Outer__adj のような本物の op としてプログラムに追加します。 数学的な根拠はただ 1 つです:

(U₁ · U₂ · ⋯ · Uₙ)†  =  Uₙ† · ⋯ · U₂† · U₁†
— 順序を反転し、それぞれを逆(dagger)に

Inverter カーネルが実際に返した Outer の逆 IR:

QDecl(const=False, type=Int, name=k, value=2)     ← ① 宣言:順序を維持、先頭に
QIf(cond="b == 1")                                ← ⑤ if:条件を維持、分岐の内部だけ逆転
  then:
    QGate(functors=[],        name=S, args=[q[0]])   ← Adjoint S → S(ファンクタが外れる)
    QGate(functors=[Adjoint], name=Z, args=[q[0]])   ← Z → Adjoint Z
QFor(i in 1..0, step=-1)                          ← ④ for:範囲をスワップ + 逆方向
  QGate(functors=[Adjoint], name=X, args=[q[i]])
QGate(functors=[Adjoint], name=Rx, args=[pi / k, q[0]])  ← ③ 逆回転
QGate(functors=[Adjoint], name=Inner, args=[q])          ← ② Inner → Adjoint Inner(推移的)

構造別の逆転規則

構造順方向逆方向規則
ゲートXAdjoint XAdjoint ファンクタをトグル
既に Adjoint のゲートAdjoint SS逆の逆 = 元どおり
古典宣言int k = 2;int k = 2;(先頭)単一代入なので位置は無関係 — 反転しない
for ループfor i in 0..1for i in 1..0, step −1逆方向の走査 + 本体の逆転
if 分岐if (b == 1) { … }条件はそのまま、本体だけ逆転条件が読む値は本体内で変わらないため安全
ネスト呼び出しInner(q)Adjoint Inner(q)対象 op の逆も推移的に要求される
文の順序①②③④⑤① の次に ⑤④③②量子文だけ逆順
なぜ宣言は反転しないのでしょうか? int k = 2; のような古典計算には「逆」というものがありません。さらに Qora は逆合成の対象内での 変数の再代入を拒否するため、宣言された値は最後まで同じ値のままです。そのため宣言をそのまま先頭に置いておけば、 逆方向のゲート(inv @ rx(pi / k))は順方向とまったく同じ角度を読み取ります。
推移的な合成 — ソースにない逆も自動で Outer の逆が Adjoint Inner を呼ぶことになったので、Inner の逆も必要になりました。 ソースのどこにも Adjoint Inner と書いたことはありませんが、コンパイラが必要性を追跡して Inner__adj まで作ります。この要求は呼び出しの連鎖に沿って続いていきます(推移的閉包)。

5ステップ 4. OpenQASM — 逆 IR をそのまま出力する

最後に Emitter(QasmEmitter)が IR を OpenQASM 3 のテキストとして出力します。 逆 IR は特別なところのない普通の IR なので、通常の文とまったく同じ経路で出力されます。 全体の出力:

(出力の直前に 2 つのパスが走ります。まず adjoint 実体化が Adjoint Outer を 本物の逆方向 def op に変えます — 以下の Inner__adj・Outer__adj はここから生まれ、 だからそれらが普通の def なのです。次に ネームマングラーが出力名を OpenQASM に 安全な形へ合わせますが、キーワード・stdgates のゲート・他の名前と実際に衝突するときだけ名前を 変えます — _ を付け、// Qora: ノートを残します。ここの名前はどれも衝突しないので そのまま通ります。エラーとコンパイル段階ビューは常に元の名前を保ちます。) 

// generated by Qora v0.18
OPENQASM 3;
include "stdgates.inc";

def Inner(qubit[2] q) {
  h q[0];
  t q[1];
}

def Outer(qubit[2] q, bit b) {
  int k = 2;
  Inner(q);
  rx(pi / k) q[0];
  for int i in [0:1] {
    x q[i];
  }
  if (b == true) {
    z q[0];
    inv @ s q[0];
  }
}

def Inner__adj(qubit[2] q) {             ← ソースになかった def — 推移的に合成された
  inv @ t q[1];
  inv @ h q[0];
}

def Outer__adj(qubit[2] q, bit b) {      ← Adjoint Outer が呼ぶ逆方向の def
  int k = 2;                             ← ① 宣言を維持
  if (b == true) {                          ← ⑤ 条件を維持、内部だけ逆
    s q[0];
    inv @ z q[0];
  }
  for int i in [1:-1:0] {                ← ④ 逆方向のループ(1, 0 の順)
    inv @ x q[i];
  }
  inv @ rx(pi / k) q[0];                 ← ③ 逆回転
  Inner__adj(q);                         ← ② 合成された逆 def の呼び出し
}

qubit[2] q;
bit r;
r = measure q[0];
Outer(q, r);
Outer__adj(q, r);                        ← Adjoint Outer(q, r) がこれに置換される
def の順序にも規則があります OpenQASM では宣言した後にしか使用できません(宣言してから使用)。そのため Emitter は、 すべての def(順方向 + 合成)を呼び出しの依存順(トポロジカルソート)で配置します — Inner__adj が、それを呼ぶ Outer__adj より必ず先に現れる理由です。

6何が逆転され、何が拒否されるのか

逆を作れない op に Adjoint を適用するとコンパイルエラーになります。 変換(Emit)の前に検証パスが IR 全体を一巡して意味エラーをすべて収集し、 1 つでもあれば OpenQASM をまったく生成しません — 誤ったプログラムが「成功したふり」をして 間違った回路を黙って出力することがないようにするためです。拒否の理由は 2 種類あります。

構造判定理由
ゲート · for · if · 宣言 · ネスト呼び出し✅ 逆転をサポートこの文書の 4 段階の規則
測定(M)· Reset❌ 恒久的に拒否物理的に非可逆 — 測定は状態を収縮させ、情報を捨てる
変数の再代入(k = k + 1;)❌ 現時点では拒否古典状態の時間的な流れの追跡が必要(効果分析の段階で)
while · repeat❌ 現時点では拒否繰り返し回数をコンパイル時に知ることができない
use(ローカルキュービット)❌ 現時点では拒否補助キュービットの後始末 = 自動 uncomputation の役割
再帰呼び出し❌ 拒否逆も再帰になり、終了を保証できない
エラーの理由は呼び出しの連鎖に沿って伝播します

Caller が測定を行う Meas を呼ぶと、Adjoint Caller(q) は次のようなコンパイルエラーになります:

[QSEM001] in `Main`: `Adjoint Caller` cannot be compiled: it calls `Meas`, which cannot be inverted (it measures a qubit, and measurement is irreversible)

括弧のチェーンをたどれば、根本原因(Meas の測定)まで読み取れます。 エラーは最初の発見で止まらず、1 回のコンパイルですべて報告されます。

Adjoint Foo(q) の 1 行がすること:コンパイラが ① Foo の逆が必要であることを追跡し、 ② IR 上で「逆順 + ゲートごとの逆」として逆 IR を合成し(必要なら Foo が呼ぶ op たちの逆まで推移的に)、 ③ Foo__adj という def として出力した後、④ 呼び出し箇所を Foo__adj(q) に置き換えます。 ユーザーは逆方向のコードを 1 行も書いていませんが、最終的な OpenQASM には完全な逆演算が入っています — C# の async がステートマシンを代わりに作ってくれるのと同じ方式のコンパイル時コード合成です。

この構造が次に開くもの — 自動 uncomputation 一時的な(補助)キュービットは、使い終えたら逆演算で元に戻しておかないと干渉が汚染されてしまいます。 今はユーザーが Adjoint と明示したときだけ逆を合成しますが、この Inverter カーネルをそのまま再利用すれば、 コンパイラが一時キュービットがもう使われなくなる地点を見つけて、逆演算を自動的に注入できます。 それが Qora の目指す Silq 式の自動 uncomputation です。
