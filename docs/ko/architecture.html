Qora 컴파일 파이프라인 설계

소스가 AST, Qora IR, 검증 패스, OpenQASM 방출까지 지나가는 기준 흐름

Qora는 파서 엔진(Janglim) 위에 얹힌 얇은 프런트엔드와, 그 위에 Qora가 전적으로 소유하는 멀티패스 미들엔드로 이루어져 있습니다. 컴파일은 작고 단일 목적을 가진 패스들의 일직선입니다. 각 패스는 한 형태의 프로그램을 받아 딱 한 가지 일만 하고, 조금 더 완성된 프로그램을 다음 패스에 넘깁니다 — 마지막 패스가 OpenQASM 3을 출력할 때까지요. 이 문서는 그 일직선을 처음부터 끝까지 한 단계씩 따라가며, 아래의 모든 AST·IR·QASM 리스팅에 실제 컴파일러 출력을 사용합니다.

0예제 — 모든 패스를 건드리는 파일 하나

모든 단계가 실제로 일하는 모습을 보기 위해, 네임스페이스와 제네릭 레지스터(Qubit[n]), 호출, 그리고 Adjoint를 한 프로그램에 담았습니다: 

namespace Prep {
    operation Fanout(Qubit[n] q) {       // 제네릭: n은 나중에, 호출부마다 정해짐
        for i in 0..n-1 { H(q[i]); }
    }
}

operation Main() {
    use reg = Qubit[3];
    Prep.Fanout(reg);                    // 네임스페이스를 넘나드는 호출
    Adjoint Prep.Fanout(reg);            // 되돌리기 — 컴파일러가 역연산을 합성
    bit r = M(reg[0]);
}

이름 세 개가 패스들을 지나며 어떻게 변하는지 보세요. Fanout은 네임스페이스를 얻고 (Prep.Fanout), 구체적 크기를 얻고(Prep.Fanout__sz3), 역연산 쌍둥이를 얻고 (Prep.Fanout__sz3__adj), 마지막으로 OpenQASM용으로 평탄화됩니다(Prep_Fanout__sz3). 이 각 변화가 바로 아래의 한 패스씩입니다.

1두 세계 — 엔진이 멈추고 Qora가 시작되는 곳

컴파일러 전체에 경계는 정확히 하나뿐입니다. 왼쪽에는 Janglim 파서 엔진이 있습니다 — 문법·렉서, 그리고 텍스트를 구문 트리로 바꾸는 일을 소유합니다. 오른쪽에는 Qora가 있습니다 — 프로그램에 의미를 부여하고, 검사하고, 변환하고, 출력하는 모든 패스가 여기 있습니다. 두 세계는 함수 하나에서 만납니다 — 로워링(lowering) — 이 함수는 엔진의 트리를 딱 한 번 읽고 다시는 돌아보지 않습니다.

소스 → AST  │ 엔진 (Janglim)
Qora │  IR → ⋯ → OpenQASM
왜 경계를 이렇게 뚜렷이 유지하나? Qora가 다뤄야 할 모든 것 — 타입, 역연산, 제네릭, 이름 안전성 — 은 엔진의 트리가 아니라 Qora 자신의 IR 위에서 처리됩니다. 그래서 엔진은 양자에 대해 아무것도 모르는 범용 파서로 남을 수 있고, Qora는 문법을 건드리지 않고도 기능(새 패스)을 추가할 수 있습니다. 엔진에 의존하는 유일한 지점은 로워링이며, 나머지 컴파일러는 파서를 통째로 교체해도 살아남습니다.

2Parse — 소스에서 AST로 (엔진의 몫)

in: 소스 텍스트  ·  out: 구문 트리  ·  담당: Janglim 엔진

엔진은 Qora 문법으로부터 렉서와 LALR 파서를 만든 뒤, 소스를 파스 트리로 바꿉니다. 이 단계에는 아무 의미가 없습니다 — Adjoint는 호출 옆에 붙은 토큰일 뿐이고, Prep.Fanout은 토큰 세 개(Prep, ., Fanout)이며, 호출과 게이트는 똑같이 생겼습니다. 예제의 실제 트리: 

Nonterminal : Program
  Nonterminal : Namespace
    Terminal : Prep
    Nonterminal : Operation
      Terminal : Fanout
      Nonterminal : Param
        Terminal : n                    ← 제네릭 크기 심볼
        Terminal : q
      Nonterminal : For
        Terminal : i
        Nonterminal : Expr ( 0 )
        Nonterminal : Expr ( n - 1 )
        Nonterminal : Gate
          Terminal : H
          Nonterminal : Qubit ( q , i )
  Nonterminal : Operation
    Terminal : Main
    Nonterminal : Use ( reg , 3 )
    Nonterminal : Gate                  ← 호출도 다른 모든 것처럼 Gate로 태깅됨
      Terminal : Prep
      Terminal : .
      Terminal : Fanout
      Nonterminal : Expr ( reg )
    Nonterminal : Gate
      Terminal : Adjoint              ← 함자는 그저 앞에 붙은 토큰
      Terminal : Prep
      Terminal : .
      Terminal : Fanout
      Nonterminal : Expr ( reg )
    Nonterminal : VarDecl
      Terminal : bit
      Terminal : r
      Nonterminal : Expr → Call ( M , Qubit ( reg , 0 ) )
이 트리는 한 번 읽힌 뒤 다시는 손대지 않습니다 이후 모든 단계는 이 트리가 아니라 IR 위에서 작동합니다. Prep.Fanout__sz3나 __adj 같은 이름은 아직 존재하지 않습니다 — 전부 이 뒤에서 태어납니다.

3Lower — AST에서 Qora IR로 (경계를 넘다)

in: 엔진 AST  ·  out: QProgram (Qora IR)  ·  패스: QoraLowering

로워링은 엔진의 트리를 한 번 순회하며 Qora 자신의 IR로 다시 세웁니다 — 강타입 레코드의 계층입니다(QOperation, QGate, QFor, QIf, QDecl, QUse, QMeasure, …). 가장 중요한 변화 둘: 문자열 토큰이 타입이 있는 노드가 되고, 함자가 노드 위의 데이터가 됩니다 (functors=[Adjoint]). 예제의 IR: 

QOperation Prep.Fanout(Qubit q)
  QFor(i in 0..n - 1)
    QGate(functors=[], name=H, args=[q[i]])

QOperation Main()
  QUse(name=reg, size=3)
  QGate(functors=[],        name=Prep.Fanout, args=[reg])
  QGate(functors=[Adjoint], name=Prep.Fanout, args=[reg])
  QDecl(const=False, type=Bit, name=r, value=QMeasure(reg[0]))
왜 함자를 데이터로 바꾸나? "이 호출이 역연산인가?"가 이제 다시 파싱할 토큰이 아니라 값(functors=[Adjoint])이므로, 이후 패스는 그 값을 뒤집기만 하면 역연산을 만들 수 있습니다 — 문자열 수술이 필요 없습니다. 미들엔드 전체가 IR 기반인 것도 같은 이유입니다: 패스들이 구조화된 노드를 읽고 쓰므로, 각 패스는 짧게 유지되고 틀리기 어렵습니다.

4Expand & Resolve — 임포트와 호출 대상 FQN

패스: ModuleLoader.Expand → Resolver.Resolve

Expand는 모듈 로더입니다. 파일에 import "lib/prep.qor"가 있었다면, 이 패스가 그 파일을 파싱·로워링하고 그 연산들을 하나의 QProgram으로 병합하여, 이후 모든 단계가 하나의 평탄한 프로그램을 보게 합니다. (예제는 아무것도 임포트하지 않으므로 여기서는 아무 일도 하지 않지만 — 파일 없음이 QSEM020, 임포트 순환이 QSEM021이 되는 곳입니다.)

이 둘 사이에서 작은 디슈가링(desugaring) 하나가 돕니다 — MeasureConditionLowering. OpenQASM에는 조건 안의 측정이 없으므로, if (M(q[0]) == 1)은 OpenQASM이 필요로 하는 두 단계 형태로 다시 쓰입니다: 측정이 끌어올려진 bit으로 빠져나오고, 그 뒤 조건이 그 비트를 검사합니다. while은 매 반복 끝마다 다시 측정하고, repeat … until은 몸통 끝에서 다시 측정합니다. 그러면 이후 모든 단계는 평범한 비트 조건만 보게 됩니다: 

operation Main() {
    use q = Qubit[2];
    if (M(q[0]) == 1) { X(q[1]); }
}
                       ↓ MeasureConditionLowering
qubit[2] q;
bit __m0;
__m0 = measure q[0];          ← 조건에서 끌어올려진 측정
if (__m0 == true) { x q[1]; }

이어 Resolve가 연산 정의와 연산 호출 대상을 정규화된 이름(FQN)으로 바꿉니다. namespace Prep 안의 연산 Fanout은 Prep.Fanout이 되고, 호출부 Prep.Fanout(reg)가 그것을 가리키는지 확인됩니다. 그래서 위 IR 리스팅이 이미 Fanout이 아니라 Prep.Fanout으로 읽히는 것입니다. 다만 여기서 모든 식별자를 처리하는 것은 아닙니다. 변수, 루프 변수, 큐비트 인덱스, 조건식 안의 이름은 다음 검증 단계의 심볼 테이블이 스코프 기준으로 확인합니다. 어떤 호출 이름이 모호했다면(open된 두 네임스페이스에서 보였다면) QSEM018이 되고, 알 수 없는 네임스페이스나 멤버는 QSEM019가 됩니다.

5Validate — 의미 게이트

in: 해소된 IR  ·  out: 오류 목록 (또는 없음)  ·  패스: QoraValidator + SymbolTableBuilder

검증은 한 가지 질문에 답하는 패스입니다: 이 프로그램은 OpenQASM이 표현할 수 있는 의미인가? IR 전체를 한 번 순회하며 모든 위반을 수집합니다(첫 오류에서 멈추지 않음). 그 결과는 단단한 게이트입니다: 오류가 하나라도 있으면 OpenQASM은 전혀 생성되지 않습니다 — 그래서 깨진 프로그램이 조용히 "성공한 척"하며 틀린 회로를 돌려주는 일은 결코 없습니다. 이때 검증기는 통합 심볼 테이블도 만들어요. 파라미터, 레지스터, 측정 비트, 변수, 상수, 루프 변수가 어느 스코프에 사는지 기록하고, 각 이름이 어디에서 사용되는지도 함께 모읍니다. 모든 규칙에는 안정된 코드 (QSEM001–QSEM025)가 있어 에디터가 물결선을 붙일 수 있습니다. 그중 몇 가지:

코드거부하는 것
QSEM001역연산이 없는 Foo에 대한 Adjoint Foo(측정·리셋·while 루프 등) — 사유는 인버터에서 연쇄됨
QSEM002유저 op에 대한 Controlled Foo(OpenQASM에는 def용 ctrl @가 없음)
QSEM006시그니처와 맞지 않는 인자의 호출 — 개수, 종류(큐비트 vs 각도), 또는 레지스터 크기
QSEM007알 수 없는 게이트/연산 이름("혹시 …?" 힌트 포함)
QSEM011재귀 호출(OpenQASM def는 재귀 불가)
QSEM016레지스터 범위를 벗어난 리터럴 큐비트 인덱스
QSEM025스코프 안에서 해소되지 않는 식별자 — 알 수 없는 이름 또는 선언 전에 사용된 고전 이름
왜 변환 패스들보다 먼저 검증하나? 이후 모든 패스가 프로그램이 이미 온전하다고 가정할 수 있게 하기 때문입니다. 단형화기는 스코프에서 해소되지 않는 이름을 결코 만나지 않고, adjoint 실체화기는 역연산 불가능한 Adjoint를 결코 만나지 않으며(QSEM001이 이미 거부함), 이미터는 잘못된 호출을 결코 만나지 않습니다. 게이트가 모두의 앞에 서 있기에, 각 변환 패스는 단순해질 수 있습니다.

6Monomorphize — 제네릭 레지스터의 특수화

in: 검증된 IR  ·  out: 구체적 크기의 op을 가진 IR  ·  패스: Monomorphizer

Fanout(Qubit[n] q)는 제네릭입니다: n은 호출이 정해주기 전까지 미지수입니다. 단형화기(monomorphizer)는 C++ 템플릿이나 Rust 제네릭처럼 작동합니다 — 각 호출부마다 n을 실제 레지스터 크기로 치환한 구체적 복사본을 찍어내고, 그 복사본을 가리키도록 호출을 다시 씁니다. 예제의 호출은 reg(큐비트 3개)를 넘기므로:

Prep.Fanout(Qubit[n])  +  크기 3 레지스터로의 호출  ⟶  Prep.Fanout__sz3(Qubit[3])

이제 크기가 구체적이므로, 프로그램은 두 번째로 검증됩니다: n이 숫자여야만 의미가 있는 검사 — 인덱스 범위, 레지스터 크기 일치 — 가 비로소 대조할 대상을 갖습니다. 이 패스 이후에는 제네릭 연산이 남지 않습니다.

7Materialize Adjoint — 역연산을 진짜 op으로

in: 특수화된 IR  ·  out: 합성된 역연산 op을 가진 IR  ·  패스: AdjointMaterializer

그 직전에 도는 패스 — within/apply 평탄화(ConjugationLowering)

Qora IR에는 표면 문법이 없는 합성 노드가 하나 있습니다: QConjugate, 즉 within/apply (켤레, conjugation) 형태로 "within을 실행하고, 그다음 apply, 그다음 within의 역연산"을 뜻합니다 — 계산 · 사용 · 언컴퓨트 샌드위치죠. within만 뒤집히므로, 그 효과(임시 보조 큐비트 설정)는 되돌려지고 apply의 효과는 살아남습니다. ConjugationLowering은 이 노드를 평범한 게이트들로 평탄화하며, 뒤따르는 역연산을 합성하기 위해 같은 인버전 커널을 재사용합니다: 

QConjugate(within=[X(a[0])], apply=[CNOT(a[0], d[0])])
                       ↓ ConjugationLowering  ( within ++ apply ++ inverse-of-within )
x a[0];                ← within (계산)
cx a[0], d[0];         ← apply   (사용)
inv @ x a[0];          ← 합성된 within의 역연산 (언컴퓨트)

이 패스가 여기서, 실체화기 바로 앞에서 도는 이유는 순서 때문입니다: 합성된 역연산이 유저 연산 (Adjoint Foo)을 호출하면, 그것도 여전히 진짜 Foo__adj op이 되어야 하는데 — 이는 실체화기가 이 패스 뒤에 돌기 때문에만 일어납니다. 아직은 소스에서 QConjugate를 만들어내는 것이 없으므로, 평범한 프로그램에서 이 패스는 아무 일도 하지 않습니다; 이것은 다가오는 자동 언컴퓨테이션이 공급할 방출 경로입니다(맺음말 참고).

예제의 Main에는 Adjoint Prep.Fanout(reg)가 있습니다. 이 패스는 그 연산 전체의 adjoint를 평범한 연산으로 바꿉니다. 인버전 커널(Inverter)을 써서 역연산 몸통을 계산하고 — 문장을 역순으로 뒤집고, 각 게이트에 dagger를 취해 — 그것을 Prep.Fanout__sz3__adj라는 이름의 진짜 op으로 프로그램에 추가한 뒤, 호출을 평범한 Prep.Fanout__sz3__adj(reg)로 다시 씁니다. H 게이트 팬아웃의 역연산은 각 H에 dagger를 취한 역방향 루프입니다:

(U₁ · U₂ · ⋯ · Uₙ)†  =  Uₙ† · ⋯ · U₂† · U₁†

이 수요는 전이적입니다: 역연산 몸통이 Adjoint Bar를 호출하면 Bar의 역연산도 실체화되고, 그 집합이 닫힐 때까지 이어집니다. 내장 게이트에 대한 adjoint(Adjoint H)는 건드리지 않습니다 — 그것은 방출 시점에 inv @ h가 됩니다. 여기서는 오직 유저 연산만 실체화됩니다.

왜 이름 붙이기 전에 실체화하나 (방출 시점이 아니라)? 합성된 역연산 def는 최종 QASM이 다른 모든 전역 이름(예컨대 Prep.Fanout__sz3__adj라는 유저 레지스터)과 충돌하지 않게 지켜야 할 이름입니다. 이름 맹글러가 돌기 전에 — 진짜 op으로 — 여기서 만들면, 그 역연산은 다른 모든 연산과 같은 충돌 해소를 거치므로, 생성된 이름이 이름 붙이기 이후에 태어나는 일이 결코 없습니다. 단형화기가 쓰는 것과 같은 수법입니다: 생성된 op은 맹글링 전에 실체화하고, 이미터는 이름을 지어낼 필요가 전혀 없습니다.

8Mangle & Check — 충돌 없는 이름, 그리고 검증

패스: NameMangler → ReferentialCheck

Mangle은 모든 Qora 이름을 유효하고 충돌 없는 OpenQASM 식별자로 대응시킵니다. 두 가지가 일어납니다: 네임스페이스가 붙은 이름은 점을 평탄화하고(Prep.Fanout__sz3 → Prep_Fanout__sz3), 충돌할 이름 — OpenQASM 키워드, stdgates 게이트명, 또는 이미 방출된 다른 이름과 — 은 유일해질 때까지 _를 덧붙입니다. 따라서 충돌은 결코 컴파일 오류가 아닙니다: 자동으로 해소되고, 이름이 조정되었음을 독자가 알 수 있도록 // Qora: 노트가 QASM 헤더에 기록됩니다.

이름은 OpenQASM을 그대로 본뜬 스코프에 산다 연산 이름과 엔트리 op의 최상위 선언은 하나의 전역 스코프를 공유하고(stdgates처럼), 각 def는 파라미터·변수를 위한 자신의 로컬 스코프를 갖습니다. 그래서 (게이트 s와 부딪히는) s라는 변수는 s_로 방출되지만, 같은 이름이 다른 def 안에서는 자유롭게 다시 나타날 수 있습니다.

Check(참조 무결성 게이트)는 맹글링 직후에 도는 안전망입니다: 프로그램이 사용하는 모든 식별자가 선언된 무언가로 해소되는지 확인합니다 — 파라미터·레지스터·변수·연산·내장. 여기서 매달린 참조는 어떤 패스가 선언은 개명했지만 그 사용처 하나는 개명하지 않았다는 뜻 — 컴파일러 버그 — 이므로, 조용히 깨진 QASM으로 방출하는 대신 QINTERNAL로 요란하게 보고됩니다. 올바른 컴파일러에서는 결코 발동하지 않습니다; 그 부류의 버그를 아예 출하하지 못하게 하려고 존재합니다.

9Emit — OpenQASM 3로 내보내는 단순 출력기

in: 맹글된 IR  ·  out: OpenQASM 3 텍스트  ·  패스: OpenQasmLowering → QasmEmitter

그 직전에 도는 패스 — OpenQASM용 타깃 로워링(OpenQasmLowering)

출력기 앞에서 마지막 IR→IR 단계 하나가 돕니다 — OpenQasmLowering, OpenQASM 특화 타깃 로워링입니다. Qora의 const는 어떤 값이든 담을 수 있는 불변 바인딩이지만(Q#의 let 방식), OpenQASM의 const는 컴파일 타임 상수 초기값을 요구합니다. 그래서 런타임에 묶이는 const c = n;은 방출 시 평범한 int c = n;으로 강등됩니다 — 여전히 재대입되지 않으므로, 불변성은 조금도 잃지 않습니다: 

int n = 2;
const c = n;          ← Qora const: 불변 바인딩, 어떤 값이든
                   ↓ OpenQasmLowering
int n = 2;
int c = n;            ← 평범한 int (OpenQASM const는 컴파일 타임이어야 함)

이것이 첫 타깃별 로워링 패스입니다 — Qora에게는 올바른 IR을 특정 백엔드의 규칙에 맞게 조정하는 곳이죠. 미래의 QIR 타깃이라면 위쪽 어떤 패스도 건드리지 않고 여기에 자신만의 로워링을 추가할 것입니다.

마지막 패스는 의도적으로 가장 덜 영리합니다: 이름을 지어내지도, 무엇을 역연산하지도 않습니다 — 모든 변환은 이미 앞에서 일어났습니다. 그저 각 연산을 def로, 엔트리 op을 최상위 몸통으로 출력할 뿐입니다. 유일한 실제 잡무는 순서 잡기입니다: OpenQASM에는 전방 선언이 없으므로, def들은 호출 의존(위상 정렬) 순서로 방출됩니다 — 그래서 Prep_Fanout__sz3__adj는 그것이 필요로 하는 것보다 앞서지 않고, 모든 def는 사용되기 전에 선언됩니다. 예제의 완전한 출력: 

// generated by Qora v0.14
OPENQASM 3;
include "stdgates.inc";

def Prep_Fanout__sz3(qubit[3] q) {      ← 네임스페이스 + 특수화 + 평탄화
  for int i in [0:3 - 1] {
    h q[i];
  }
}

def Prep_Fanout__sz3__adj(qubit[3] q) {  ← 합성된 역연산 (소스에 없던 것)
  for int i in [3 - 1:-1:0] {            ← 역방향 루프
    inv @ h q[i];                         ← dagger된 게이트
  }
}

qubit[3] reg;
bit r;
Prep_Fanout__sz3(reg);
Prep_Fanout__sz3__adj(reg);              ← Adjoint Prep.Fanout(reg)가 이 호출이 됨
r = measure reg[0];
왜 이미터를 이렇게 단순하게 두나? 출력만 하는 출력기는 신뢰하기 쉽기 때문입니다. 역연산·특수화·이름 붙이기가 모두 각자의 패스에 살면, 마지막 텍스트 생성 단계에는 내릴 결정이 하나도 남지 않습니다 — 그래서 그것이 쓰는 OpenQASM은 건네받은 IR 그대로, 문자 하나까지 똑같습니다.

이 형태는 의도된 것입니다: 엔진과의 경계 하나, 그다음 각자 IR에 한 가지 일만 하는 작은 패스들의 일직선, 그리고 그 앞에 선 검증 게이트 덕분에 이후 모든 것이 단순해질 수 있습니다. 언어 기능을 추가하는 것은 보통 한 패스를 추가하거나 확장하는 일이지 — 컴파일러를 다시 쓰는 일이 아니며 — 끝의 출력 전용 이미터는 여러분이 얻는 OpenQASM이 패스들이 만든 IR 그대로임을 뜻합니다.

이것이 향하는 곳 — 자동 언컴퓨테이션 Silq식 자동 언컴퓨테이션 — 임시 큐비트가 더 이상 필요 없어지는 지점을 찾아 그 역연산을 자동으로 주입하는 것 — 이야말로 이 패스들이 향해 온 목표입니다. 첫 조각은 이미 자리를 잡았습니다: ConjugationLowering이 within/apply 노드를 계산 · apply · 언컴퓨트로 평탄화해 방출하므로, 이후의 경로 전체(역연산 합성 → __adj 닫힘 → 맹글링 → QASM)가 처음부터 끝까지 검증되어 있습니다. 남은 것은 그 노드를 스스로 만들어내는 분석입니다 — 라이브니스(임시 큐비트가 언제 죽는지)와 효과 분석(안전하게 되돌릴 수 있는지) — 그다음 작은 주입기가 각 소멸 지점에서 QConjugate를 방출합니다.
