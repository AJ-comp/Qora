The Adjoint Compilation Pipeline — From Source to OpenQASM

The complete journey of one example — gates, declarations, for, if, and nested calls — through four stages

In Qora, when you write Adjoint Foo(q), the compiler automatically synthesizes a subroutine (Foo__adj) that performs the inverse of Foo and swaps the call for it. This document follows exactly where and in what order that happens, one step at a time, using real compiler output. Every AST, IR, and OpenQASM listing below was produced by actually compiling the example.

1The example program — one file with all five constructs

We packed all five static constructs that inverse synthesis supports — gates, classical declarations, for loops, if branches, and nested calls — into a single program. 

operation Inner(Qubit[2] q) {          // target of the nested call
    H(q[0]);
    T(q[1]);
}

operation Outer(Qubit[2] q, bit b) {
    int k = 2;                         // ① classical declaration
    Inner(q);                          // ② nested call
    Rx(pi/k, q[0]);                    // ③ gate (uses the declared value as an angle)
    for i in 0..1 {                    // ④ for loop
        X(q[i]);
    }
    if (b == 1) {                      // ⑤ if branch
        Z(q[0]);
        Adjoint S(q[0]);               //    a gate that already carries the functor
    }
}

operation Main() {
    use q = Qubit[2];
    bit r = M(q[0]);
    Outer(q, r);
    Adjoint Outer(q, r);               // ★ this one line triggers the entire inverse synthesis
}

2Stage 1. AST — the syntax tree the parser builds

The parser (the Janglim engine) turns the source into a syntax tree. At this stage there is no trace of an "inverse" at all. Even Adjoint Outer(q, r) is just "one call statement with an Adjoint token attached." Looking only at the subtree for Outer: 

Operation
├─ "Outer"
├─ Param(2, q)   Param(bit, b)
├─ VarDecl(int, k, Expr: 2)              ← int k = 2;
├─ Gate("Inner", Expr: q)                ← calls and gates share the same Gate tag
├─ Gate("Rx", Expr: pi / k, Qubit(q,0))
├─ For(i, 0, 1)
│   └─ Gate("X", Qubit(q,i))
└─ If
    ├─ Condition(b == 1)
    ├─ Gate("Z", Qubit(q,0))
    └─ Gate("Adjoint", "S", Qubit(q,0))  ← the functor is a child token
This tree is never modified afterwards The later stages only read this tree. Nothing like Outer___adj even exists at this point — it is created fresh in stage 4.

3Stage 2. QoraIR — the intermediate representation the compiler owns

Lowering (QoraLowering) converts the engine's AST once into Qora's own IR (QProgram). From here on, everything is handled as strongly typed nodes (QGate, QFor, QIf, …) instead of string tags. The IR for Outer: 

QOperation Outer(Qubit[2] q, Bit b)
  QDecl(const=False, type=Int, name=k, value=2)
  QGate(functors=[],        name=Inner, args=[q])
  QGate(functors=[],        name=Rx,    args=[pi / k, q[0]])
  QFor(i in 0..1)
    QGate(functors=[],      name=X,     args=[q[i]])
  QIf(cond="b == 1")
    then:
      QGate(functors=[],        name=Z, args=[q[0]])
      QGate(functors=[Adjoint], name=S, args=[q[0]])

Notice: the functor has become data on the node, as in functors=[Adjoint]. Because "is this gate inverted?" is expressed as a value rather than as text, the next stage can build the inverse simply by flipping that value.

4Stage 3. Inverter — synthesizing the inverse on the IR

Since Main contains Adjoint Outer(...), the compiler needs the inverse of Outer. The Inverter is a pure transformation pass that takes IR and returns inverse IR. Its mathematical basis is a single fact:

(U₁ · U₂ · ⋯ · Uₙ)†  =  Uₙ† · ⋯ · U₂† · U₁†
— reverse the order, and take the inverse (dagger) of each

The inverse IR of Outer that the Inverter actually returned:

QDecl(const=False, type=Int, name=k, value=2)     ← ① declaration: order kept, stays first
QIf(cond="b == 1")                                ← ⑤ if: condition kept, only the branch body inverted
  then:
    QGate(functors=[],        name=S, args=[q[0]])   ← Adjoint S → S (the functor is stripped)
    QGate(functors=[Adjoint], name=Z, args=[q[0]])   ← Z → Adjoint Z
QFor(i in 1..0, step=-1)                          ← ④ for: range swapped + reversed
  QGate(functors=[Adjoint], name=X, args=[q[i]])
QGate(functors=[Adjoint], name=Rx, args=[pi / k, q[0]])  ← ③ inverse rotation
QGate(functors=[Adjoint], name=Inner, args=[q])          ← ② Inner → Adjoint Inner (transitive)

Why aren't declarations inverted? A classical computation like int k = 2; has no such thing as an "inverse." And because Qora rejects variable reassignment inside anything targeted for inverse synthesis, a declared value stays the same to the very end. So by leaving the declaration in place at the front, the inverse-direction gates (inv @ rx(pi / k)) read exactly the same angle as the forward direction.
Transitive synthesis — even inverses absent from the source, automatically Since the inverse of Outer now calls Adjoint Inner, the inverse of Inner became necessary too. Nowhere in the source did we ever write Adjoint Inner, but the compiler tracks the need and builds Inner___adj as well. This demand keeps propagating along the call chain (a transitive closure).

5Stage 4. OpenQASM — emitting the inverse IR as-is

Finally, the Emitter (QasmEmitter) lowers the IR to OpenQASM 3 text. The inverse IR is ordinary IR with nothing special about it, so it is emitted through exactly the same path as regular statements. The full output:

(One more pass runs right before emission: the name mangler appends _ to every user-defined name — q → q_, Outer → Outer_ — so your names can never collide with OpenQASM's keywords or stdgates' gate names. That is why every name below carries a trailing underscore, and why the synthesized inverse of Inner_ is named Inner___adj. Errors and the stages view keep the original names; only the QASM is mangled.) 

// generated by Qora v0.13
OPENQASM 3;
include "stdgates.inc";

def Inner_(qubit[2] q_) {
  h q_[0];
  t q_[1];
}

def Outer_(qubit[2] q_, bit b_) {
  int k_ = 2;
  Inner_(q_);
  rx(pi / k_) q_[0];
  for int i_ in [0:1] {
    x q_[i_];
  }
  if (b_ == true) {
    z q_[0];
    inv @ s q_[0];
  }
}

def Inner___adj(qubit[2] q_) {           ← a def that was not in the source — synthesized transitively
  inv @ t q_[1];
  inv @ h q_[0];
}

def Outer___adj(qubit[2] q_, bit b_) {    ← the inverse def that Adjoint Outer will call
  int k_ = 2;                           ← ① declaration kept
  if (b_ == true) {                        ← ⑤ condition kept, only the inside inverted
    s q_[0];
    inv @ z q_[0];
  }
  for int i_ in [1:-1:0] {              ← ④ reversed loop (order: 1, 0)
    inv @ x q_[i_];
  }
  inv @ rx(pi / k_) q_[0];               ← ③ inverse rotation
  Inner___adj(q_);                       ← ② call to the synthesized inverse def
}

qubit[2] q_;
bit r_;
r_ = measure q_[0];
Outer_(q_, r_);
Outer___adj(q_, r_);                      ← Adjoint Outer(q, r) is replaced by this
The def order follows a rule too OpenQASM only lets you use something after declaring it (declare-before-use). So the Emitter lays out all defs (forward + synthesized) in call-dependency order (a topological sort) — which is why Inner___adj always appears before the Outer___adj that calls it.

6What gets inverted, and what gets rejected

Applying Adjoint to an op whose inverse cannot be built is a compile error. Before emitting, a validation pass walks the entire IR once and collects every semantic error, and if there is even one, no OpenQASM is generated at all — so a broken program never quietly "pretends to succeed" and hands you a wrong circuit. The reasons for rejection fall into two families.

ConstructVerdictReason
Gates · for · if · declarations · nested calls✅ Inversion supportedthe four-stage rules in this document
Measurement (M) · Reset❌ Rejected permanentlyPhysically irreversible — measurement collapses the state and throws information away
Variable reassignment (k = k + 1;)❌ Rejected for nowneeds tracking of classical state over time (in the effect-analysis stage)
while · repeat❌ Rejected for nowthe iteration count cannot be known at compile time
use (local qubits)❌ Rejected for nowcleaning up ancilla qubits is the job of automatic uncomputation
Recursive calls❌ Rejectedthe inverse would be recursive too, and termination cannot be guaranteed
Error reasons propagate along the call chain

If Caller calls a Meas that measures, then Adjoint Caller(q) becomes this compile error:

[QSEM001] in `Main`: `Adjoint Caller` cannot be compiled: it calls `Meas`, which cannot be inverted (it measures a qubit, and measurement is irreversible)

Following the chain of parentheses leads you all the way to the root cause (the measurement in Meas). Errors do not stop at the first find — they are all reported in a single compile.

7One-page summary

User code ─Parse─▶ AST (no trace of the inverse) ─Lowering─▶ QoraIR (functors as data) ─Validate─▶ STOP on semantic errors ─Inverter─▶ inverse QoraIR (toggle·swap·reverse) ─Emitter─▶ OpenQASM (__adj defs + call replacement)

What the single line Adjoint Foo(q) does: the compiler ① tracks that the inverse of Foo is needed, ② synthesizes the inverse IR on the IR itself as "reversed order + per-gate inverse" (transitively covering the ops Foo calls, if needed), ③ emits it as a def named Foo__adj, and ④ swaps the call site for Foo__adj(q). The user never wrote a single line of inverse-direction code, yet the final OpenQASM contains the complete inverse — the same kind of compile-time code synthesis as C#'s async building a state machine for you.

What this structure opens up next — automatic uncomputation Temporary (ancilla) qubits must be returned to their original state with the inverse after use, or interference gets contaminated. Right now the inverse is synthesized only when the user explicitly writes Adjoint, but by reusing this very Inverter, the compiler can find the point where a temporary qubit is no longer used and inject the inverse automatically. That is the Silq-style automatic uncomputation Qora is heading toward.
