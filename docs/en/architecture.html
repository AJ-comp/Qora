The Compilation Pipeline — Every Stage from Source to OpenQASM

How one file flows through parse, lowering, the IR passes, and emission — what each stage does, and in what order

Qora is a small front end sitting on top of a parser engine (Janglim), plus a multi-pass middle end that Qora owns entirely. Compilation is a straight line of small, single-purpose passes: each takes the program in one form, does exactly one job, and hands the next pass a slightly more finished program — until the last pass prints OpenQASM 3. This document walks that line end to end, one stage at a time, using real compiler output for every AST, IR, and QASM listing below.

Source ─Parse─▶ AST ─Lower─▶ Qora IR ─Expand─▶ merged ─Resolve─▶ qualified names ─Validate─▶ (STOP on error) ─Monomorphize─▶ specialized ─Materialize Adjoint─▶ + inverse ops ─Mangle─▶ collision-free ─Check─▶ ─Emit─▶ OpenQASM 3

0The example — one file that touches every pass

To see every stage do real work, we use one small program that packs in a namespace, a generic register (Qubit[n]), a call, and an Adjoint: 

namespace Prep {
    operation Fanout(Qubit[n] q) {       // generic: n is fixed later, per call site
        for i in 0..n-1 { H(q[i]); }
    }
}

operation Main() {
    use reg = Qubit[3];
    Prep.Fanout(reg);                    // call across a namespace
    Adjoint Prep.Fanout(reg);            // undo it — the compiler synthesizes the inverse
    bit r = M(reg[0]);
}

Watch three names travel through the passes: Fanout gains a namespace (Prep.Fanout), gains a concrete size (Prep.Fanout__sz3), gains an inverse twin (Prep.Fanout__sz3__adj), and finally flattens for OpenQASM (Prep_Fanout__sz3). Each of those changes is one pass, below.

1Two worlds — where the engine stops and Qora begins

There is exactly one boundary in the whole compiler. Left of it lives the Janglim parser engine: it owns the grammar, the lexer, and turning text into a syntax tree. Right of it lives Qora: every pass that gives the program meaning, checks it, transforms it, and prints it. The two worlds meet at a single function — lowering — which reads the engine's tree once and never looks back.

Source → AST  │ engine (Janglim)
Qora │  IR → ⋯ → OpenQASM
Why keep the boundary this sharp? Everything Qora needs to reason about — types, inverses, generics, name safety — is done on Qora's own IR, not on the engine's tree. So the engine can stay a generic parser with no quantum knowledge, and Qora can add features (new passes) without ever touching the grammar. The one place that depends on the engine is lowering; the rest of the compiler would survive swapping the parser out.

2Parse — source to AST (the engine's job)

in: source text  ·  out: syntax tree  ·  owner: Janglim engine

The engine builds a lexer and an LALR parser from Qora's grammar, then turns the source into a parse tree. At this stage there is no meaning — Adjoint is just a token sitting next to a call, Prep.Fanout is three tokens (Prep, ., Fanout), and a call and a gate look identical. The real tree for our example: 

Nonterminal : Program
  Nonterminal : Namespace
    Terminal : Prep
    Nonterminal : Operation
      Terminal : Fanout
      Nonterminal : Param
        Terminal : n                    ← the generic size symbol
        Terminal : q
      Nonterminal : For
        Terminal : i
        Nonterminal : Expr ( 0 )
        Nonterminal : Expr ( n - 1 )
        Nonterminal : Gate
          Terminal : H
          Nonterminal : Qubit ( q , i )
  Nonterminal : Operation
    Terminal : Main
    Nonterminal : Use ( reg , 3 )
    Nonterminal : Gate                  ← a call is tagged Gate, like everything else
      Terminal : Prep
      Terminal : .
      Terminal : Fanout
      Nonterminal : Expr ( reg )
    Nonterminal : Gate
      Terminal : Adjoint              ← the functor is just a leading token
      Terminal : Prep
      Terminal : .
      Terminal : Fanout
      Nonterminal : Expr ( reg )
    Nonterminal : VarDecl
      Terminal : bit
      Terminal : r
      Nonterminal : Expr → Call ( M , Qubit ( reg , 0 ) )
This tree is read once, then never touched again Every later stage works on the IR, not this tree. Nothing named Prep.Fanout__sz3 or __adj exists yet — those are all born downstream.

3Lower — AST to Qora IR (crossing the boundary)

in: engine AST  ·  out: QProgram (Qora IR)  ·  pass: QoraLowering

Lowering walks the engine's tree once and rebuilds it as Qora's own IR — a hierarchy of strongly-typed records (QOperation, QGate, QFor, QIf, QDecl, QUse, QMeasure, …). Two changes matter most: string tokens become typed nodes, and functors become data on the node (functors=[Adjoint]). The IR of our example: 

QOperation Prep.Fanout(Qubit q)
  QFor(i in 0..n - 1)
    QGate(functors=[], name=H, args=[q[i]])

QOperation Main()
  QUse(name=reg, size=3)
  QGate(functors=[],        name=Prep.Fanout, args=[reg])
  QGate(functors=[Adjoint], name=Prep.Fanout, args=[reg])
  QDecl(const=False, type=Bit, name=r, value=QMeasure(reg[0]))
Why turn the functor into data? Because "is this call inverted?" is now a value (functors=[Adjoint]) rather than a token to re-parse, a later pass can build an inverse just by flipping that value — no string surgery. This is the same reason the whole middle end is IR-based: passes read and write structured nodes, so each one stays short and hard to get wrong.

4Expand & Resolve — imports and fully-qualified names

passes: ModuleLoader.Expand → Resolver.Resolve

Expand is the module loader. If the file had import "lib/prep.qora", this pass parses that file, lowers it, and merges its operations into one QProgram so everything downstream sees a single flat program. (Our example imports nothing, so this is a no-op here — but it is where a missing file becomes QSEM020 and an import cycle becomes QSEM021.)

Resolve then turns every simple name into a fully-qualified name (FQN). Inside namespace Prep, the operation Fanout becomes Prep.Fanout, and the call site Prep.Fanout(reg) is checked to point at it. This is why the IR listing above already reads Prep.Fanout, not Fanout. If a name were ambiguous (visible from two opened namespaces) it would be QSEM018; an unknown namespace or member is QSEM019. Names are now unique keys the rest of the pipeline can trust.

5Validate — the semantic gate

in: resolved IR  ·  out: a list of errors (or none)  ·  pass: QoraValidator

Validation is the pass that answers one question: does this program mean something OpenQASM can express? It walks the whole IR once and collects every violation (no stopping at the first). The result is a hard gate: one error and no OpenQASM is produced at all, so a broken program can never quietly "succeed" and hand back a wrong circuit. Every rule has a stable code (QSEM001–QSEM024) so the editor can attach a squiggle. A few of them:

CodeRejects
QSEM001Adjoint Foo where Foo has no inverse (measurement, reset, a while loop, …) — reason chained from the inverter
QSEM002Controlled Foo on a user op (OpenQASM has no ctrl @ for a def)
QSEM006a call whose arguments do not match the signature — count, kind (qubit vs angle), or register size
QSEM007an unknown gate or operation name (with a "did you mean …?" hint)
QSEM011recursive calls (OpenQASM defs cannot recurse)
QSEM016a literal qubit index outside its register's range
Why validate before the transforming passes? Because it lets every later pass assume the program is already sound. The monomorphizer never meets an unknown name; the adjoint materializer never meets a non-invertible Adjoint (QSEM001 already rejected it); the emitter never meets a bad call. Each transforming pass gets to be simple because the gate stands in front of them all.

6Monomorphize — specializing generic registers

in: validated IR  ·  out: IR with concrete-size ops  ·  pass: Monomorphizer

Fanout(Qubit[n] q) is generic: n is unknown until a call fixes it. The monomorphizer works like a C++ template or a Rust generic — for each call site, it stamps out a concrete copy of the operation with n replaced by the actual register size, and rewrites the call to point at that copy. Our call passes reg (3 qubits), so:

Prep.Fanout(Qubit[n])  +  call with a size-3 register  ⟶  Prep.Fanout__sz3(Qubit[3])

Because sizes are now concrete, the program is validated a second time: checks that only make sense once n is a number — index bounds, register-size matches — finally have something to run against. After this pass, no generic operations remain.

7Materialize Adjoint — inverses become real ops

in: specialized IR  ·  out: IR with synthesized inverse ops  ·  pass: AdjointMaterializer

Our Main contains Adjoint Prep.Fanout(reg). This pass turns that whole-operation adjoint into an ordinary operation. Using the inversion kernel (Inverter), it computes the inverse body — reverse the statements, dagger each gate — adds it to the program as a real op named Prep.Fanout__sz3__adj, and rewrites the call to a plain Prep.Fanout__sz3__adj(reg). The inverse of a fan-out of H gates is the reversed loop with each H daggered:

(U₁ · U₂ · ⋯ · Uₙ)†  =  Uₙ† · ⋯ · U₂† · U₁†

The demand is transitive: if an inverse body calls Adjoint Bar, then Bar's inverse is materialized too, and so on until the set closes. Adjoint on a built-in gate (Adjoint H) is left alone — that becomes inv @ h at emit time; only user operations are materialized here.

Why materialize before naming (and not at emit time)? A synthesized inverse def is a name the final QASM must keep clear of every other global name (a user register also called Prep.Fanout__sz3__adj, say). By creating it here — as a real op, before the name mangler runs — the inverse flows through the same collision resolution as every other operation, so no generated name is ever born after naming. It is the same trick the monomorphizer uses: materialize generated ops before mangling, and the emitter never has to invent a name.

8Mangle & Check — collision-free names, then verified

passes: NameMangler → ReferentialCheck

Mangle maps every Qora name to a valid, collision-free OpenQASM identifier. Two things happen: a namespaced name flattens its dots (Prep.Fanout__sz3 → Prep_Fanout__sz3), and any name that would collide — with an OpenQASM keyword, a stdgates gate name, or another emitted name — gets _ appended until it is unique. A collision is therefore never a compile error: it is resolved automatically, and a // Qora: note is written into the QASM header so the reader knows a name was adjusted.

Names live in scopes that mirror OpenQASM Operation names and the entry op's top-level declarations share one global scope (just like stdgates); each def gets its own local scope for its parameters and variables. That is why a variable named s (which clashes with the s gate) is emitted as s_, while the same name is free to reappear inside a different def.

Check (the referential-integrity gate) is a safety net that runs right after mangling: it verifies that every identifier the program uses resolves to something declared — a parameter, register, variable, operation, or built-in. A dangling reference here would mean a pass renamed a declaration but not one of its uses — a compiler bug — so it is reported loudly as QINTERNAL rather than emitted as silently-broken QASM. On a correct compiler it never fires; it exists to make that whole class of bug impossible to ship.

9Emit — the dumb printer to OpenQASM 3

in: mangled IR  ·  out: OpenQASM 3 text  ·  pass: QasmEmitter

The last pass is deliberately the least clever: it invents no names and inverts nothing — every transformation already happened upstream. It just prints each operation as a def and the entry op as the top-level body. Its only real chore is ordering: OpenQASM has no forward declarations, so defs are emitted in call-dependency (topological) order — which is why Prep_Fanout__sz3__adj appears before nothing that it needs, and every def is declared before it is used. The complete output for our example: 

// generated by Qora v0.14
OPENQASM 3;
include "stdgates.inc";

def Prep_Fanout__sz3(qubit[3] q) {      ← namespaced + specialized + flattened
  for int i in [0:3 - 1] {
    h q[i];
  }
}

def Prep_Fanout__sz3__adj(qubit[3] q) {  ← synthesized inverse (not in the source)
  for int i in [3 - 1:-1:0] {            ← reversed loop
    inv @ h q[i];                         ← daggered gate
  }
}

qubit[3] reg;
bit r;
Prep_Fanout__sz3(reg);
Prep_Fanout__sz3__adj(reg);              ← Adjoint Prep.Fanout(reg) became this call
r = measure reg[0];
Why keep the emitter this dumb? Because a printer that only prints is easy to trust. When inversion, specialization, and naming all live in their own passes, the final text-producing step has no decisions left to make — so the OpenQASM it writes is simply the IR it was handed, character for character.

10The whole pipeline at a glance

StagePassTurns … into …
ParseJanglim enginesource text → syntax tree (no meaning yet)
LowerQoraLoweringengine tree → Qora IR (typed nodes; functors as data)
ExpandModuleLoaderimports → one merged program
ResolveResolversimple names → fully-qualified names
ValidateQoraValidatorIR → every semantic error (a hard gate on emission)
MonomorphizeMonomorphizergeneric Qubit[n] → concrete-size copies, then re-validate
MaterializeAdjointMaterializerAdjoint Foo → a real Foo__adj op + a plain call
MangleNameManglerQora names → collision-free OpenQASM identifiers
CheckReferentialCheckverifies every used name resolves (else QINTERNAL)
EmitQasmEmitterfinal IR → OpenQASM 3 text (a pure printer)

The shape is deliberate: one boundary with the engine, then a line of small passes that each do one thing to the IR, with a validation gate up front so everything after it can be simple. Adding a language feature usually means adding or extending one pass — not rewriting the compiler — and the print-only emitter at the end means the OpenQASM you get is exactly the IR the passes built.

Where this is heading — automatic uncomputation The same inversion kernel the adjoint materializer uses will be reused by a future pass that finds where a temporary qubit is no longer needed and injects its inverse automatically — Silq-style automatic uncomputation. Because inverse synthesis is already a materialize-before-mangle pass, that new pass slots into the same line without disturbing anything downstream.

Every AST, IR, and OpenQASM listing in this document was produced by compiling the example above with the real Qora compiler (qora --json --stages).
